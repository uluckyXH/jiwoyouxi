# 空当接龙进入与发牌动效及资源就绪设计

更新时间：2026-07-17

适用范围：`entry/src/main/ets/games/freecell/`，以及空当接龙从大厅进入时的首屏体验。

## 1. 结论

可以为进入空当接龙增加一段原生“发牌入场”动效，但不能把资源加载完成当作游戏可用的前置条件。

首开时，空当接龙的 rawfile 视觉资源约为 6.7 MB，其中背景图约 1.5 MB，标题图约 506 KB。它们首次解码时可能晚于页面切换；如果直接在资源到达后再开始整段动画，用户会感觉游戏先卡住、随后又多等一次。

第一版采用以下原则：

- 大厅到游戏页继续复用已有的全局场景转场和顶部身份胶囊。
- 空当接龙页面先立即给出暖色稳定底板，绝不出现白屏，也不等待全部装饰资源。
- 关键资源在页面内并行预热；资源及时就绪时播放约半秒的发牌动效。
- 关键资源超过短时预算仍未就绪时，直接显示完整静态牌局，不再补播发牌动画。
- 发牌期间仅锁住牌桌和玩法工具，返回大厅始终可用；任意一次牌桌触摸只会跳过动效，不会误执行移动。
- 不改变发牌种子、规则、计分、计时起点、提示、撤销或自动归位逻辑。

这样动画是“准备充分时的奖励”，不是加载慢时的额外等待。

## 2. 当前基础与问题

目前已有的能力：

- `Index.ets` 通过 `GameSceneTransition` 完成大厅到游戏页的原生左推、淡入和轻微缩放。
- `GameEntryBadge` 会在顶部短暂显示游戏图标和名称。
- `FreecellPage` 已使用 `layoutReady`：先获得真实区域尺寸，再挂载主要棋盘，避免默认 `390 x 760` 布局先渲染后重排。
- 页面背景已有 `FreecellTheme.pageFallback` 暖色兜底，因此资源未显示时不会是纯白。
- 52 张牌面由 ArkUI 的文字和形状组合绘制，不依赖 52 张独立牌图；资源压力主要来自背景、标题、槽位、装饰和工具栏图片。

当前缺口：

- `layoutReady` 后背景、标题、装饰、棋盘和 HUD 一次性出现，进入感较硬。
- 没有“关键图片是否已经可稳定显示”的独立状态，首次进入仍可能看见兜底背景切换为正式背景。
- 若直接对 52 张牌逐张起定时器或为每张牌维护独立状态，会增加主线程状态更新和重组成本，不符合游戏性能约束。

## 3. 目标与非目标

### 目标

- 从大厅进入空当接龙时，先看到稳定场景，再看到短促、可识别的发牌过程。
- 首次资源解码慢时保持可退出、无白屏、无无限加载。
- 动画结束后牌局的布局与现在完全一致，玩家可立即开始操作。
- 发牌动效在普通手机竖屏下总时长控制在 0.45 到 0.60 秒。
- 不在帧循环、`build()` 或每张牌上执行资源加载和复杂计算。

### 非目标

- 本期不改空当接龙规则，不重新洗牌，不改变可解牌局生成方式。
- 本期不为“新开一局”重复播放入场动画；新局应保持立即可玩。
- 本期不增加新的音频资源。现有点击、提示音仍按现有方式初始化和复用。
- 本期不在大厅预先解码整套空当接龙图片，避免未进入游戏时额外占用内存和影响大厅启动。
- 本期不把小游戏改造成 WebView、HTML 或 Web 动画方案。

## 4. 鸿蒙官方原生动画选型

本方案只使用 ArkTS / ArkUI 官方能力，不引入 WebView、CSS、GIF 序列或第三方动画库。2026-07-17 已通过华为开发者官网当前文档核对以下接口。

| 需求 | 采用的官方能力 | 本方案中的用途 |
| --- | --- | --- |
| 页面内状态变化 | `this.getUIContext().animateTo(...)` | 正式背景、HUD、遮罩的透明度与轻微位移变化 |
| 卡牌按轮出现 | `transition(TransitionEffect...)` | `ForEach` 在每轮新增牌时，以透明度、平移和缩放组合完成落位 |
| 关键图片就绪 | `Image.onComplete(...)` | 图片数据加载且解码成功后，记为关键资源已就绪 |
| 图片异常 | `Image.onError(...)` | 标记资源失败并走静态牌局兜底，不阻塞进入 |
| 资源加载慢的视觉底板 | `Image.alt(...)` 与现有 `pageFallback` | 保持暖色底板和现有组件自然降级，避免白色块 |

官方约束直接落实为以下规则：

- 不调用已从 API 18 起废弃的全局 `animateTo`；统一使用绑定当前页面 UI 上下文的 `this.getUIContext().animateTo(...)`。
- 不在 `aboutToAppear()` 或 `aboutToDisappear()` 中启动动画。入场状态只能在真实布局已建立、节点已挂载后的 `onAreaChange`、组件 `onAppear` 或图片 `onComplete` 路径中切换。
- 组件新增和删除使用 `TransitionEffect`，不使用已废弃的 `TransitionOptions`。它天然适合初始发牌轮次中 `ForEach` 新增牌节点的场景。
- `onFinish` 只用于清理视觉状态，不能承担“牌局已开始”“开放操作”“计时开始”等业务正确性。官方文档明确说明，系统关闭过渡动画或 UIAbility 转入后台时，结束回调可能立即触发。
- 不使用 `animateToImmediately`、`contentTransition`、GIF 或 PixelMap 序列。前两者分别是更高版本的特定场景能力，后两者会为本次简单入场引入无必要的资源和内存成本。

推荐的组合形态如下，代码仅表示实现方向，具体常量集中到 `FreecellMotion.ets`：

```ts
this.getUIContext().animateTo({
  duration: FreecellMotion.sceneRevealDuration,
  curve: Curve.EaseOut
}, () => {
  this.isIntroSceneVisible = true;
});

// 每轮新增的牌节点使用组件内转场，而不是单独创建 52 个定时器。
.transition(
  TransitionEffect.OPACITY
    .combine(TransitionEffect.translate({ x: dealOffsetX, y: dealOffsetY }))
    .combine(TransitionEffect.scale({ x: 0.96, y: 0.96 }))
    .animation({ duration: FreecellMotion.dealCardDuration, curve: Curve.EaseOut, delay: columnDelay })
)
```

官方资料：

- [UIContext.animateTo](https://developer.huawei.com/consumer/cn/doc/harmonyos-references/arkts-apis-uicontext-uicontext#animateto)
- [组件内转场 transition / TransitionEffect](https://developer.huawei.com/consumer/cn/doc/harmonyos-references/ts-transition-animation-component)
- [Image.onComplete / onError](https://developer.huawei.com/consumer/cn/doc/harmonyos-references/ts-basic-components-image#oncomplete)

## 5. 与当前 App 主题的适配

空当接龙不是科技感控制台，也不是高速街机。当前页面已经建立了暖色庭院、木质牌匾、奶油色纸牌和鸡窝装饰的视觉语言，入场动效应强化“摊开一桌纸牌”的感觉，而不是覆盖它。

- 底色沿用 `FreecellTheme.pageFallback` 的 `#fff0c9`；正式背景只做温和淡入，不出现冷色加载幕或黑色遮罩。
- 标题、统计牌和工具栏沿用深棕 `#4b260b`、木棕 `#68340d`、奶油 `#fff4d7` 的现有层级，只做 6 至 8vp 的轻微上移和淡入。
- 纸牌保持现有奶油牌面、棕色描边和静态阴影；入场只使用 `opacity + translate + scale`，不做翻面、旋转、强弹簧、扫光、爆炸粒子或全屏特效。
- 牌的运动方向以右侧牌背装饰为视觉来源，但实际平移距离设上限，避免左侧列跨越整屏飞行而遮挡标题和身份胶囊。
- 底部庭院装饰和吉祥物只在牌桌稳定后轻淡入，不参与发牌动画，也不遮挡卡牌或触摸区域。
- 顶部已有游戏身份胶囊，因此不新增第二个游戏名横幅；页面内部动效聚焦“场景定下、牌局摊开”。

这套节奏要克制、温暖、像实物卡牌落到木桌上。动画完成后所有尺寸、坐标、字体和层级都回到当前 UI，不产生布局跳变。

## 6. 进入时序

```text
点击大厅空当接龙卡片
  -> 已有场景转场与身份胶囊
  -> FreecellPage 获得真实布局尺寸，显示暖色底板和稳定棋盘轮廓
  -> 并行预热关键图片
  -> 关键图片及时就绪：背景/标题淡入，牌局按 7 轮发到 8 列
  -> READY：解除棋盘和工具栏操作锁

关键图片超时或加载失败
  -> 直接完整显示静态牌局
  -> READY：解除操作锁，不补播迟到动画
```

### 阶段定义

| 阶段 | 可见内容 | 可操作性 | 离开方式 |
| --- | --- | --- | --- |
| `WAITING_LAYOUT` | `pageFallback` 暖色底板 | 仅返回大厅 | 等待 `onAreaChange` |
| `WARMING_ASSETS` | 正确尺寸的背景色、棋盘轮廓、必要槽位 | 仅返回大厅 | 关键资源就绪或到达兜底时限 |
| `DEALING` | 正式背景和 HUD 轻入，牌从牌背位置按轮进入 8 列 | 返回大厅、轻触牌桌跳过 | 7 轮结束或用户跳过 |
| `READY` | 完整现有棋盘 | 全部正常操作 | 正常游戏流程 |

`WAITING_LAYOUT` 和 `WARMING_ASSETS` 不显示“加载中”文案或旋转器。已有页面转场和稳定底板足够传达正在进入；额外加载提示会放大等待感。

## 7. 视觉方案

### 7.1 场景层次

1. 暖色 fallback 始终在最底层。
2. 正式花园背景在解码完成后用 `opacity` 从 0 过渡到 1，时长约 180 ms；中间保留同色轻遮罩，避免颜色突变。
3. 标题、计分 HUD、空槽和底部工具栏以轻微上移加淡入出现，不改变任何固定尺寸。
4. 牌桌的 8 列按空当接龙常见的发牌节奏出现：每轮每列增加一张，前四列共 7 张，后四列共 6 张。
5. 每张新显示的牌从右侧牌背装饰的方向短距离移到目标位置，带透明度和轻微缩放，不改变最终坐标、牌间距或层级。

发牌不是逐张用 52 个定时器串行播放，而是 7 个“轮次”状态。每轮同时处理 8 个列位，列内增加短延迟，使人眼看到由右向左/由上至下的牌桌建立过程。

### 7.2 推荐时长

| 动作 | 时长/节奏 | 说明 |
| --- | --- | --- |
| 正式背景和 HUD 入场 | 180 至 220 ms | 与全局转场重叠时仍保持轻量 |
| 首轮发牌前停顿 | 50 至 70 ms | 给场景一个稳定落点 |
| 每轮发牌 | 45 至 55 ms | 总计 7 轮，不产生拖沓感 |
| 列内延迟 | 12 至 18 ms | 只用于同一轮的视觉错开 |
| 单张落位曲线 | `Curve.EaseOut` 或轻阻尼曲线 | 不使用强回弹，避免牌面抖动 |
| 完整发牌总时长 | 不超过 560 ms | 含首轮停顿与最后一轮落位 |

不要再对整个页面叠加一段大幅缩放或大位移。外层已经有全局场景转场，空当接龙内部只负责牌桌建立感。

### 7.3 交互细节

- `DEALING` 时，牌桌和工具栏不接受移动、提示、撤销、自动归位、新局请求，避免用户输入排队到动画结束后误执行。
- 返回大厅按钮和系统返回保持可用；离开时必须取消所有入场计时器。
- `DEALING` 时轻触牌桌会立即展示完整牌局并进入 `READY`，这次触摸不视为选牌或落牌。
- 不增加显眼的“跳过”按钮，避免破坏原有紧凑 HUD；交互行为应是自然的。

## 8. 资源就绪策略

### 8.1 分级

关键资源只覆盖首屏结构，不等待全部装饰：

- `page_bg_coop_garden.png`
- `klondike_title_banner_cut.png`
- `freecell_empty_slot_cut.png`
- 四张 foundation slot 图
- `card_back_checkered.png`

延后资源包括前景装饰、吉祥物、规则牌、角标、工具栏图标和统计卡底图。它们可随 ArkUI 的正常图片解码逐步出现，不影响发牌是否开始。

牌面文字和花色由代码绘制，不进入图片预热清单。

### 8.2 就绪与兜底

- 将 `Image.onComplete` 和 `Image.onError` 绑定到实际显示的关键图片。官方定义的 `onComplete` 在图片数据加载且解码成功时触发，正好可作为资源就绪信号。
- 不创建第二套不可见 `Image` 节点来“预热”，避免重复节点和潜在重复解码；`FreecellAssetWarmup` 只聚合实际图片的资源 ID、完成回调和失败回调。
- 不读取 rawfile 字节，不在主线程手动解码图片，也不在 `build()` 中发起资源加载。
- 从布局就绪开始设置短时预算，建议 360 ms。关键资源全部完成时立即开始发牌；超过预算则直接静态进入 `READY`。
- 任一资源回调失败不能阻塞页面。页面保留 fallback 和已有图片组件的自然降级路径。
- 已进入过一次且系统图片缓存命中时，仍播放同样的短发牌动效；缓存只缩短资源准备，不改变玩法状态。

这样不会把资源加载时长绑定到固定动画时长。慢设备最多看到稳定的静态牌局，而不是“先空白、再多等半秒”。

## 9. 模块拆分

`FreecellPage.ets` 当前已超过开发规范建议的 800 行。实现本方案时，不应继续把资源状态、计时器和动画 Builder 堆入该文件。

建议拆分：

```text
entry/src/main/ets/games/freecell/
  FreecellPage.ets                 页面组装、生命周期、状态连接
  FreecellIntroTypes.ets           IntroPhase、资源状态类型
  FreecellIntroRules.ets           轮次、可见牌判断、超时决策等纯函数
  FreecellIntroController.ets      入场计时器管理和取消逻辑
  FreecellAssetWarmup.ets          关键图片 ID、实际 Image 回调和就绪聚合
  FreecellIntroLayer.ets           背景/HUD 的轻入场遮罩与视觉层
  FreecellMotion.ets               空当接龙私有时长、延迟和曲线 token
  FreecellBoard.ets                增加只读 intro 展示参数和交互开关
```

职责边界：

- `FreecellIntroRules.ets` 不依赖 ArkUI，输入为牌列长度、轮次和资源状态，输出为当前可见牌与是否结束；可以独立测试。
- `FreecellIntroController.ets` 只管理少量轮次定时器和取消行为，不生成牌局，也不处理移动规则。
- `FreecellAssetWarmup.ets` 只报告实际图片的资源结果，不重复渲染图片，也不修改页面玩法状态。
- `FreecellBoard.ets` 继续负责牌面和手势；只根据 `introPhase` 决定初始渲染时哪些牌可见，以及是否接受输入。
- `FreecellPage.ets` 在 `aboutToDisappear()` 和返回大厅路径中统一停止入场控制器，保证不会在离开页面后写入状态。

## 10. 性能约束

- 初始牌局仍只生成一次。`createSolvableGame(randomSeed())` 是 52 张牌的轻量同步规则计算，不需要为此引入 Worker。
- 不为 52 张牌分别创建 `@State`、定时器、资源对象或音频对象。
- 发牌只推进 7 次轮次状态；每次状态改变只影响当前轮新增牌的可见性。
- 不在 `build()` 内创建随机牌局、读取资源、排序或做图片预处理。
- 不对棋盘宽高、卡牌最终坐标或布局比例做动画，避免小屏重排、文字抖动和命中区域漂移。
- 背景、标题和槽位图片通过 `$rawfile` 引用并在页面生命周期中复用；不在新局、撤销、提示或帧循环中重新加载。
- 离开页面、快速返回大厅、重复进入时必须取消预热超时和发牌轮次计时器。

## 11. 测试与验收

### 自动化

新增 `entry/src/ohosTest/ets/test/FreecellIntroRules.test.ets`，使用 HarmonyOS 官方 Hypium / ArkXTest 体系覆盖：

- 8 列 `7/7/7/7/6/6/6/6` 的各轮可见牌数量。
- 首轮、最后一轮、跳过和超时的状态转换。
- `READY` 状态下展示完整牌局，不影响现有移动规则。
- 资源失败或超时直接静态进入，绝不滞留在 `WARMING_ASSETS`。

入场规则测试不依赖图片解码时序；图片回调、ArkUI 转场和真实资源解码通过构建后的真机验收覆盖。

### 真机验收

- 冷启动后第一次从大厅进入：无白屏、黑屏、明显闪烁或超过短时预算的不可操作状态。
- 再次进入：发牌节奏自然，不和顶部身份胶囊造成明显遮挡。
- 低端或资源加载慢场景：最多直接显示静态完整牌局，仍可返回和开始操作。
- 发牌中轻触牌桌：立即完整展牌，不产生选牌、移动、计时或得分变化。
- 发牌中点返回或使用系统返回：无残留计时器、无页面离开后的状态更新。
- 在开发者选项关闭过渡动画后进入：应直接到完整牌局，不依赖 `onFinish` 才进入 `READY`。
- 新开一局、撤销、提示、自动归位、结算、返回大厅与最高分行为和改动前一致。
- 小屏、常规手机、宽屏下，牌面、HUD 和工具栏没有重叠或尺寸跳变。

完成后在 `测试报告/` 中留下首次进入、重复进入、快速返回和小屏测试记录；构建命令按项目规范执行 `./scripts/build-app.sh --stacktrace`。

## 12. 实施顺序

1. 先拆出入场类型、纯轮次规则和私有动效 token，补 Hypium 规则测试。
2. 实现 `FreecellAssetWarmup` 和 360 ms 静态兜底，先验证首开不白屏、不阻塞。
3. 给 `FreecellBoard` 加入只读展示轮次和交互开关，先实现无资源依赖的 7 轮发牌。
4. 接入背景/HUD 的轻入场和牌桌跳过行为，验证返回与生命周期清理。
5. 真机测量首次进入和二次进入，再根据实际观感微调轮次间隔；不通过增加总时长来掩盖资源慢。

## 13. 后续可选项

- 单独设计“新开一局”的洗牌/重新发牌反馈；它不应复用本次进入动效，以免影响频繁重开的效率。
- 若真机确认图片解码仍是瓶颈，再评估压缩大背景、提供较低分辨率预览图或在大厅空闲时做极小范围预热。该优化需先有实测证据，不能默认扩大常驻内存。
- 若后续增加系统“减少动态效果”设置，可让入场直接走静态 `READY` 路径，而不影响游戏可玩性。
