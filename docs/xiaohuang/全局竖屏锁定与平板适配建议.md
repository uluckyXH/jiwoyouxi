# 全局竖屏锁定与平板适配建议

更新时间：2026-07-23

## 背景

当前《鸡窝游戏》主要按手机竖屏体验开发。首页、收藏页、设置页以及多数小游戏都以竖屏手机为主要目标。项目 `module.json5` 已声明支持 `phone` 和 `tablet`，但目前还没有系统性平板布局适配。

如果平板或折叠屏进入横屏，现有页面很容易出现：

- 首页卡片过度拉宽。
- 底部 Tab 和内容比例失衡。
- Canvas 游戏可玩区过大，坐标、碰撞和 UI 层级失去原本节奏。
- 个别游戏的顶部/底部控件距离过远，操作效率变差。

因此在完成真正的平板适配前，建议先采用“全局锁竖屏”的保守策略，避免横屏下页面直接失控。

## 当前工程情况

### module.json5

`entry/src/main/module.json5` 当前声明：

```json5
"deviceTypes": [
  "phone",
  "tablet"
]
```

但 `EntryAbility` 还没有配置 `orientation`，因此系统可以根据设备、旋转和窗口策略进入不同方向。

### EntryAbility

`entry/src/main/ets/entryability/EntryAbility.ets` 当前只处理：

- 沉浸式窗口。
- 全屏布局。
- 加载 `pages/Index`。

没有主动设置窗口首选方向。

### 弹射鸡蛋

`entry/src/main/ets/games/eggSling/EggSlingPage.ets` 当前有页面级横屏逻辑：

- 进入页面时调用 `setPreferredOrientation(window.Orientation.LANDSCAPE)`。
- 离开页面时恢复 `window.Orientation.UNSPECIFIED`。
- 通过 `display.getDefaultDisplaySync().orientation` 和页面宽高判断横竖屏，用于横屏提示。

这套逻辑和“全局永远竖屏”的策略冲突。如果要全局锁竖屏，需要冻结或移除弹射鸡蛋的横屏请求。

## 推荐策略

### 第一阶段：全局锁竖屏

目标：

- App 在手机、平板上都保持竖屏。
- 不因为设备旋转触发横屏布局。
- 不继续维护单个游戏的临时横屏逻辑。

建议改动：

1. 在 `entry/src/main/module.json5` 的 `EntryAbility` 配置中增加：

```json5
"orientation": "portrait"
```

2. 在 `EntryAbility.ets` 获取主窗口后补运行时兜底：

```ts
mainWindow.setPreferredOrientation(window.Orientation.PORTRAIT).catch((error: Object) => {
  hilog.warn(DOMAIN, 'WorldOfRPSNative', 'Failed to lock portrait orientation: %{public}s', JSON.stringify(error));
});
```

3. 处理 `EggSlingPage.ets`：

- 移除进入时的 `window.Orientation.LANDSCAPE`。
- 移除退出时恢复 `window.Orientation.UNSPECIFIED`。
- 或临时统一改成 `window.Orientation.PORTRAIT`。

推荐直接移除页面级方向切换，避免后续页面之间互相覆盖窗口方向。

## 为什么不先做平板横屏

平板横屏不是简单放大页面。当前游戏合集里有多种页面形态：

- 首页和收藏页是卡片菜单。
- 扫雷、2048、俄罗斯方块是棋盘类。
- 合成大西瓜、RPS Battle 是 Canvas/物理/自绘场景。
- 弹射鸡蛋曾经按横屏玩法讨论过，但和当前合集统一竖屏方向不一致。

如果直接开放横屏，需要分别处理：

- 内容最大宽度。
- 横屏双栏或居中布局。
- 游戏画布最大尺寸。
- 触摸区域和按钮距离。
- 安全区、状态栏、底部手势条。
- 各游戏的横屏坐标和碰撞边界。

这不是一个小改动。当前更务实的做法是先锁竖屏，保证体验稳定。

## 第二阶段：平板竖屏适配

锁竖屏不等于已经完成平板适配。平板竖屏仍然会比手机宽很多，后续建议做以下优化：

### App 大厅

- 首页、收藏页、成就页、设置页内容增加最大宽度。
- 大屏下内容居中，避免卡片横向拉得过宽。
- 底部 Tab 保持满宽或跟内容最大宽度对齐，需要单独评估。

建议初始最大宽度：

```text
430vp 到 520vp
```

### 游戏页面

棋盘类游戏：

- 棋盘设置最大宽度。
- 操作按钮和状态信息跟棋盘宽度对齐。

Canvas/物理类游戏：

- 可玩区不直接吃满平板宽高。
- 使用手机竖屏比例作为基础世界，平板上居中或适度放大。
- UI 控件不贴太远。

### 弹射鸡蛋

如果后续仍要保留弹射鸡蛋，需要重新评估方向：

- 方案 A：改成竖屏玩法，和合集方向统一。
- 方案 B：隐藏/暂缓弹射鸡蛋，等横屏体系成熟后再放出。
- 方案 C：保留独立横屏，但这会破坏“全局永远竖屏”的策略，不建议当前阶段采用。

## 风险

- 某些设备可能仍以自由窗口/平行视界方式运行，`orientation` 和 `setPreferredOrientation` 不一定完全等价于强制固定窗口比例。
- 锁竖屏后，弹射鸡蛋当前玩法空间可能更紧，需要暂缓或重新设计。
- 平板竖屏下虽然不会横屏乱飞，但仍可能显得内容过宽，需要第二阶段最大宽度约束。

## 验收点

- 手机竖屏进入 App 正常。
- 手机旋转设备后 App 不切横屏。
- 平板竖屏进入 App 正常。
- 平板旋转设备后 App 不切横屏。
- 从大厅进入任意游戏后仍保持竖屏。
- 从游戏返回大厅后仍保持竖屏。
- 弹射鸡蛋不再主动把窗口切成横屏。
- 首页、收藏页、设置页没有因为锁方向出现布局回退。
