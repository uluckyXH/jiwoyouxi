# HarmonyOS 原生动效全量改造方案

更新时间：2026-06-10

当前实施状态见 `APP动画清单.md`。本文保留为完整设计背景和长期方案说明；后续是否继续新增动画，以清单中的“暂缓清单”和“验收清单”为准。

## 结论

这个项目适合做鸿蒙原生动效方案，技术上可行，也符合当前工程方向。

当前工程已经是 HarmonyOS NEXT 原生项目，代码主要是 `.ets` 文件，使用 ArkTS + ArkUI 的声明式组件写法，比如 `@Component`、`@State`、`Stack`、`Column`、`Row`、`Canvas`。ArkTS 可以理解为面向 HarmonyOS 应用开发的 TypeScript 风格语言，但它不是浏览器里的普通 TS，也不是 WebView。UI、状态和动画都运行在 ArkUI 原生体系里。

因此动效改造也应该走 ArkUI 原生能力：

- 页面/场景切换：`Navigation`、`NavDestination`、`animateTo`、`transition`。
- 组件出现/消失：`transition(TransitionEffect...)`。
- 状态变化动画：`this.getUIContext().animateTo(...)`。
- 大厅卡片到游戏页的上下文承接：`geometryTransition`。
- 游戏内部反馈：ArkUI 属性动画、Canvas 自绘动画、少量粒子/图片序列。

短期建议不立刻重构成 `Navigation` 路由栈，而是在当前 `Index.ets` 的薄路由外面补一层原生场景转场。中期再按需要迁移到 `Navigation + NavPathStack`。

## 官方依据

本方案参考华为 HarmonyOS 官方文档，查询时间为 2026-06-10：

- [Navigation转场动画](https://developer.huawei.com/consumer/cn/doc/harmonyos-guides/arkts-navigation-animation)：`Navigation` 支持系统默认转场、自定义转场和共享元素转场。默认转场使用弹簧曲线，时长与设备和物理参数相关，不建议业务逻辑依赖默认动画结束时机。
- [页面转场动画 (不推荐)](https://developer.huawei.com/consumer/cn/doc/harmonyos-guides/arkts-page-transition-animation)：官方更推荐使用 `Navigation` 转场动画和模态转场。
- [显式动画 animateTo](https://developer.huawei.com/consumer/cn/doc/harmonyos-references/ts-explicit-animation)：用于把状态变化包进显式动画；建议使用 `getUIContext().animateTo()` 明确 UI 上下文。
- [组件内转场 transition](https://developer.huawei.com/consumer/cn/doc/harmonyos-references/ts-transition-animation-component)：适合组件插入和删除时的过渡，支持透明度、位移、缩放、组合和非对称入退场。
- [组件内隐式共享元素转场 geometryTransition](https://developer.huawei.com/consumer/cn/doc/harmonyos-references/ts-transition-animation-geometrytransition)：通过相同 id 绑定 in/out 组件，建立空间位置上的连续关系；必须配合 `animateTo` 才有动画效果。
- [共享元素转场 sharedTransition](https://developer.huawei.com/consumer/cn/doc/harmonyos-references/ts-transition-animation-shared-elements)：适用于 `@ohos.router` 页面路由跳转。当前项目不是多页面 router 结构，短期不优先使用。
- [动画曲线概述](https://developer.huawei.com/consumer/cn/doc/harmonyos-guides/arkts-curve-overview)：物理弹簧曲线更符合用户对自然运动的认知，建议优先用于页面级转场。
- [应用UX体验建议](https://developer.huawei.com/consumer/cn/doc/harmonyos-guides/experience-suggestions-ux)：层级转场宜使用左右位移，新建转场宜使用上下位移，搜索等上下文连续场景宜使用共享元素。
- [帧率体验建议](https://developer.huawei.com/consumer/cn/doc/harmonyos-guides/performance-frame-rate)：应用内转场过程应避免连续丢帧和卡顿，后续真机验收要关注转场流畅度。

## 当前结构判断

现有项目结构适合先做轻量原生动效改造：

- `entry/src/main/ets/pages/Index.ets`：当前是合集入口、游戏切换和存档桥接层。它是薄路由，职责还可控。
- `entry/src/main/ets/pages/HubPage.ets`：大厅、最近游玩、收藏、成就、设置集中在这里。
- `entry/src/main/ets/shell/GameRegistry.ets`：游戏元信息统一注册。
- `entry/src/main/ets/app/CoopStorage.ets`：最高分、最近游玩、收藏等本地存储统一走这里。
- `entry/src/main/ets/games/<gameId>`：每个小游戏独立目录，扫雷已经拆到 `Page / Rules / Types / Theme / Board / Hud / ResultOverlay / FeatherLayer`，这个拆分方向健康。

现在的短板不是代码已经乱到必须重构，而是缺少“合集级体验基础设施”：

- 缺统一动效 token。
- 缺统一场景转场壳。
- 缺统一按钮按压反馈。
- 缺统一暂停/结算弹层动效规范。
- 各小游戏内部反馈动效不一致。

因此第一阶段应保持现有结构，补原生动效层；不要一上来大改路由。

## 选型

### 第一阶段：原生 ArkUI 场景转场

使用：

- `this.getUIContext().animateTo(...)`
- `.transition(TransitionEffect...)`
- 外层 `Stack` / `Column` 包装游戏页和大厅页

适合当前结构，因为 `Index.ets` 通过 `activeGame` 条件分支切页面。只要在分支根组件上加 `transition`，并把 `activeGame` 的变化放进 `animateTo`，就能让大厅和游戏页的插入/删除产生原生转场。

第一阶段不使用：

- `pageTransition`：官方标注不推荐。
- `sharedTransition`：它服务于 `@ohos.router` 页面跳转，当前工程不是这个形态。
- 直接动画游戏主体宽高：Canvas、棋盘、物理游戏容器容易出现内容跳变和掉帧。

### 第二阶段：组件内共享元素

使用：

- `geometryTransition(id)`
- `animateTo`
- 关闭或弱化同时存在的普通场景位移

适合做“点大厅卡片，卡片图标/色块过渡到游戏页标题”的一镜到底感。这个阶段要保证同一个 id 只绑定一对 in/out 组件，避免多个游戏卡片同时参与共享元素。

### 第三阶段：Navigation 化

使用：

- `Navigation`
- `NavPathStack`
- `NavDestination.systemTransition`
- `NavDestination.customTransition`
- `customNavContentTransition`

当游戏数量继续增加，或者需要更标准的系统返回栈体验时，再做这一阶段。迁移时需要重新验证返回键、最高分、最近游玩、每个游戏的生命周期。

## 动效原则

1. 原生优先。所有动效用 ArkTS/ArkUI 能力实现，不引入 WebView 或前端动画库。
2. 快而可感知。页面级转场 `220ms` 到 `320ms`；按钮反馈 `80ms` 到 `140ms`；弹层 `180ms` 到 `240ms`。
3. 层级清楚。进入游戏是下钻，方向向左推进；返回大厅是回退，方向反向。
4. 轻量稳定。页面壳只做 `opacity / translate / scale`，避免动画游戏主体尺寸。
5. 曲线自然。页面级动效优先物理/阻尼曲线；普通淡入淡出使用 `Curve.EaseOut` 或 `Curve.Friction`。
6. 输入收敛。转场期间锁住重复点击，避免多次打开游戏或连续返回。
7. 业务不依赖默认动画时长。默认转场可能因设备和系统设置变化，存档和计时逻辑不能依赖动画结束时间。
8. 游戏动效服务玩法。每个小游戏只加能提升反馈的动效，不做干扰判断和操作的装饰。

## 新增代码结构

建议新增以下文件：

```text
entry/src/main/ets/design/CoopMotion.ets
entry/src/main/ets/shell/GameSceneTransition.ets
entry/src/main/ets/components/MotionPressable.ets
entry/src/main/ets/components/CoopModalTransition.ets
```

职责：

- `CoopMotion.ets`：统一时长、曲线、位移、缩放、帧率期望等 token。
- `GameSceneTransition.ets`：统一大厅和游戏页切换动画。
- `MotionPressable.ets`：统一按钮、卡片的按下缩放和透明度反馈。
- `CoopModalTransition.ets`：统一暂停、结算、设置弹层的遮罩和面板转场。

第一版可以只落 `CoopMotion.ets` 和 `GameSceneTransition.ets`，其余组件在后续 PR 中逐步抽出。

## 动效 Token

建议第一版 token：

```ts
export const CoopMotion = {
  scenePushDuration: 260,
  scenePopDuration: 220,
  sharedElementDuration: 300,
  modalDuration: 220,
  modalMaskDuration: 140,
  pressDuration: 90,
  tabDuration: 160,
  microDuration: 120,
  scenePushOffsetVp: 24,
  scenePopOffsetVp: 18,
  sceneScaleFrom: 0.985,
  pressScale: 0.975
};
```

曲线建议：

- 页面进出：`Curve.Friction` 或当前 SDK 支持的物理弹簧曲线。
- 弹窗进入：弹簧/阻尼曲线，避免过冲太大。
- 弹窗退出：`Curve.EaseOut`。
- 按压反馈：`Curve.EaseOut`。
- 游戏内轻反馈：按玩法选择，优先短促、清晰。

## 场景规格

### 大厅进入游戏

触发：点击大厅游戏卡片。

能力：`animateTo + transition`。

效果：

- 大厅整体向左 `-18vp`，透明度 `1 -> 0`，缩放 `1 -> 0.985`。
- 游戏页从右侧 `24vp` 进入，透明度 `0 -> 1`，缩放 `0.985 -> 1`。
- 时长 `260ms`。
- 转场期间禁止重复打开游戏。
- 最近游玩记录可以在点击时立即写入，不依赖动画结束。

### 游戏返回大厅

触发：游戏内“大厅”按钮、系统返回键。

能力：`animateTo + transition`。

效果：

- 游戏页向右 `24vp`，透明度 `1 -> 0`。
- 大厅从左侧 `-18vp` 回来，透明度 `0 -> 1`。
- 时长 `220ms`。
- 方向必须和进入相反，让用户明确是回退。

### 大厅游戏卡片按压

触发：手指按下、松开、取消。

能力：`animateTo`。

效果：

- 按下缩放 `1 -> 0.975`。
- 阴影略收，透明度不低于 `0.96`。
- 松手 `90ms` 恢复。
- 不改变卡片布局，不导致网格重排。

### 大厅 Tab 切换

触发：首页、收藏、成就、设置切换。

能力：`animateTo`。

效果：

- 新内容从 `8vp` 下方轻移入。
- 透明度 `0 -> 1`。
- 时长 `160ms`。
- 不使用大幅左右位移，避免大厅内部层级感混乱。

### 暂停弹层

触发：游戏内暂停。

能力：`transition`。

效果：

- 遮罩淡入 `120ms`。
- 面板缩放 `0.96 -> 1`，透明度 `0 -> 1`。
- 关闭时淡出，时长 `160ms`。
- 暂停出现后游戏计时和主循环必须停止，不能等动画结束再停。

### 结算弹层

触发：胜利、失败、游戏结束。

能力：`transition + 游戏内自定义动效`。

效果：

- 遮罩先淡入。
- 弹窗从 `0.94` 弹到 `1`。
- 主按钮、次按钮可延迟 `40ms` 出现。
- 游戏专属特效保留在游戏模块内，例如扫雷羽毛、白块失败闪烁、俄罗斯方块消行动效。

### 共享元素

触发：点击大厅游戏卡片进入游戏。

能力：`geometryTransition`。

效果：

- 大厅卡片 icon 或 accent 色块过渡到游戏页标题 icon。
- 共享元素 id 使用 `game-card-${gameId}`。
- 只在第二阶段做，第一阶段先完成基础场景转场。
- 使用共享元素时减少普通场景位移，避免两个动效抢焦点。

## 各游戏动效方案

### 石头剪刀布大作战 `rps`

目标：让观战乱斗更有节奏，但不影响模拟判断。

建议：

- 开局：阵营计数和战场 Canvas 淡入。
- 响指、黑洞、暴走等机制触发：对应状态条或标记短促闪烁。
- 结算：胜利阵营徽标轻微放大，报告面板淡入。
- 不建议：为每个单位都加 ArkUI 组件动画，战场实体仍由 Canvas 自绘。

### 别踩白块 `whiteTiles`

目标：强化点击反馈和失败原因。

建议：

- 正确点击黑块：格子按下 `scale 0.96`，快速淡出或变浅。
- 连击分数：分数轻微上浮或放大 `120ms`。
- 漏点/误点：所在列短促横向抖动，失败遮罩淡入。
- 新行出现：纵向滚动保持稳定，不额外装饰。

### 扫雷 `minesweeper`

目标：保留推理效率，补足鸡窝主题反馈。

建议：

- 翻格：打开格子用 `scale 0.94 -> 1` + 透明度，不做复杂翻牌，避免影响批量展开。
- 插旗：旗子从 `0.75` 弹到 `1`，时长 `120ms`。
- 踩雷：当前已有羽毛层和失败弹窗，保留；踩中格可做一次 `scale 1 -> 1.06 -> 1`。
- 胜利：自动插旗后弹窗进入，少量羽毛慢飘。
- 不建议：给空白连锁展开每个格子做长延迟，会拖慢扫雷反馈。

### 俄罗斯方块 `tetris`

目标：提升消行和操作反馈，不干扰输入。

建议：

- 方块锁定：落点位置轻微闪烁或压缩 `80ms`。
- 消行：整行高亮 `120ms` 后清除。
- 升级：等级文本轻微放大。
- 底部控制按钮：按下缩放，长按保持轻微高亮。
- 不建议：旋转/移动方块用 ArkUI 动画补间，方块运动必须保持即时响应。

### 合成大西瓜 `suika`

目标：突出物理合成反馈。

建议：

- 水果投放：待投水果轻微上下浮动。
- 合成：新水果 `scale 0.82 -> 1.08 -> 1`，同时分数轻弹。
- 接近失败线：失败线低频呼吸，不要持续高亮。
- 失败：遮罩和结果弹窗统一走全局弹层转场。
- 不建议：给所有水果常驻装饰动画，避免影响物理观感和帧率。

### 弹射鸡蛋 `eggSling`

目标：强化拖拽弹弓手感和命中反馈。

建议：

- 拖拽：弹弓皮筋跟手，力度条或轨迹点透明度随拉力变化。
- 松手：鸡蛋沿 Canvas 或状态更新运动，按钮/提示不抢焦点。
- 命中目标：目标短促缩放或闪光，分数上浮。
- 失败/过关：结算弹层统一动效。
- 不建议：轨迹点过多或常驻粒子，避免小屏视觉噪声。

## 实施顺序

### P0：合集级原生转场

1. 新增 `CoopMotion.ets`。
2. 新增 `GameSceneTransition.ets`。
3. 改 `Index.ets`：`openGame`、`exitToHub`、`onBackPress` 统一走 `animateTo`。
4. 转场期间锁住重复点击。
5. 验证 6 个游戏能进入、返回、记录最近游玩。

### P1：大厅和弹层统一反馈

1. 大厅游戏卡片按压反馈。
2. 大厅 Tab 内容切换反馈。
3. 抽统一暂停弹层转场。
4. 抽统一结算弹层转场规范，先不强行重构每个游戏，只在后续游戏打磨时接入。

### P2：游戏内微动效

按游戏逐个接：

1. 扫雷：翻格、插旗、踩雷微反馈。
2. 俄罗斯方块：消行、锁定、控制按钮。
3. 白块：点击、失败抖动、分数。
4. 合成大西瓜：合成弹跳、失败线。
5. 弹射鸡蛋：拉力、轨迹、命中。
6. RPS：机制触发、结算报告。

### P3：共享元素和 Navigation

1. 给大厅游戏卡片和游戏页标题预留统一 icon/accent。
2. 接 `geometryTransition`，先选一个游戏试点。
3. 如果效果稳定，再推广到所有游戏。
4. 游戏数量继续增加后，再评估是否迁移 `Navigation + NavPathStack`。

## 验收标准

功能：

- 从大厅进入 6 个游戏都有统一入场动效。
- 每个游戏点“大厅”和按系统返回都使用统一退场动效。
- 转场过程中快速重复点击不会打开多个游戏或卡死。
- 最近游玩、收藏、最高分、游戏内 `recordScore` 不受影响。
- 关闭系统过渡动画时，业务状态仍然正确。

体验：

- 进入游戏能感知来自所点击卡片。
- 返回大厅方向与进入相反。
- 卡片按压有反馈但不重排。
- 暂停和结算弹层出现不突兀。
- 小屏无文字和按钮重叠，无内容压到系统手势区。
- 游戏内动效不影响判断和操作节奏。

性能：

- `./scripts/build-app.sh --stacktrace` 构建通过。
- 真机进入/退出各游戏无明显卡顿。
- 游戏内持续动画不因转场明显掉帧。
- 后续用 DevEco Studio AppAnalyzer 或真机观测检查应用内转场帧率，目标是无连续丢帧、无明显卡顿。

## 风险和约束

- `activeGame` 变化会触发页面插入/删除，必须确认游戏页面计时器、主循环在退出时正确停止。
- `transition` 的消失动画不要依赖子组件回调，必要时由 `Index.ets` 用统一状态控制转场锁。
- Canvas、棋盘和物理容器不要做宽高动画，避免内容跳变。
- `geometryTransition` 同一个 id 只能绑定一对 in/out 组件，不能让多个游戏卡片同时参与。
- 默认系统转场时长不可控，业务逻辑不要依赖默认转场结束。
- 真机开发者选项可能关闭过渡动画，动画完成回调可能立即执行，状态逻辑要能兜住。

## 不建议

- 不建议继续无动画直接切换。
- 不建议用 `pageTransition` 作为主方案。
- 不建议每个游戏设计完全不同的页面切换方式。
- 不建议用 Web 动画思路改 ArkUI 页面。
- 不建议先大规模迁移 `Navigation`，再同时加游戏内动效；风险会叠加。
- 不建议常驻大量粒子和装饰动画，尤其是小游戏主循环里。

## 推荐下一步

先做 P0。也就是只改合集外壳：

1. `CoopMotion.ets`
2. `GameSceneTransition.ets`
3. `Index.ets` 动画切换
4. 转场锁和系统返回接入

这一步完成后，整个 App 的“进入游戏/退出游戏”会立刻变得更像鸿蒙原生应用，同时风险控制在大厅和路由层，不会影响各小游戏内部玩法。
