# 扫雷计分设计

## 目标

扫雷后续不再做难度选择，默认固定为 8x8 棋盘。计分规则需要保持简单、直观，玩家不用理解复杂倍率，也不会因为失败局影响最高分。

核心原则：

- 只有通关才产生正式得分。
- 只有通关成绩才参与最高分记录。
- 失败只展示本局排查进度，不记录最高分。
- 分数主要奖励通关速度，越快越高。

## 棋盘基准

当前默认棋盘：

```text
8 x 8 = 64 格
鸡雷数量 = 10
安全格数量 = 54
```

不做难度后，计分公式不需要再引入难度倍率、鸡雷数量奖励或难度基础分。

## 胜利计分

胜利时计算正式得分：

```text
胜利得分 = max(100, 10000 - 用时秒数 * 10)
```

说明：

- `10000` 是满分基准。
- 每耗时 1 秒扣 10 分。
- 最低保底 100 分，避免极慢通关后出现 0 分或负分。
- 得分只在通关时记录到最高分。

示例：

```text
38 秒通关：10000 - 38 * 10 = 9620
60 秒通关：10000 - 60 * 10 = 9400
120 秒通关：10000 - 120 * 10 = 8800
```

## 失败反馈

失败时不计算正式得分，不写入最高分。

失败弹窗展示排查进度：

```text
已排查 23/54
```

其中：

- `23` 是已经翻开的安全格数量。
- `54` 是全部安全格数量。

这样玩家能知道自己推进到了哪里，但不会误以为失败局也在参与排行榜或最高分。

## UI 展示

顶部 HUD：

```text
最高 9620
```

顶部最高分只显示历史通关最高分。如果当前胜利分超过历史最高分，胜利弹窗和顶部 HUD 可以即时显示新的最高分。

胜利弹窗：

```text
鸡窝安全！
用时 00:38 · 得分 9620 · 最高 9620
```

失败弹窗：

```text
踩到鸡雷了！
已排查 23/54
```

失败弹窗不再显示“得分”和“最高”放在同一行，避免玩家误解失败也算正式成绩。

## 记录规则

游戏结束时：

- 胜利：计算胜利得分，调用 `recordScore(score)`。
- 失败：不调用 `recordScore`，只在弹窗中展示排查进度。

最高分判断保持现有全局记录逻辑，但输入来源只允许胜利局。

## 实现影响

需要调整的位置：

```text
entry/src/main/ets/games/minesweeper/MinesweeperRules.ets
entry/src/main/ets/games/minesweeper/MinesweeperPage.ets
entry/src/main/ets/games/minesweeper/MinesweeperResultOverlay.ets
entry/src/main/ets/games/minesweeper/MinesweeperHud.ets
```

实现方式：

- 使用 `minesweeperWinScore(elapsed)` 计算胜利正式得分。
- 使用 `openSafeCellCount(cells)` 和 `minesweeperSafeCellTotal(difficulty)` 计算失败排查进度。
- `finish(false)` 不再调用 `recordScore`。
- 失败弹窗根据 `openSafeCellCount / safeCellTotal` 展示进度。
- HUD 的 `currentScore` 只在胜利后参与最高分即时展示，失败局不会把当前翻格数显示成最高分。

## 验收标准

- 通关后显示得分，并更新最高分。
- 通关越快，得分越高。
- 失败后只显示排查进度，不更新最高分。
- 失败后顶部最高分不因为本局翻格数量变化。
- 重新开始后最高分仍保留历史通关最高分。
