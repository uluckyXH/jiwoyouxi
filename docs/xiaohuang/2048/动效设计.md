# 鸡窝 2048 动效设计

更新时间：2026-06-14

## 当前结论

鸡窝 2048 的首要目标仍然是“基础玩法正确、交互可信”。动效只能服务于操作反馈，不能再影响用户判断棋盘状态。

当前阶段采用保守方案：

- 保留标准 2048 的即时落位结果。
- 保留合并后卡片本体的轻微弹跳。
- 暂停全屏碎花、粒子、羽毛、贴纸飞散等装饰特效。
- 允许合并卡片内部出现少量圆点庆祝效果。
- 暂不恢复滑动位移动画，等合并和新块反馈稳定后再单独处理。

## 这次碎花方案的问题

已试过的碎花方案是：合并后在格子上方叠一层临时效果，再让 6 个小色块从中心散开。

实际观感不成立，原因包括：

- 碎花尺寸太小，在手机上看起来像一个彩点。
- 色块都从格子中心出现，初始状态过于集中，用户感知不到“散开”。
- 棋盘 `Stack` 有 `clip(true)`，花片只能在棋盘内活动，不适合做大范围飞散。
- 2048 的核心反馈是“合并后的卡片变大”，碎花抢了注意力，却没有强化规则结果。
- 如果用额外覆盖层做数字放大，视觉上像假卡片闪一下，不像真实卡片弹起来。

结论：

```text
首版不做纸片式碎花，也不做全屏烟花。
如果后续一定要做，只允许在合并弹跳稳定之后，用 Canvas 或素材序列单独实现，并限制数量和层级。
```

## 鸿蒙 ArkUI 动画调研结论

本方案参考 HarmonyOS / ArkUI 官方文档和当前项目已有实现。

官方文档：

- [显式动画 animateTo](https://developer.huawei.com/consumer/cn/doc/harmonyos-references/ts-explicit-animation)：将状态变化包进显式动画，项目里优先使用 `this.getUIContext().animateTo(...)`。
- [通用变换属性 scale](https://developer.huawei.com/consumer/cn/doc/harmonyos-references/ts-universal-attributes-transformation)：组件可通过 `scale({ x, y })` 做缩放变换。
- [组件内转场 transition](https://developer.huawei.com/consumer/cn/doc/harmonyos-references/ts-transition-animation-component)：适合组件插入和删除，不适合直接承担 2048 棋盘规则动画。
- [动画曲线概述](https://developer.huawei.com/consumer/cn/doc/harmonyos-guides/arkts-curve-overview)：反馈类动画可优先使用 `Curve.EaseOut`、`Curve.Friction` 这类自然曲线。
- [帧率体验建议](https://developer.huawei.com/consumer/cn/doc/harmonyos-guides/performance-frame-rate)：游戏内动效要避免连续掉帧，粒子数量和重绘范围要受控。

项目内已有参考：

- `entry/src/main/ets/components/MotionPressable.ets`：按钮按压使用 `scale` 绑定状态，再用 `getUIContext().animateTo(...)` 改状态。
- `entry/src/main/ets/components/CoopModalTransition.ets`：弹层进入使用 `opacity / scale / translate`，状态变化由 `animateTo` 驱动。
- `docs/xiaohuang/游戏切换动画设计方案.md`：App 级动画已经明确优先走 ArkUI 原生能力，不引入 WebView 或前端动画库。

本游戏的关键规则：

```text
动画只能读规则结果，不能反向参与规则计算。
```

也就是说：

- `boardTiles` 是真实棋盘，只给规则层使用。
- `displayTiles` 是展示棋盘，可以短暂带 `scale / opacity` 等视觉字段。
- 合并、新块、后续滑动都只能修改展示字段，不能修改真实规则字段。

## 推荐实现模型

### 合并弹跳

合并后让真实合并卡片本体弹一下，而不是叠假卡片。

目标效果：

```text
scale 1.00 -> 1.12/1.16 -> 1.00
```

建议参数：

| 参数 | 建议值 | 说明 |
| --- | --- | --- |
| 放大延迟 | `16ms - 32ms` | 等最终棋盘先落位，避免闪烁 |
| 放大时长 | `80ms - 100ms` | 快速、有反馈 |
| 回落时长 | `110ms - 150ms` | 回落比放大略慢 |
| 放大倍数 | `1.10 - 1.16` | 不能超过太多，避免顶到旁边格子 |
| 曲线 | `Curve.EaseOut` + `Curve.Friction` | 放大干脆，回落软一点 |

ArkUI 写法方向：

```ts
.scale({ x: tile.scale, y: tile.scale })

this.getUIContext().animateTo({
  duration: 90,
  curve: Curve.EaseOut
}, () => {
  this.displayTiles = this.withMergeScale(this.displayTiles, merges, 1.16);
});

this.getUIContext().animateTo({
  duration: 130,
  curve: Curve.Friction
}, () => {
  this.displayTiles = this.withMergeScale(this.displayTiles, merges, 1);
});
```

注意：

- ArkUI 状态数组要整体替换，不能只原地改数组里某个对象后期待稳定刷新。
- 定时器需要在 `aboutToDisappear` 和下一次移动前清理。
- 合并时可以锁输入，也可以不锁；如果不锁，必须保证下一次移动前清掉上一次脉冲定时器。
- 真机验证后，父页面直接改 `displayTiles[].scale` 不稳定，当前采用“合并目标格临时组件”方案。
- 临时组件由自己的 `@State` 和 `aboutToAppear()` 触发 `animateTo`，比父页面数组状态驱动更可靠。

### 卡片内圆点庆祝

参考截图中的效果，合并目标卡片内部可以出现 6 个暖黄色/橙色圆点。

这个效果不是纸片碎花，也不是全屏烟花，而是卡片内部的小范围庆祝点。

目标表现：

```text
合并后的数字卡片弹一下，同时几个圆点从数字附近弹到卡片周围。
```

约束：

- 只跟随合并目标卡片出现。
- 圆点在卡片内部活动，不飞出棋盘。
- 数量固定为 `6` 个左右。
- 颜色使用亮黄、橙黄、浅橙、浅粉、浅青等低饱和色板，不使用高饱和彩纸色。
- 每次合并根据合并事件 id 选择不同色板，形成稳定的“随机色彩”效果，避免重渲染时跳色。
- 生命周期跟随合并临时组件，结束后一起移除。
- 不参与规则计算，不表示新方块或额外状态。

实现建议：

```text
Chicken2048MergePulseTile
  - 自己管理 pulseScale
  - 自己管理 burstActive
  - aboutToAppear 后触发 scale 和圆点位置变化
```

不要再采用父页面统一更新粒子数组的方式，否则容易出现“状态变了但真机动画不可见”的问题。

### 新块出现

新生成的 `2/4` 需要轻微出现反馈，但不能盖住判断。

目标效果：

```text
scale 1.00 -> 1.12 -> 1.00
```

建议参数：

| 参数 | 建议值 | 说明 |
| --- | --- | --- |
| 时长 | `110ms - 150ms` | 比合并略轻 |
| 曲线 | `Curve.EaseOut` | 不需要弹过头 |
| 触发位置 | 新增 tile 所在格子 | 不影响其它卡片 |

实现要求：

- 规则层最好返回 `spawnedTile`，页面层不用猜哪个是新块。
- 首版如果规则函数暂时只返回最终数组，可以用 `isNew` 标记。
- 新块出现不能导致已有 tile 重新创建或闪烁。
- 新块不做缩小出现，也不做淡入标记，只做轻微弹一下，避免误以为它是特殊状态。
- 当前实现用 `Chicken2048SpawnPulseTile` 作为独立临时层，动画结束后移除。

### 滑动动画

滑动动画是最容易让 2048 看起来“规则错了”的部分，暂缓。

恢复前必须满足：

- 有真实棋盘 `boardTiles` 和动画棋盘 `displayTiles` 两套状态。
- 移动期间输入锁定，或只缓存一个方向。
- 动画帧不能参与下一次 `moveChicken2048Tiles` 计算。
- 合并场景要先显示两个旧块滑向同一格，再切换成合并后的新块。

建议参数：

| 参数 | 建议值 | 说明 |
| --- | --- | --- |
| 移动时长 | `90ms - 120ms` | 2048 标准手感偏快 |
| 曲线 | `Curve.EaseOut` | 清楚但不拖 |
| 输入策略 | 动画中忽略输入 | 首版先保守 |

恢复顺序：

1. 单独实现无合并滑动。
2. 再实现单合并滑动。
3. 再实现一行多个合并。
4. 最后处理连续快速输入。

### 碎花和粒子

首版禁用。

后续如果要加，只允许满足这些条件：

- 只在合并 `128+` 或 `512+` 时触发，低等级不触发。
- 单次粒子不超过 `6` 个。
- 粒子尺寸不得小于 `8vp`。
- 粒子运动距离至少为 tile 尺寸的 `0.35`，否则看起来像原地闪点。
- 粒子层必须 `hitTestBehavior(HitTestMode.None)`。
- 粒子不能遮挡数字超过 `120ms`。
- 不使用散落 `Row` 点位堆叠在中心的做法；优先 Canvas 或单独素材。

可选视觉方向：

```text
鸡毛 1-2 根 + 小星 2-3 个
```

不推荐：

```text
碎花纸片、彩色小点、大面积烟花、长时间漂浮粒子
```

## 当前实现范围

当前实现应该保持：

- 标准 2048 规则正确。
- 有效移动后同步更新最终棋盘。
- 合并卡片本体弹跳。
- 新块正常大小出现后轻微弹一下，不做缩小或淡入标记。
- 无纸片碎花和全屏粒子。
- 有合并卡片内部圆点庆祝。
- 无滑动位移动画。

实现备注：

```text
合并弹跳可以用目标格上方的同款临时卡片层实现。
这个层只展示合并后的数字和样式，做 scale 1 -> 1.24 -> 1 后移除。
它不参与规则计算，也不表示额外方块；用户视觉上看到的是合并后的目标卡片弹一下。
当前实现进一步在这个临时卡片层内部加了 6 个圆点，从中心附近弹到卡片边缘，模拟轻量烟花/庆祝点。
圆点色彩从几套低饱和色板中按合并事件 id 轮换。
新生成的数字用独立临时层做 scale 1 -> 1.12 -> 1，不改变真实棋盘。
```

代码位置：

```text
entry/src/main/ets/games/chicken2048/Chicken2048Page.ets
```

重点字段：

```text
boardTiles      真实规则棋盘
displayTiles    展示棋盘
tile.scale      卡片缩放展示字段
```

## 验收标准

玩法验收：

- `2 2 0 0` 向左后得到 `4 0 0 0`，并只新增一个 `2/4`。
- `2 2 2 2` 向左后得到 `4 4 0 0`，并只新增一个 `2/4`。
- 无效移动不新增块。
- 快速点击方向键不会出现数字乱跳或多次合并。

动效验收：

- 合并后的真实卡片能明显放大一次。
- 新生成的随机数字能轻微弹一下，但不缩小、不淡入。
- 放大不造成棋盘布局移动。
- 放大不遮挡相邻格数字太久。
- 没有纸片碎花、全屏烟花或闪烁假卡片。
- 圆点只在合并目标卡片内部出现，不能让用户误以为其它方块被标记。
- 连续合并时多个合并格都能弹一下。

性能验收：

- 一次移动只触发必要的数组替换。
- 不创建常驻粒子。
- 不使用全屏动画层。
- `./scripts/build-app.sh --stacktrace` 构建通过。

## 后续实施顺序

1. 保持当前合并弹跳，先真机确认观感。
2. 新增新块出现动画，范围只影响新 tile。
3. 如果新块动画稳定，再恢复滑动动画。
4. 如果滑动动画再次让规则观感变乱，立即回退到同步落位版本。
5. 等玩法和基础动效稳定后，再接鸡窝主题素材。
