# 石头剪刀布大作战绝地求生机制迁移设计

日期：2026-07-21
状态：第一版已实现，待真机验收
适用范围：新版 `entry/src/main/ets/games/rpsBattle/` 的 `团队占点` 模式增强机制

## 结论

`绝地求生` 是原版团队占点预设里的增强机制，对应开关 `lastStand`。它不是独立模式，也不是全局自由组合机制；在新版里应先作为 `团队占点` 的下一项增强能力接入。

新版迁移口径：

- `团队占点` 默认机制包最终应包含：中心据点、黑洞、绝地求生、能量道具、团队道具。
- 当前已实现：中心据点、黑洞、绝地求生、能量道具、团队道具。
- 下一步迁移：小心叛徒或众生平等的模式机制。
- 绝地求生默认开启，但和黑洞一样，允许玩家开局前关闭。
- 不影响经典乱斗、小心叛徒、众生平等的当前运行时。

绝地求生的核心作用是给弱势阵营一个短暂翻盘窗口：只剩 1 个单位时可能分裂；新版当前已把复活救援从原版的 `1 -> 0` 调整为剩余 `3` 个以内即可触发，团灭后也可以进入援军倒计时。

## 实施记录

2026-07-21 已完成第一版原生接入：

- 新增 `RpsBattleLastStandSystem.ets`，独立处理弱势分裂、复活印记、复活请求和机制快照。
- `团队占点` 的 `lastStand` 从 `upcoming` 移到 `enabled`，默认开启，开局前可关闭。
- `RpsBattleEngine.ets` 在黑洞、碰撞和据点之后调度绝地求生；团队占点不再按清场提前结束，优先据点达标或时间结算。
- `RpsBattleRenderer.ets` 用 Canvas 绘制复活印记倒计时和分裂/复活脉冲，不引入 ArkUI 帧内状态刷新。
- `RpsBattlePerformanceLogger.ets` 增加 `lastStandMs`、`lastStandEvents/frame`、`lastStandRevives/frame`。
- 保留本地辅助规则脚本用于快速回归；官方验证仍以 HarmonyOS Hypium / ArkXTest、DevEco 构建和真机验收为准。

2026-07-21 对抗节奏调参：

- 复活印记触发阈值从“阵营清零”提前到“阵营剩余 `3` 个以内且数量下降”；如果阵营已团灭，冷却结束后也可以再次排援军。

- 新增 HarmonyOS Hypium 测试文件 `entry/src/ohosTest/ets/test/RpsBattleLastStandSystem.test.ets`，覆盖剩余 3 个触发、快速复活、冷却内不重复和团灭后冷却援军。

## 原版行为

原版入口：

- UI 开关：`entry/src/main/ets/games/rps/RpsPage.ets` 的 `lastStand`
- 预设：`entry/src/main/ets/model/Presets.ets` 中 `zones` 默认 `lastStand = true`
- 逻辑：`entry/src/main/ets/model/GameEngine.ets`

原版团队占点默认开启：

```text
controlZones = true
blackHole = true
lastStand = true
powerUps = true
teamPowerUps = true
```

原版 `lastStand` 逻辑：

| 条件 | 行为 | 限制 |
| --- | --- | --- |
| 某阵营只剩 `1` 个单位 | 有概率触发绝地分裂，在源单位附近生成 1 个同阵营单位 | 每个阵营最多 1 次 |
| 某阵营从 `1` 变为 `0` | 有概率留下复活印记 | 每个阵营最多 1 次 |
| 复活印记到期 | `5s` 后在最后位置复活 1 个同阵营单位 | 印记触发后移除 |

原版关键数值：

| 参数 | 原版值 |
| --- | --- |
| 分裂触发条件 | 当前阵营数量等于 `1` |
| 分裂概率 | 每次更新 `0.018` |
| 分裂出生偏移 | 源单位 `x/y ± 18` |
| 分裂出生速度倍率 | `0.94` |
| 复活触发条件 | 当前数量 `0` 且上一帧数量 `1` |
| 复活概率 | `0.26` |
| 复活延迟 | `5000ms` |
| 复活出生位置 | 该阵营最后已知位置 |
| 复活出生速度倍率 | `1.12` |

## 原版问题

原版规则能制造戏剧性，但直接搬到新版有几个问题：

1. 分裂概率是“每次更新概率”，会受帧率影响。60fps 下只要最后一个单位存活一小段时间，触发概率会被放大。
2. 原版复活印记不阻止清场结算。也就是说某阵营刚留下复活印记，游戏仍可能已经因为只剩一个阵营而结束。
3. 原版事件提示依赖战报列表，新版目前没有完整长战报面板，需要用 HUD / Canvas / 轻量提示表达。
4. 新版已经做过性能优化，不能因为绝地求生在每帧里做大量数组筛选、对象创建或 ArkUI 状态刷新。

## 新版设计目标

- 对齐原版的“弱势方分裂 / 复活”玩法感受。
- 避免帧率相关概率，确保 30fps、60fps 下触发节奏一致。
- 只在 `团队占点` 中接入，保持四模式机制包清晰。
- 默认开启、开局前可关闭，遵守当前机制分层规则。
- 不让复活印记被清场结算立即吞掉。
- 保持性能可测，新增 `lastStandMs`、`lastStandEvents/frame` 等日志字段。

## 新版机制规则

### 1. 机制状态

`lastStand` 从 `upcoming` 迁移到 `enabled` 后：

```text
中心据点    核心    不可关闭
黑洞        增强    默认开，可关闭
绝地求生    增强    默认开，可关闭
能量道具    增强    默认开，可关闭
团队道具    增强    默认开，可关闭，依赖能量道具
```

规则解析：

```text
hasControlZone = true
hasBlackHole = 黑洞开关结果
hasLastStand = 绝地求生开关结果
```

当 `hasLastStand = false`：

- 不初始化绝地求生系统。
- 不执行绝地求生 update。
- 不绘制复活印记。
- 性能日志字段保持稳定，耗时和事件数为 0。

### 2. 分裂

新版建议把原版“每帧 0.018 概率”改成“按时间节流的概率判定”：

| 参数 | 建议值 |
| --- | --- |
| 触发条件 | 某阵营数量等于 `1` |
| 检查间隔 | 每个阵营 `1s` 最多判定一次 |
| 单次概率 | `0.66` |
| 成功后 | 在最后一个单位附近生成 1 个同阵营单位 |
| 出生偏移 | `±18vp`，并 clamp 到战场内 |
| 每阵营限制 | 最多成功 1 次 |

这样保留“有概率绝地反扑”的感觉，同时不依赖设备帧率。`0.66` 接近原版 `60fps` 下每秒至少触发一次 `0.018` 帧概率的体感；最后一个单位存活越久，越可能获得分裂机会，但不会每帧疯狂抽奖。

### 3. 复活

新版复活从“阵营清零后的最后机会”调整为“冷却援军”，用于增加团队占点的持续对抗可能性：

| 参数 | 建议值 |
| --- | --- |
| 触发条件 | 某阵营剩余 `3` 个以内并且数量下降；或阵营已为 `0` 且冷却结束 |
| 单次概率 | 稳定触发 |
| 复活延迟 | `2.2s` |
| 复活位置 | 该阵营最后已知位置 |
| 复活出生速度倍率 | `1.18` |
| 援军冷却 | 同阵营 `5s` |
| 同时存在限制 | 同阵营最多 1 个待触发复活印记 |

新版要补一个关键规则：**团队占点不再因为清场提前结束本局。**

补充规则：

- 低人数触发后进入复活印记倒计时，倒计时结束补回一个同阵营单位。
- 如果阵营已经团灭，只要绝地求生开着、冷却结束、据点和时间还未结算，仍可再次进入援军倒计时。
- 同阵营已有复活印记时，不重复创建第二个。
- 复活印记可以在阵营尚未清零时触发，表现上相当于一次弱势增援。

建议结算顺序：

```text
更新移动 / 碰撞 / 黑洞 / 据点
更新绝地求生：分裂、复活印记、到期复活
判断胜负
```

胜负判断时：

- `经典乱斗` 仍然按清场结算。
- `团队占点` 不按清场结算，只按据点达到 `100` 或时间结束结算。
- 如果某阵营团灭，绝地求生开启时可以等待冷却援军回场。

这样玩家能看懂“留下复活印记”这个事件，不会刚提示就立刻结束，也能把团队占点的重点从吞噬清场拉回据点争夺。

### 4. 与黑洞的关系

黑洞吞噬会让单位直接消失，不计入转阵。绝地求生应能感知黑洞造成的减员：

- 黑洞吞噬导致某阵营降到 `3` 个以内，可以触发复活判定。
- 黑洞吞噬导致某阵营只剩 `1`，后续可以触发分裂判定。
- 复活出来的单位仍然可能被黑洞吸引和吞噬。

为了避免黑洞刚吞掉复活单位导致体验过短，第一版不做特殊保护。真机如果感觉复活无意义，再考虑给复活单位 `0.8s` 免吸引或短暂无敌。

### 5. 与中心据点的关系

绝地求生不改变中心据点得分逻辑：

- 弱势方分裂后，新单位可以参与据点压力。
- 复活单位可以重新进入据点争夺。
- 据点达到 `100` 仍然立即结算，不等待复活印记。

原因：团队占点的核心目标是抢点，绝地求生只提供追赶机会，不能否定据点达标胜利。

## UI 设计

### 配置页

`团队占点` 的“本局机制”卡更新为：

```text
中心据点    中央据点涨到100获胜      核心
黑洞        随机危险点吸入单位        开 / 关
绝地求生    弱势阵营分裂或援军        开 / 关
能量道具    刷新加速、护盾、分裂道具  开 / 关
团队道具    拾取后全队共享效果        开 / 关
```

`绝地求生` 默认开。玩家关闭后，开局整局不运行该系统。

### 战斗页

第一版建议只做轻量表达，不做复杂大动画：

- 分裂触发：在分裂单位附近绘制短暂脉冲圈，HUD 或轻量提示显示“绝地分裂”。
- 复活印记：Canvas 上在最后位置绘制小型倒计时标记。
- 复活完成：复活位置出现同阵营单位和短暂亮圈。

不做：

- 全屏慢动作。
- 长时间遮罩。
- 资源型插画。
- 每帧 ArkUI 文本列表刷新。

### 结算和战报口子

首版只在结果数据里预留统计字段，后续战报再展示：

- `lastStandSplits`
- `lastStandRevives`
- `lastStandExpiredMarks`

如果本轮不做完整战报，至少保证性能日志和调试日志可看。

## 技术拆分建议

新增：

```text
entry/src/main/ets/games/rpsBattle/RpsBattleLastStandSystem.ets
```

职责：

- 持有每阵营分裂使用状态、援军冷却和复活印记状态。
- 记录每阵营最后位置。
- 根据当前 counts、上一帧 counts 和援军冷却判断触发。
- 返回需要新增的单位请求和复活印记快照。
- 不直接操作 ArkUI 状态，不负责 Canvas 绘制。

建议新增类型：

```text
RpsBattleReviveMark
RpsBattleLastStandState
RpsBattleSpawnRequest
```

引擎接入：

- `RpsBattleEngine.reset()` 根据 `hasLastStand` 初始化或清空系统。
- `RpsBattleEngine.update()` 在黑洞、碰撞和据点之后调度绝地求生。
- `RpsBattleEngine.resultAt()` 在团队占点中不再按清场提前结束，改为据点或时间结算。
- `RpsBattleSnapshot` 增加绝地求生快照，供 Canvas 渲染。

渲染接入：

- `RpsBattleRenderer.draw()` 增加复活印记和分裂脉冲绘制。
- 复活印记绘制只使用 Canvas，不引入 ArkUI 组件。

配置接入：

- `RpsBattleModeConfig.ets` 中 `lastStand.state` 从 `upcoming` 改为 `enabled`。
- `lastStand.canDisable = true`。
- `RPS_BATTLE_MODE_RULES.zones.hasLastStand = true`。

## 性能要求

绝地求生系统是低频事件系统，不应成为每帧主要开销。

实现要求：

- 不在每帧对 `entities` 做多次 `filter`。
- 复用引擎已有 counts 和 faction list，避免重复扫描。
- 每阵营最多维护少量状态，固定数组优先。
- 复活印记最多 3 个，不需要复杂空间索引。
- Canvas 绘制只画小圈和倒计时，不做重阴影、大渐变或粒子。

性能日志建议新增：

```text
lastStandMs
lastStandEvents/frame
lastStandRevives/frame
```

期望：

- 大局 36 实体下，`lastStandMs` 常态低于 `0.2ms`。
- 没有触发事件时，`lastStandEvents/frame = 0`。
- 关闭绝地求生后，`lastStandMs` 接近 `0`。

## 测试计划

规则测试：

1. `hasLastStand = false` 时不触发分裂、不留下复活印记。
2. 某阵营只剩 1 个且达到检查时间时，有概率触发分裂。
3. 每个阵营最多分裂一次。
4. 某阵营剩余 3 个以内且数量下降时，留下复活印记。
5. 复活印记约 2.2 秒后生成同阵营单位。
6. 同阵营已有复活印记时不重复创建。
7. 团灭阵营在冷却结束后可以再次进入援军倒计时。
8. 据点达到 100 时仍然立即结算。
9. 关闭绝地求生后，性能日志字段为 0。
10. 团队占点不再按清场提前结束。

真机验收：

- 团队占点默认显示绝地求生为开启。
- 关闭绝地求生后开局，战斗中不会出现分裂或复活印记。
- 开启绝地求生后，弱势阵营只剩 1 个时能看到分裂反馈。
- 阵营降到 3 个以内后如果留下印记，能看到约 2.2 秒倒计时和复活。
- 阵营团灭后，只要绝地求生开启且冷却结束，仍能看到援军回场。
- 团队占点本局结束应更多来自据点到 100 或时间到，而不是清场。
- 大局模式下 `RpsBattlePerf` 没有明显退化。

## 风险和待确认

1. 分裂触发概率需要真机调手感。当前为 `1s` 间隔、`0.66` 单次概率，不是最终数值。
2. 团队占点不按清场提前结束后，时间局可能明显变长，需要真机确认节奏。
3. 据点达标不等待复活印记，保持团队占点目标权威。
4. 是否给复活单位短暂无敌，首版暂不做，等测试感受。
5. 文案“绝地求生”是否后续改成更自有表达，当前先沿用原版模式机制名。
