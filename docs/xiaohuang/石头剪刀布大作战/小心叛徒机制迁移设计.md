# 石头剪刀布大作战小心叛徒机制迁移设计

日期：2026-07-23
状态：第一版已实现，待真机验收
适用范围：新版 `entry/src/main/ets/games/rpsBattle/` 的 `小心叛徒` 模式

## 实施记录

2026-07-23 已按手机短局反馈完成第二轮平衡调优：

- 首次叛徒从原版 `20s` 后触发，调整为开局 `0.8s..1.5s` 内出现首个标记。
- 叛徒倒戈延迟保持 `2s`，所以玩家能在约 `2.8s..3.5s` 看到首次转阵。
- 常规叛徒间隔从 `10s` 调整为 `6.8s`，更适合手机短局。
- 新增 `小心叛徒` 专用早期清场保护：开局前 `12s` 只剩一个阵营时，不立即清场，优先安排救场叛徒。
- 救场叛徒每局最多 `2` 次，且要求唯一存活阵营至少还有 `2` 个单位，避免无限拖局或只剩一个单位时来回改名。
- 救场叛徒优先投靠当前唯一存活阵营的克制方，让已归零阵营有机会回场。
- 缩圈改为开局显示轻危险边界，`4s` 后才慢速推进，速度从 `9px/s` 降到 `4.5px/s`。
- 黑洞在 `小心叛徒` 模式下首次出现调整为 `6s..10s`；团队占点仍沿用原来的 `9s..16s`，不扩大影响面。
- `RpsBattleRenderer.ets` 只使用 Canvas 原生绘制缩圈提示，不引入 WebView、HTML 或跨端依赖。
- Hypium 测试同步覆盖开局叛徒、救场叛徒、清场保护窗口和缩圈开局提示。

2026-07-23 已完成第一版：

- `叛徒` 已设为 `小心叛徒` 模式核心机制，默认开启且不可关闭。
- `缩圈`、`黑洞` 已设为默认开启的可关闭机制。

- 新增 `RpsBattleTraitorSystem.ets`：负责叛徒标记、早期叛徒、常规触发、倒计时叛变请求和脉冲快照。
- 新增 `RpsBattleShrinkSystem.ets`：负责缩圈延迟、padding 推进、边界保护和快照。
- `RpsBattleEngine.ets` 接入叛徒转阵、短暂暴走、缩圈边界和死斗清场结算。
- `RpsBattleRenderer.ets` 接入缩圈暗角、叛徒倒计时环、目标阵营提示和叛变脉冲。
- `RpsBattlePerformanceLogger.ets` 增加 `shrinkMs`、`traitorMs`、`traitorMarks/frame`、`traitorConversions/frame`。
- 新增 Hypium 测试文件 `RpsBattleTraitorSystem.test.ets`、`RpsBattleShrinkSystem.test.ets`。

## 目标

`小心叛徒` 不是新设计一套玩法，而是把旧版 `rps` 里已有的模式机制迁移到新版 `rpsBattle`。

本模式第一目标是制造不稳定死斗：边界持续压缩、黑洞打乱站位、叛徒倒戈改写阵营关系，最终仍以清场为主要结算。

实现必须遵守：

- 使用 HarmonyOS 原生 ArkTS / ArkUI / Canvas，不引入 WebView、HTML 或跨端运行壳。
- 按 `docs/鸿蒙原生开发规范.md` 拆分：页面只组装 UI，规则放到独立系统文件，类型放到 `RpsBattleTypes.ets`。
- 不直接复制旧版 `GameEngine.ets` 的代码，只把旧版行为作为规则参考，在新版引擎里重写。
- 纯规则必须可测试，后续实现时优先补 HarmonyOS 官方 Hypium / ArkXTest 测试。
- 构建验证使用项目规范命令 `./scripts/build-app.sh --stacktrace`；本地 DevEco 路径需要时使用 `DEVECO_STUDIO_APP` 环境变量。

## 原版模式来源

旧版预设来自 `entry/src/main/ets/model/Presets.ets`：

```text
key: traitor
name: 小心叛徒
subtitle: 缩圈 · 黑洞 · 叛徒
summary: 边界收缩，叛徒定时改写战局。
```

旧版 `presetOptions('traitor')` 打开的能力：

```text
shrink = true
blackHole = true
traitor = true
deathmatch = true
```

也就是说，原版“小心叛徒”不是限时得分模式，而是带事件干扰的死斗清场模式。

新版当前状态：

- `RpsBattleModeConfig.ets` 已有 `traitor` 模式入口。
- 机制列表为 `叛徒 / 缩圈 / 黑洞`。
- `叛徒` 是核心机制，不可关闭。
- `缩圈`、`黑洞` 是默认开启的可关闭机制。
- 黑洞系统复用团队占点已完成的移动端适配，并在缩圈开启时优先生成在当前安全边界内。

## 原版叛徒机制

旧版核心逻辑在 `entry/src/main/ets/model/GameEngine.ets`：

```text
updateTraitors
queueTraitor
applyTraitorMarks
hasTraitorMark
```

### 触发时间

原版规则：

- 开局先设置 `nextTraitorAt = roundStart + 20s`。
- 常规触发从开局 `20s` 后开始。
- 之后每 `10s` 检查一次。
- 开局 `10s..15s` 内，如果场上只剩两个阵营，会触发一次早期叛徒。
- 早期叛徒每局最多一次。

### 标记数量

常规触发时：

- 从当前存活阵营里随机选阵营。
- 每次触发 `1` 或 `2` 个叛徒，概率约各 `50%`。
- 实际数量不超过当前存活阵营数。
- 同一轮触发不会重复选择同一个来源阵营。

### 候选单位

旧版选择规则：

- 来源阵营必须有存活单位。
- 候选单位不能已经有叛徒标记。
- 从候选单位中随机选一个。

注意：如果单位被标记后又被普通转化，只要实体还活着，原版仍会在倒计时结束后把它改到叛徒目标阵营。这会带来戏剧性反转，应保留。

### 目标阵营

原版目标阵营是：

```text
随机选择一个不同于来源阵营的阵营
```

它不要求目标阵营当前存活。因此叛徒可能把已归零阵营重新带回场上。

这个行为对“小心叛徒”的翻盘感很重要，迁移时不应改成“只能投靠当前存活阵营”，除非真机测试证明结算节奏失控。

### 倒计时和叛变

原版常量：

```text
TRAITOR_DELAY = 2000ms
```

行为：

- 被选中的单位出现紫色倒计时圈。
- `2s` 后单位阵营改为目标阵营。
- 叛变后短暂暴走约 `0.9s`。
- 如果倒计时期间单位死亡或被黑洞吞噬，标记消失，不产生叛变。

### 视觉表现

旧版 `CanvasRenderer.drawTraitorMarks()`：

- 在被标记单位外圈画紫色进度圆环。
- 进度按 `startAt -> triggerAt` 线性推进。
- 到点后单位变阵营，由实体自身颜色体现身份变化。

新版建议保留紫色倒计时圈，并增加更清晰的目标提示：

- 外圈主色仍用危险紫。
- 倒计时末段可以短暂闪烁。
- 圈内或外圈小段可以用目标阵营颜色暗示，但不要做大面积遮挡。

## 缩圈机制理解

原版 `小心叛徒` 同时开启缩圈。

旧版规则：

- 开局 `7s` 后开始缩圈。
- 每帧按固定速度增加 `shrinkPadding`。
- 最大压缩量为短边 `28%`。
- 边界不会无限压到中心，左右/上下都有最小可玩区域保护。
- 每约 `5.2s` 发一次“缩圈正在推进”事件。

迁移判断：

- 缩圈是“小心叛徒”的空间压力来源。
- 它会把单位压向中间，提高碰撞和叛徒影响力。
- 首版可以先做叛徒本体，再做缩圈；但最终模式应恢复 `缩圈 + 叛徒 + 黑洞 + 死斗` 的组合。

## 黑洞机制关系

原版 `小心叛徒` 默认开启黑洞。

新版已有 `RpsBattleBlackHoleSystem.ets`：

- 已做移动端半径和吸引范围上限。
- 已支持按模式规则开启/关闭。
- 已支持避开中心据点生成。

迁移到 `小心叛徒` 时建议复用同一套系统，不另写黑洞。

差异点：

- `小心叛徒` 没有中心据点，所以黑洞不需要避开据点。
- 如果后续缩圈压缩边界，黑洞生成点需要落在当前安全边界内。
- 黑洞可以作为可关闭机制，默认开启。

## 新版机制分层

建议 `小心叛徒` 的机制分层如下：

| 机制 | 状态 | 默认 | 是否可关 | 理由 |
| --- | --- | --- | --- | --- |
| 基础追逃 | 核心 | 开 | 不可关 | 所有模式的底层玩法 |
| 转化 | 核心 | 开 | 不可关 | 石头剪刀布大作战核心 |
| 叛徒 | 核心 | 开 | 不可关 | 模式名称和主玩法来源 |
| 缩圈 | 可选机制 | 开 | 可关 | 原版默认开启，但可以作为压力调节项 |
| 黑洞 | 可选机制 | 开 | 可关 | 原版默认开启，新版已支持机制开关 |

不建议允许关闭 `叛徒`。如果关闭叛徒，这个模式会退化成“缩圈黑洞乱斗”，与模式名称不一致。

## 结算规则

原版是死斗模式：

- 只剩一个阵营时清场结算。
- 不按时间结算。
- 缩圈和黑洞只是加快决出胜者。
- 叛徒事件用于打断稳定优势。

新版首版建议保持：

```text
traitor 模式默认 deathmatch
aliveCount <= 1 时 clear 结算
```

是否等待叛徒倒计时完成：

- 原版不会因为已有叛徒标记而阻止清场。
- 如果标记期间其他阵营被吞完，对局仍可能清场结束。
- 首版按原版保留这个行为。
- 如果真机测试发现叛徒经常“还没来得及发生就结束”，再考虑加入 `pending traitor grace`，但那属于新版调参，不是原版复刻。

## 新版代码拆分建议

按开发规范，不把规则塞进 `RpsBattlePage.ets` 或 ArkUI `build()`。

建议新增/修改：

```text
entry/src/main/ets/games/rpsBattle/
  RpsBattleTypes.ets
  RpsBattleModeConfig.ets
  RpsBattleEngine.ets
  RpsBattleRenderer.ets
  RpsBattlePerformanceLogger.ets
  RpsBattleTraitorSystem.ets
  RpsBattleShrinkSystem.ets
```

### RpsBattleTypes.ets

新增类型：

```ts
export interface RpsBattleTraitorMark {
  entityId: number;
  fromFaction: RpsBattleFaction;
  toFaction: RpsBattleFaction;
  startAtSec: number;
  triggerAtSec: number;
}

export interface RpsBattleTraitorPulse {
  entityId: number;
  faction: RpsBattleFaction;
  x: number;
  y: number;
  startedAtSec: number;
}

export interface RpsBattleTraitorState {
  marks: RpsBattleTraitorMark[];
  pulses: RpsBattleTraitorPulse[];
  triggeredCount: number;
}
```

`RpsBattleModeRules` 建议增加：

```ts
hasShrink: boolean;
hasTraitor: boolean;
```

`RpsBattleEntity` 建议增加：

```ts
traitorRageUntilSec: number;
```

不建议复用 `speedBoostUntilSec` 表达叛徒暴走。道具加速和叛徒暴走语义不同，拆开更利于调参和测试。

### RpsBattleTraitorSystem.ets

职责：

- 管理下一次叛徒触发时间。
- 管理早期叛徒是否已经触发。
- 管理叛徒标记列表。
- 生成待执行叛变请求。
- 维护脉冲和统计计数。

建议 API：

```text
reset(elapsedSec)
clear()
resize(ratioX, ratioY)
update(entities, counts, elapsedSec, random) -> RpsBattleTraitorConversion[]
snapshot() -> RpsBattleTraitorState | null
hasPendingMarks() -> boolean
```

`RpsBattleTraitorSystem` 不直接修改实体阵营，因为新版引擎有阵营缓存和计数缓存。系统只返回请求，由 `RpsBattleEngine` 统一调用类似 `convertEntity()` 的方法处理，避免缓存错位。

### RpsBattleShrinkSystem.ets

职责：

- 根据棋盘尺寸、经过时间和 `dt` 计算当前安全边界。
- 输出 `padding / bounds / progress`。
- 不触碰 ArkUI 状态，不绘制 UI。

建议先不要把缩圈逻辑散落到 `RpsBattleEngine.resolveBounds()` 里。引擎只读取 shrink system 的当前边界，实体反弹仍走现有边界处理。

### RpsBattleEngine.ets

接入顺序建议：

```text
1. 更新缩圈边界
2. 更新黑洞
3. 更新叛徒标记和到期叛变
4. 移动实体
5. 道具拾取
6. 碰撞和转化
7. 模式结算
```

这样更接近原版：叛徒到点后先换阵营，再参与本帧移动和碰撞。

需要注意：

- 叛变不是普通碰撞转化，但仍要更新阵营缓存。
- 叛变后给短暂暴走速度上限或视觉脉冲。
- 被黑洞吞掉的实体，相关叛徒标记要清理。
- 如果叛徒目标阵营已经归零，叛变后该阵营重新进入场上。

### RpsBattleRenderer.ets

新增绘制：

- `drawTraitorMarks(ctx, traitorState, elapsedSec, themeKey)`。
- 标记圆环绘制在实体上方，避免被实体遮挡。
- 叛变脉冲用短时扩散环，不新增 ArkUI 动画。
- 缩圈边界建议用 Canvas 半透明边框或暗角，不用 ArkUI 覆盖层每帧刷新。

## UI 设计建议

开局配置页：

- `小心叛徒` 卡片选中后展示：
  - 核心：叛徒
  - 默认机制：缩圈、黑洞
- `叛徒` 标记为核心，不显示开关。
- `缩圈 / 黑洞` 显示开关，默认开启。
- 在机制未实现前继续显示“后续”，不能伪装成可生效。

战斗 HUD：

- 顶部或状态条显示 `叛徒倒计时` 不是必须，因为标记在单位身上更直观。
- 最近事件可以显示：
  - `石头阵营出现叛徒`
  - `叛徒加入 剪刀`
  - `缩圈正在推进`
- 不新增大型提示弹层，避免遮挡战场。

## 性能要求

叛徒机制本身不应成为性能瓶颈。

实现要求：

- 每帧最多处理少量标记，`marks` 数组通常不超过 `2`。
- 候选实体筛选不要使用频繁分配的大量临时数组；优先循环计数后按随机下标选中。
- 不在 ArkUI `@State` 中保存完整叛徒系统对象。
- Canvas 绘制只读取 snapshot，不触发 ArkUI 组件树重建。
- 性能日志建议增加：
  - `traitorMs`
  - `traitorMarks/frame`
  - `traitorConversions/frame`
  - `shrinkMs`

## 风险和决策点

### 是否允许关闭叛徒

建议不允许。叛徒是模式核心，关闭后模式语义不成立。

### 是否等待 pending 叛徒再清场

首版按原版：不等待。这样更接近旧版死斗节奏。

如果测试时叛徒存在感不足，再单独设计“叛徒保护窗口”，例如 pending 标记剩余小于 `1s` 时暂缓清场。

## 手机短局平衡反馈

2026-07-23 真机方向反馈：手机屏幕小，单位密度高，碰撞频率高，`小心叛徒` 第一版容易在机制触发前就被基础吞噬清场。

这说明不能只照搬原版桌面/大场景触发时间。当前第一版的问题是：

- `缩圈` 原版节奏约 `7s` 后开始，手机上可能还没开始就已经清场。
- `黑洞` 当前首次出现约 `9s..16s`，在手机短局中经常来不及上场。
- `叛徒` 常规触发约 `20s`，明显晚于手机小局清场时间。
- 死斗清场结算优先级太直接，模式特色会被基础转化吞噬抢走。

因此后续需要把 `小心叛徒` 调成手机优先的短局节奏。核心原则：

```text
叛徒机制必须早于清场发生
早期清场需要被叛徒机制打断
缩圈和黑洞是氛围/扰局，不是主要救场
```

### 开局危险态建议

不建议 `0s` 同时强触发叛徒、缩圈和黑洞。三者一起爆发会让开局过乱，也容易由随机事件直接决定胜负。

建议改成分层开局危险态：

| 时间 | 机制 | 建议行为 |
| --- | --- | --- |
| `0s` | 缩圈氛围 | 先显示轻微危险边界或预告，不立刻强压 |
| `0.8s..1.5s` | 首个叛徒标记 | 从随机存活阵营或当前优势阵营标记 1 个叛徒 |
| `2.8s..3.5s` | 首个叛变生效 | 让玩家开局就看到模式核心 |
| `4s` | 缩圈慢速推进 | 比原版更早，但速度不能太猛 |
| `6s..10s` | 黑洞出现 | 作为扰局事件，仍保留移动端上限 |

### 早期清场保护

建议新增 `traitor` 模式专用保护窗口：

| 参数 | 建议值 |
| --- | --- |
| 保护窗口 | 开局前 `12s` |
| 触发条件 | 只剩一个阵营，且叛徒机制开启 |
| 行为 | 暂不立即清场，强制安排一次救场叛徒 |
| 来源阵营 | 当前唯一存活或明显优势阵营 |
| 目标阵营 | 优先选择已归零阵营 |
| 次数限制 | 每局 `1..2` 次 |

救场叛徒不是无限复活，而是让“小心叛徒”的核心机制有机会打断早期滚雪球。

如果真机仍然过快，可以继续调两个旋钮：

- 保护窗口从 `12s` 提到 `15s`。
- 首次叛徒从 `1.5s` 提前到 `0.8s`。

### 缩圈和黑洞的定位

缩圈在手机上要谨慎，因为缩圈会提高碰撞密度，可能进一步加快吞噬。

建议：

- 缩圈可以早一点出现，但必须慢速推进。
- 缩圈首轮主要提供紧张感，不作为平衡主手段。
- 黑洞不要太早太强，否则会变成随机清场。
- 黑洞仍默认开启、可关闭，并继续使用移动端范围上限。

### 下一版推荐目标

下一版不应只做“时间提前”，而应做完整的短局平衡：

1. 首次叛徒提前到 `0.8s..1.5s`。
2. 常规叛徒间隔从 `10s` 缩短到 `6s..8s`。
3. 前 `12s` 增加清场保护。
4. 清场保护触发时强制安排救场叛徒。
5. 缩圈提前但降速。
6. 黑洞保留较晚出现和移动端上限。

验收重点：

- 手机标准局不应在叛徒出现前结束。
- 玩家能在前几秒看到“小心叛徒”的核心体验。
- 早期清场会被叛徒打断，但不会无限拖局。
- 缩圈和黑洞不会把对局变成纯随机事件。

### 叛徒是否投靠已死亡阵营

建议保留原版行为：可以。这个机制能让已归零阵营突然回场，是“小心叛徒”的戏剧性来源。

### 缩圈是否第一轮一起做

不建议。叛徒本体先做清楚，缩圈再接。两个机制同时做，真机反馈不容易判断问题来自哪里。

## 当前结论

`小心叛徒` 的核心应该按原版理解为：

```text
死斗清场 + 叛徒倒戈 + 缩圈压迫 + 黑洞扰局
```

当前代码已进入“手机短局可玩性调优”阶段，不再是只迁移原版参数。后续真机观察重点：

- 开局是否稳定看得到第一次叛徒标记和倒戈。
- 早期滚雪球是否会被救场叛徒打断。
- 救场次数是否够用，但不会把对局拖成无法结束。
- 缩圈是否只是制造压力，而不是进一步加速清场。
- 黑洞是否能扰局，但不会在手机端直接决定胜负。
