# 石头剪刀布大作战基础玩法 MVP 设计

日期：2026-07-08
状态：新版 `rpsBattle` 首个可玩版本设计

## 一、目标

新版 `rpsBattle` 首个版本先做“最基本但完整”的石头剪刀布大作战：

- 有独立入口，可以从大厅进入新版游戏。
- 有选择界面，玩家先选择基础模式和支持阵营，再开始。
- 有一局完整基础乱斗：三阵营自动移动、追击、碰撞、转化、结束。
- 有暂停、返回大厅、重新开始、基础结果展示。
- 只使用鸿蒙原生 ArkTS / ArkUI 代码，不引入 Node `.mjs` 作为玩法测试或运行依赖。

首版不追求旧版所有机制，不做黑洞、叛徒、悬赏、缩圈、多局赛制和战报长面板。先把主循环和可玩闭环做稳。

## 二、代码范围

新版代码包：

```text
entry/src/main/ets/games/rpsBattle/
```

当前已存在：

- `RpsBattlePage.ets`：页面骨架、顶部栏、后续承载选择界面和战场。
- `RpsBattleTheme.ets`：新版页面和顶部栏主题。

建议新增：

- `RpsBattleTypes.ets`：阵营、实体、对局状态、选项、结算结果类型。
- `RpsBattleRules.ets`：循环克制、胜负判断、初始配置生成等纯规则。
- `RpsBattleEngine.ets`：基础自动乱斗模拟。
- `RpsBattleBoard.ets`：战场绘制或战场组件。
- `RpsBattleSetupPanel.ets`：选择界面。
- `RpsBattleResultOverlay.ets`：结果层。

首版可以先少拆一点，但不要把规则、绘制、选择界面、结算全部堆进 `RpsBattlePage.ets`。

## 三、页面状态

首版页面状态建议收敛为四个：

| 状态 | 名称 | 说明 |
| --- | --- | --- |
| `setup` | 选择界面 | 进入新版游戏后的默认状态 |
| `playing` | 对局中 | 自动乱斗进行中 |
| `paused` | 暂停 | 保留当前战场，可继续或回大厅 |
| `result` | 结算 | 展示赢家、支持阵营是否猜中、再来一局 |

页面顶部栏始终存在，样式继续沿用鸡窝水果摊：

- 顶部栏占据页面顶部，不做悬浮玻璃卡片。
- 左侧固定“大厅”按钮。
- 中间标题“石头剪刀布大作战”。
- 右侧在 `setup/result` 可留空，在 `playing/paused` 可放“暂停/继续”按钮。

## 四、选择界面

选择界面是首版必须有的入口，不直接开打。

### 4.1 必选内容

首版选择界面只放三组控制：

| 控制项 | 默认值 | 说明 |
| --- | --- | --- |
| 模式 | 经典乱斗 | 首版只有一个可用模式，其它模式先不展示 |
| 支持阵营 | 石头 | 玩家选择自己支持的一方 |
| 规模 | 标准 | 控制三方初始人数 |

规模建议：

| 规模 | 初始人数 | 目标 |
| --- | --- | --- |
| 小局 | `5/5/5` | 快速看懂规则 |
| 标准 | `8/8/8` | 默认体验 |
| 大局 | `12/12/12` | 画面更热闹 |

### 4.2 界面布局

从上到下：

1. 顶部栏。
2. 模式标题区：显示“经典乱斗”和一句短说明。
3. 支持阵营选择：石头、剪刀、布三段式按钮。
4. 规模选择：小局、标准、大局三段式按钮。
5. 开始按钮：点击后创建对局并进入 `playing`。

选择界面不要放大段说明文字。规则提示应短：

```text
石头克剪刀，剪刀克布，布克石头
```

## 五、基础玩法

### 5.1 阵营

三阵营固定：

| id | 阵营 | 图标 | 克制 |
| --- | --- | --- | --- |
| `rock` | 石头 | `✊` | 剪刀 |
| `scissors` | 剪刀 | `✌` | 布 |
| `paper` | 布 | `✋` | 石头 |

首版只做固定主题，不做主题切换。

### 5.2 单位

每个单位至少包含：

| 字段 | 说明 |
| --- | --- |
| `id` | 唯一编号 |
| `faction` | 当前阵营 |
| `x/y` | 坐标 |
| `vx/vy` | 速度 |
| `radius` | 碰撞半径 |

首版不做护盾、怒气、道具、障碍。

### 5.3 移动逻辑

每帧执行：

1. 找到最近的猎物阵营单位，朝猎物移动。
2. 找到最近的天敌阵营单位，距离过近时轻微躲避。
3. 加一点随机游走，避免单位全部挤成直线。
4. 限制最大速度。
5. 处理边界反弹。
6. 处理碰撞和转化。

为了首版稳定，可先只做最近猎物追击和边界反弹；躲避和随机游走可以作为第二步补强。

### 5.4 碰撞转化

不同阵营单位发生碰撞时：

1. 判断双方克制关系。
2. 胜方不变。
3. 败方转化为胜方阵营。
4. 记录一次转化，用于结果展示或简单计数。

同阵营碰撞只做分离，不改变阵营。

### 5.5 结束条件

首版采用双结束条件：

| 条件 | 说明 |
| --- | --- |
| 清场胜利 | 场上只剩一个阵营 |
| 限时胜利 | `60s` 到时后人数最多的阵营获胜 |

如果限时结束时并列：

- 结果显示“平局”。
- 支持阵营不算胜利。

## 六、对局 HUD

对局中至少显示：

- 三方当前人数。
- 倒计时。
- 玩家支持阵营。

HUD 可以放在顶部栏下方，不要遮挡战场主体。首版不需要复杂战报流。

## 七、结果界面

结果层展示：

- 赢家阵营或平局。
- 玩家支持阵营。
- 是否支持成功。
- 三方最终人数。
- 按钮：再来一局、重新选择、回大厅。

`recordScore` 首版可以记录赢家最终人数或本局最高转化数。为了和旧高分标签“最高存活数”一致，建议先记录赢家最终人数。

## 八、首版验收标准

完成后应满足：

1. 大厅最后一个入口可以进入新版 `rpsBattle`。
2. 进入后先看到选择界面，而不是直接开局。
3. 顶部栏样式与鸡窝水果摊一致：占据顶部、左侧大厅、中间标题。
4. 玩家可以选择支持阵营和对局规模。
5. 点击开始后能看到三阵营单位自动乱斗。
6. 单位碰撞后能按石头剪刀布规则转化。
7. 清场或倒计时结束后进入结果界面。
8. 可以再来一局、重新选择、回大厅。
9. 构建通过 `./scripts/build-app.sh --stacktrace`。
10. 不新增 `scripts/test-rps-battle-rules.mjs`，不为了 Node 测试改变 ArkTS 类型写法。

## 九、暂不做

- 下注积分。
- 黑洞、悬赏、缩圈、叛徒、灭霸响指。
- 多局赛制。
- 自定义主题和自定义阵营名。
- 长战报和精彩事件回放。
- Node `.mjs` 规则测试。

这些内容后续从现有文档组和旧版代码中逐步迁移，不进入基础 MVP。
