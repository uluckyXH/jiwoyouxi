# 石头剪刀布大作战黑洞机制开关与移动端适配设计

日期：2026-07-16
状态：讨论稿，待确认后实施
适用范围：新版 `entry/src/main/ets/games/rpsBattle/`

## 背景

`团队占点` 当前已经接入中心据点和黑洞。真机测试反馈两个问题：

1. 黑洞在手机端有明显“全屏性质”，吸引范围过大，玩家感觉不是局部危险点，而是整局被黑洞接管。
2. 机制包虽然默认开启，但玩家希望可以手动关闭已实现的增强机制，尤其是黑洞。最新口径是：除模式核心机制外，其他已实现机制都应该允许开局前关闭。

当前实现里，黑洞沿用了原版绝对数值：

| 参数 | 当前值 |
| --- | --- |
| 初始核心半径 | `24` |
| 初始吸引范围 | `82` |
| 成长倍率 | `1.3` |
| 核心半径上限 | `116` |
| 吸引范围上限 | `310` |

在手机竖屏战场里，`310` 的吸引半径意味着直径可达到 `620`，很容易覆盖大部分甚至全部横向可玩区。它保留了原版成长逻辑，但不适合直接套到新版手机画布。

## 设计目标

- 保留原版团队占点的默认机制包：中心据点默认开启，黑洞默认开启。
- 已实现的增强机制允许玩家在开局前关闭，不只针对黑洞。
- 黑洞仍然是局部危险事件，不变成全屏吸入。
- 关闭黑洞后，团队占点仍然成立：中心据点照常运行、结算和 HUD 不受影响。
- 所有实现继续遵守 HarmonyOS 原生方案：ArkTS / ArkUI / Canvas，不引入 WebView 或 HTML 游戏页。
- 每帧性能开销可控，关闭机制后不让对应系统空转。

## 机制开关分层

机制不应该全部同等对待。新版按三类处理：

| 类型 | 示例 | 默认 | 是否允许关闭 | 说明 |
| --- | --- | --- | --- | --- |
| 核心机制 | 团队占点的 `中心据点` | 开启 | 不允许 | 关闭后模式目标会失效 |
| 已实现增强机制 | `黑洞`，后续的绝地/道具等 | 开启 | 允许 | 对齐原版默认体验，但玩家可以关掉 |
| 后续机制 | `绝地求生`、`能量道具`、`团队道具` | 目标默认开启 | 当前不允许 | 未实现前只展示“后续”，不能点击；实现后移入“已实现增强机制” |

因此 `团队占点` 当前应表现为：

```text
中心据点  核心  已锁定
黑洞      默认开  可关闭
绝地求生  后续  不可用
能量道具  后续  不可用
团队道具  后续  不可用
```

## UI 方案

### 入口位置

沿用当前选择页的 `模式机制` 区域，不额外新增一个复杂设置页。

选中 `团队占点` 后，机制区显示当前模式相关机制。每一行只承担一个明确动作：

- 核心机制：显示锁定态，不给开关。
- 已实现增强机制：显示 `Toggle/Switch`，默认打开，可以关闭。
- 后续机制：显示 `后续` 标签，整行禁用，不可点击。

### 文案建议

| 机制 | 主文案 | 副文案 | 右侧状态 |
| --- | --- | --- | --- |
| 中心据点 | `中心据点` | `中央据点涨到100获胜` | `核心` |
| 黑洞 | `黑洞` | `随机危险点吸入单位` | 开关 |
| 绝地求生 | `绝地求生` | `弱势阵营分裂或复活` | `后续` |
| 能量道具 | `能量道具` | `刷新加速、护盾、分裂道具` | `后续` |
| 团队道具 | `团队道具` | `拾取后全队共享效果` | `后续` |

### 交互规则

- 黑洞开关只在开局前调整。
- 点击 `开始乱斗` 后，本局使用当前开关状态。
- 战斗中不提供黑洞开关，避免运行中状态不一致。
- 切换玩法模式时，机制开关回到该模式默认值。
- 后续机制实现后，如果属于当前模式默认包，默认打开；只要不是该模式的核心机制，就允许玩家在开局前关闭。

## 黑洞移动端适配方案

### 问题判断

当前黑洞不是“无限变大”，它已有固定上限：

- 核心半径最多 `116`。
- 吸引范围最多 `310`。

但固定上限没有考虑手机战场短边，所以在手机端会看起来像全屏。问题不在成长逻辑本身，而在上限没有做视口适配。

### 推荐数值

保留原版数值作为大屏上限，同时增加战场短边比例上限：

| 参数 | 建议规则 |
| --- | --- |
| 初始核心半径 | `24` |
| 初始吸引范围 | `82` |
| 成长倍率 | `1.3` |
| 动态核心半径上限 | `min(116, shortSide * 0.16)` |
| 动态吸引范围上限 | `min(310, shortSide * 0.42)` |
| 最小可用吸引上限 | 不低于初始吸引范围 `82` |

这样在手机短边约 `360` 时：

```text
核心半径上限约 57
吸引范围上限约 151
吸引直径约 302，不再覆盖全屏
```

在平板或宽屏上：

```text
继续接近原版 116 / 310 的上限
```

### 行为边界

- 黑洞仍然会吞噬单位后变大。
- 黑洞不会无限成长。
- 黑洞不会全屏吸入。
- 黑洞吸引范围是危险区，不是必死区；只有进入核心才会死亡。
- 黑洞吞噬是减员，不是转阵，不增加 `totalConversions`。

### 出现位置

当前黑洞随机出现在战场内并避开边缘。移动端建议再加两条约束：

1. 不贴边：继续保留边缘安全距离。
2. 不直接盖住中心据点核心：首次出现点不应落在中心据点圆内。

黑洞仍然可以出现在据点附近，保持原版“打破堆叠”的作用，但不要一出生就覆盖整个据点核心，让玩家有反应空间。

## 运行规则设计

### 配置状态

模式配置仍然提供默认机制包：

```ts
zones default:
  controlZone = true
  blackHole = true
  lastStand = future
  powerUps = future
  teamPowerUps = future
```

玩家开关只作为本局覆盖项：

```ts
selectedMechanics:
  controlZone = true
  blackHole = true | false
```

规则解析时必须保护核心机制：

- `controlZone` 不能被关闭。
- `upcoming` 机制不能被开启。
- 只有 `state = enabled && canDisable = true` 的机制允许玩家覆盖；后续新增机制实现后应默认设置为可关闭，除非它被定义为模式核心。

### 引擎接入

开局创建 `RpsBattleEngine` 时，把解析后的规则传入：

```text
hasControlZone = true
hasBlackHole = 用户开关结果
```

当 `hasBlackHole = false`：

- 不初始化 `RpsBattleBlackHoleSystem`。
- 不执行黑洞 update。
- 不绘制黑洞。
- 性能日志保持字段稳定，`blackHoleMs=0`、`blackHoleKills/frame=0`。

## 验收标准

### 机制开关

- 选择 `团队占点` 时，黑洞默认开启。
- 玩家关闭黑洞后开局，整局不出现黑洞。
- 关闭黑洞后，中心据点得分、胜利和结算正常。
- `中心据点` 显示为核心锁定，不能关闭。
- 未实现机制显示为后续，不可点击。

### 移动端黑洞

- 手机端黑洞最大吸引圈不覆盖整个战场。
- 黑洞吞噬后会成长，但达到动态上限后停止继续扩大。
- 黑洞核心和吸引圈视觉清晰，玩家能判断危险范围。
- 黑洞不会一出生就直接覆盖中心据点核心。

### 性能日志

必看字段：

```text
fps
update
draw
blackHoleMs
blackHoleKills/frame
zoneMs
zoneInside/frame
```

期望：

- 黑洞开启时 `blackHoleMs` 常态低于 `0.3ms`。
- 黑洞关闭时 `blackHoleMs` 接近 `0`。
- 大局 `large / 36 entities` 不因为机制开关 UI 或动态上限出现明显掉帧。

## 实施拆分

建议分两步做，方便真机验证：

1. 机制开关落地：UI 开关、规则解析、引擎按开关启停黑洞。
2. 黑洞移动端适配：动态上限、出生位置约束、测试补齐。

这两步可以同一轮提交，但测试时要分开看日志：

- 先测“关闭黑洞后是否完全不运行”。
- 再测“开启黑洞后是否不再全屏”。

## 待确认点

当前推荐口径：

- 黑洞默认开启，但允许关闭；后续绝地、道具等增强机制实现后也按同样规则处理。
- 黑洞保留吞噬成长，但使用动态上限。
- 手机端最大吸引范围按短边 `42%` 限制。
- 手机端最大核心半径按短边 `16%` 限制。
- 黑洞不直接出生在中心据点核心内。

如果后续真机感觉黑洞仍然偏强，优先调低 `shortSide * 0.42`，不要先删除黑洞机制。
