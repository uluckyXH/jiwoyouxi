# 石头剪刀布大作战团队占点玩法迁移设计

日期：2026-07-15
状态：原版行为梳理与分阶段迁移方案，P1 中心据点已实现，P2 黑洞已接入，P3 绝地求生第一版已接入；黑洞开关和移动端范围调整见 `黑洞机制开关与移动端适配设计.md`

## 结论

`经典乱斗` 在新版里已经具备核心闭环：

- 三阵营自动追捕。
- 循环克制。
- 碰撞转阵。
- 限时或清场结算。
- 倒计时、暂停、重开、音效、性能日志。

`团队占点` 已按分阶段方式迁移。原版的团队占点不是单一机制，而是一个预设组合：

```ts
controlZones = true
blackHole = true
lastStand = true
powerUps = true
teamPowerUps = true
```

新版不建议一次把这 5 个机制全部接入。P1 先做 `团队占点` 模式骨架和中心据点核心，把“这局的目标从清场变成争夺中心”做清楚；P2 接入黑洞；P3 接入绝地求生；P4 已接入能量道具和团队道具第一版。

当前口径：

- P1 不实现黑洞、绝地求生、普通道具、团队道具；P2 已接入黑洞；P3 已接入绝地求生第一版；P4 已接入能量道具和团队道具第一版。
- 当前团队占点默认机制包已包含中心据点、黑洞、绝地求生、能量道具、团队道具。
- 不把原版所有开关塞进新版选择页。
- 不改 `经典乱斗` 当前玩法。
- 所有新增代码继续走 HarmonyOS 原生 ArkTS / ArkUI / Canvas，不引入 WebView、HTML 游戏页或跨端壳。
- `团队占点` 最终应默认带原版机制包，但未迁移完成的机制只显示“后续”，不允许玩家开启运行。

## 原版据点规则

### 生成

原版只生成一个中心据点：

- 手机宽度下半径 `48`。
- 宽屏半径 `62`。
- `x = width * 0.5`。
- `y = bounds.top + (bounds.bottom - bounds.top) * 0.5`。
- 初始 `owner = -1`。
- 初始 `score = 0`。

### 占领判定

每帧统计据点圆内三方单位数量：

```text
pressure[type] = 当前阵营在据点内的单位数量
```

然后找唯一最多的一方：

- 如果某阵营数量唯一最多，该阵营成为 `owner`。
- 如果并列最多，不切换 owner，不涨分。
- 如果圈内没人，不涨分。

### 进度增长

当存在唯一 owner 时：

```text
score += ownerInsideCount * 2.8 * dt
score <= 100
```

含义：

- 据点里同阵营人数越多，进度越快。
- 单个单位也能占点，但速度慢。
- 多方僵持时不涨分。

### 胜利条件

如果任一据点：

```text
score >= 100 && owner >= 0
```

立即结算：

```text
winner = owner
reason = 据点达标
```

据点胜利优先于限时结算。原版清场仍然可以提前结束；新版当前已为持久占点调低清场权重，团队占点不再按清场立即结束。


例如：


### Canvas

- 在中心画圆。
- 未归属时使用深色边框。
- 有 owner 时边框使用 owner 阵营颜色。
- 圆内填充半透明白色。

- 圆心显示分数。
- 外圈显示 0 到 100 的进度。
- 圈内归属颜色跟当前主题阵营一致。
- 僵持时显示中性色，不要误导玩家正在增长。

## 原版组合机制

### 黑洞

原版团队占点同时开启 `blackHole`：

- 首次出现时间为开局后 `9..16s`。
- 出现在战场内随机位置。
- 初始核心半径 `24`，吸引范围 `82`。
- 进入吸引范围后单位被拉向黑洞。
- 进入核心区域后单位死亡，不发生转阵。
- 每吞噬一个单位，核心半径和吸引范围乘 `1.3`。
- 半径上限 `116`，吸引范围上限 `310`。

对团队占点的作用：

- 打破中心据点附近的稳定堆叠。
- 给领先方制造风险。
- 让据点附近路线不完全可预测。

新版建议：中心据点稳定后作为 P2 接入。黑洞会引入死亡、吸引、绘制、音效反馈和性能日志，必须独立一轮实现和验收，不和绝地、道具混在一起。

### 新版黑洞 P2 实施范围

本轮实现 `团队占点` 默认黑洞机制。最新讨论口径调整为：黑洞默认开启，但作为已实现增强机制，允许玩家在开局前关闭；移动端还需要用战场短边限制最大吸引范围，避免黑洞变成全屏性质。详细方案见 `黑洞机制开关与移动端适配设计.md`。

要做：

1. `zones` 模式下默认启用黑洞。
2. 开局后 `9..16s` 随机出现。
3. 黑洞出现在战场内随机位置，避开边缘。
4. 外圈吸引单位，内核吞噬单位。
5. 吞噬单位后单位直接消失，不计入转阵。
6. 黑洞每吞噬一个单位，核心半径和吸引范围乘 `1.3`，并设置上限。
7. Canvas 绘制黑洞内核、吸引圈和轻量脉冲。
8. 性能日志增加 `blackHoleMs` 和 `blackHoleKills/frame`。
9. 规则测试覆盖未激活不影响、激活后吸引、吞噬减员、半径增长和快照隔离。

暂不做：

- 多黑洞。
- 黑洞预警倒计时 UI。
- 黑洞专属音效。
- 战斗中动态开关。
- 与叛徒模式共享黑洞。

第一版数值沿用原版：

| 参数 | 值 |
| --- | --- |
| 首次出现 | `9..16s` |
| 初始核心半径 | `24` |
| 初始吸引范围 | `82` |
| 吸引力 | `(1 - distance / reach) * 280` |
| 吞噬判定 | `distance < radius + entity.radius * 0.35` |
| 成长倍率 | `1.3` |
| 核心半径上限 | `116` |
| 吸引范围上限 | `310` |

### 绝地求生

原版团队占点同时开启 `lastStand`：

- 某阵营只剩 `1` 个单位时，有概率触发一次分裂。
- 分裂会在源单位附近生成同阵营新单位。
- 某阵营从 `1` 变成 `0` 时，有概率留下复活印记。
- 复活印记 `5s` 后在最后位置复活一个单位。
- 分裂和复活每阵营各限制一次。

对团队占点的作用：

- 避免弱势方在据点模式中太快出局。
- 让落后阵营还有抢回据点的机会。

新版 P3 已完成第一版接入。实现细节见 `绝地求生机制迁移设计.md`：

- `团队占点` 默认开启绝地求生，开局前可关闭。
- 某阵营只剩 `1` 个单位时，按时间节流概率触发一次分裂，避免帧率影响概率。
- 某阵营剩余 `3` 个以内且数量下降时，留下约 `2.2s` 复活印记。
- 阵营已团灭时，只要绝地求生开着且冷却结束，仍可进入援军倒计时。
- 同阵营援军有 `5s` 冷却，同一时间最多一个复活印记。
- 团队占点不再按清场立即结算；据点到 `100` 或时间到才结算。
- Canvas 绘制复活印记倒计时和分裂/复活脉冲。
- 性能日志增加 `lastStandMs`、`lastStandEvents/frame`、`lastStandRevives/frame`。

### 能量道具和团队道具

原版团队占点同时开启 `powerUps` 和 `teamPowerUps`。

道具刷新：

- 首个道具约开局 `5s` 后出现。
- 后续间隔约 `6.5s`，带随机浮动。
- 场上最多 `4` 个。
- 道具存在 `12s` 后过期。

新版当前调参：

- 首个道具约开局 `3s` 后出现。
- 后续间隔约 `4s`，带 `-0.8s..+1s` 随机浮动。
- 场上最多 `6` 个。
- 道具按 `加速 -> 护盾 -> 分裂` 循环刷新，保证真机验收能稳定看到三类效果。

道具类型：

| 类型 | 效果 |
| --- | --- |
| 加速 | 暴走约 `6s`，速度增强 |
| 护盾 | 获得至少 1 层护盾 |
| 分裂 | 在附近生成一个同阵营单位 |

如果 `teamPowerUps` 开启：

- 任意单位拾取道具后，全队同阵营单位获得同类效果。
- 事件提示为 `{阵营} 全队获得{道具名}`。

对团队占点的作用：

- 鼓励单位争夺地图资源。
- 团队道具会突然改变据点圈内压力。

新版建议：作为 P4 接入。它需要道具刷新、拾取检测、状态效果、Canvas 道具绘制、事件提示，不应和中心据点同批做。

## 新版 P1 中心据点范围

P1 只做 `团队占点` 的最小可玩闭环。这里的“机制”只指支撑该模式成立的中心据点规则，不包含黑洞、绝地、道具等附加机制。黑洞已在 P2 单独接入，绝地求生已在 P3 单独接入。

### 要做

1. `RpsBattleModeKey = 'zones'` 时，进入团队占点模式。
2. 模式配置层能表达“当前启用中心据点，后续可挂机制”。
3. 引擎维护一个 `RpsBattleControlZone`。
4. 每帧统计据点内三方人数。
5. 唯一最多阵营成为 owner。
6. owner 按人数涨分。
7. 分数到 100 后以 `zone` 原因结算。
8. HUD 显示据点 owner 和进度。
9. Canvas 绘制中心据点、owner 色和进度。
10. 性能日志增加据点相关计数或耗时字段。

### P1 暂不做

- 普通道具运行逻辑。
- 团队道具运行逻辑。
- 多据点。
- 死斗。
- 自定义规则开关。
- 新的复杂战报系统。

这些附加机制不进入 P1 运行时；其中黑洞已在 P2 进入运行时，绝地求生已在 P3 进入运行时。

## 新版数据设计

### 模式配置和机制口子

当前已有 `RpsBattleModeConfig.ets` 承载选择页卡片。后续不要把机制文案、运行规则和 UI 状态混在 `RpsBattlePage.ets` 里。

建议将模式配置分成两层：

```ts
export interface RpsBattleModePreset {
  key: RpsBattleModeKey;
  title: string;
  subtitle: string;
  objective: string;
  mechanics: string[];
  enabledMechanics: string[];
  upcomingMechanics: string[];
}

export interface RpsBattleModeRules {
  key: RpsBattleModeKey;
  hasControlZone: boolean;
  hasBlackHole: boolean;
  hasLastStand: boolean;
  hasPowerUps: boolean;
  hasTeamPowerUps: boolean;
}
```

第一阶段 `zones` 只开启中心据点：

```ts
{
  key: 'zones',
  hasControlZone: true,
  hasBlackHole: false,
  hasLastStand: false,
  hasPowerUps: false,
  hasTeamPowerUps: false
}
```

P2 接入黑洞后，`zones.hasBlackHole` 默认调整为 `true`。P3 接入绝地求生后，`zones.hasLastStand` 默认调整为 `true`。选择页展示“已启用：中心据点 / 黑洞 / 绝地求生”，其中中心据点是核心机制不可关闭，黑洞和绝地求生是已实现增强机制，允许玩家在开局前关闭。后续每接入一个机制，只改对应规则字段和对应系统文件，不改选择页主结构。

### 机制默认选中策略

`团队占点` 的最终目标是对齐原版预设：选择 `团队占点` 后默认开启中心据点、黑洞、绝地求生、能量道具、团队道具。

但新版要按迁移状态分级：

| 机制 | 当前迁移状态 | 最终默认 | 是否允许关闭 |
| --- | --- | --- | --- |
| 中心据点 | 已启用 | 开启 | 不允许关闭 |
| 黑洞 | 已启用 P2 | 开启 | 允许开局前关闭 |
| 绝地求生 | 已启用 P3 | 开启 | 允许开局前关闭 |
| 能量道具 | 已实现 | 开启 | 允许关闭 |
| 团队道具 | 已实现 | 开启 | 允许关闭，依赖能量道具 |

说明：

- `中心据点` 是团队占点的核心机制，关闭后就不是团队占点，因此不做关闭入口。
- 黑洞、绝地、道具属于模式增强机制。黑洞和绝地求生已接入运行时，应默认跟随团队占点开启，但允许玩家在开局前关闭。
- 未实现机制在 UI 中只显示为 `后续` 或禁用态，不能点击启用。
- 如果玩家关闭某个增强机制，仍然保留 `modeKey = 'zones'`，只是在运行规则中关闭对应字段。

建议新增：

```ts
export type RpsBattleFinishReason = 'none' | 'clear' | 'time' | 'draw' | 'zone';
```

`RpsBattleOptions` 增加：

```ts

```

引擎判断：

```ts
if (modeRules.hasControlZone) {
```

注意：

- 当前只允许已实现且可验证的增强机制暴露为玩家可配开关；`blackHole`、`lastStand`、`powerUps`、`teamPowerUps` 可以开局前关闭，其中 `teamPowerUps` 依赖 `powerUps`。

## 开发规范和拆分要求

本模式必须继续按 HarmonyOS 原生工程方式开发：

- UI 使用 ArkUI 组件。
- 大量移动实体和据点绘制使用 Canvas。
- 规则和模拟使用 ArkTS 纯逻辑，不依赖 WebView、HTML、CSS 动画或浏览器 API。
- 音效继续复用现有 `AudioService` 路线，不在帧循环中重复加载资源。

按 `docs/鸿蒙原生开发规范.md`，本次不应该继续扩大 `RpsBattlePage.ets` 的职责。建议拆分如下：

```text
entry/src/main/ets/games/rpsBattle/
  RpsBattleModeConfig.ets       模式卡片、模式运行规则、后续机制展示
  RpsBattleZoneSystem.ets       中心据点状态、更新、胜利判断
  RpsBattleBlackHoleSystem.ets  黑洞出现、吸引、吞噬和成长
  RpsBattleLastStandSystem.ets  弱势分裂、复活印记和复活请求
  RpsBattleRenderer.ets         Canvas 绘制据点和单位
  RpsBattleHud.ets              据点状态和进度展示
  RpsBattleEngine.ets           主循环调度各系统，不写 ArkUI
  RpsBattleTypes.ets            模式、据点、结算原因等类型
  RpsBattleRules.ets            克制、计数、结算等纯规则
```

职责边界：

- `RpsBattlePage.ets` 只传入 `selectedModeKey`、负责生命周期和页面状态。
- `RpsBattleEngine.ets` 只调度，不直接写大量据点细节。
- `RpsBattleZoneSystem.ets` 只处理中心据点，不知道 HUD 怎么展示。
- `RpsBattleBlackHoleSystem.ets` 只处理黑洞状态和吞噬名单，不负责移除数组缓存。
- `RpsBattleRenderer.ets` 只读快照画图，不修改玩法状态。
- `RpsBattleHud.ets` 只显示节流后的快照，不每帧接管引擎状态。

后续机制也按同样方式补独立系统：

```text
RpsBattlePowerUpSystem.ets
```

不要把这些机制继续堆进 `RpsBattleEngine.ets` 的大方法里。

### 据点半径

新版棋盘尺寸和原版不同，建议先用：

```text
radius = clamp(min(boardWidth, boardHeight) * 0.15, 44, 62)
```

原因：

- 小屏不至于太小。
- 大屏不至于占满战场。
- 与原版 `48 / 62` 接近。

### 涨分系数

第一版沿用原版：

```text
score += ownerInsideCount * 2.8 * dt
```

真机体验后再调。

如果 60s 内太难达到 100，可以考虑：

```text
score += ownerInsideCount * 3.2 * dt
```

但不要先改，先对齐原版。

```text
已启用：中心据点 / 黑洞 / 绝地求生
可关闭：黑洞 / 绝地求生 / 能量道具 / 团队道具
```

真机验收时需要确认用户知道黑洞、绝地求生、能量道具和团队道具可开关，且团队道具依赖能量道具。

### HUD

当前 HUD 显示：

- 三方数量。
- 剩余时间。
- 支持阵营。
- 转阵数。

团队占点需要新增一块轻量状态：

```text
据点 73 / 100
归属 中国队
```

建议放在现有 HUD 第二行，替代或补充“转阵”信息，不要新增很高的卡片。

### Canvas

中心据点绘制层级：

1. 背景。
2. 据点底圈和进度环。
3. 单位。
4. 转阵脉冲。

据点在单位下面，避免遮挡棋子。

视觉要求：

- 未归属：中性描边。
- 有 owner：使用主题阵营色。
- 僵持：边框可轻微变灰或显示中性色。
- 进度环要能看到接近 100 的压力。

## 性能影响

据点第一阶段每帧需要扫描所有实体一次：

```text
O(n)
```

当前人数上限为 60，成本可控。

注意：

- 不要把据点状态写入 ArkUI `@State` 每帧刷新；ArkUI 只接收节流后的 HUD 快照。
- 据点扫描使用平方距离判断，避免每个单位都 `Math.sqrt()`。
- 据点 pressure 计数尽量复用固定结构，减少每帧临时对象。
- HUD 仍按当前节流同步，不因为据点分数增长而每帧重建 ArkUI 文本。
- Canvas 每帧绘制据点可以接受，路径数量很少。
- `classic` 模式不执行 `RpsBattleZoneSystem.update()`，避免给现有玩法增加空转成本。
- 性能日志中新增 `zoneMs`、`zoneInside/frame` 或同等字段，确认扫描成本。
- 大局 36 个单位、最大 60 个单位都要真机观察，不能只看模拟器。

建议目标：

| 指标 | 目标 |
| --- | --- |
| `classic` 模式 | 性能日志不退化 |
| `zones` 大局 36 单位 | 稳定接近 60fps |
| `zoneMs` | 常态低于 `0.3ms` |
| `blackHoleMs` | 常态低于 `0.3ms` |
| `draw` | 据点绘制后不明显上升 |

如果 `zoneMs` 或 `blackHoleMs` 超预期，优先优化距离判断、数据复用和移除流程，不先降低玩法内容。

## 实施步骤

1. 在 `RpsBattleModeConfig.ets` 增加模式运行规则，明确 `zones` 当前启用中心据点和黑洞。
2. `RpsBattleOptions` 增加 `modeKey`。
3. `RpsBattlePage.startGame()` 把 `selectedModeKey` 传入 options。
4. `RpsBattleTypes` 增加 `RpsBattleControlZone` 和 `zone` 结算原因。
5. 新增 `RpsBattleZoneSystem.ets`，封装中心据点初始化、压力统计、涨分和胜利判断。
6. 新增 `RpsBattleBlackHoleSystem.ets`，封装黑洞激活、吸引、吞噬和成长。
7. `RpsBattleEngine.reset()` 根据模式规则初始化 `RpsBattleZoneSystem` 和 `RpsBattleBlackHoleSystem`。
8. `RpsBattleEngine.update()` 在 `hasControlZone` 时调度据点系统，在 `hasBlackHole` 时调度黑洞系统。
9. `RpsBattleEngine.resultAt()` 或等价结算路径接入据点 `score >= 100`。
10. `RpsBattleSnapshot` 暴露只读据点和黑洞状态。
11. `RpsBattleRenderer.draw()` 增加据点底圈、owner 色、进度环、黑洞吸引圈和核心。
12. `RpsBattleHud` 增加据点状态展示，仍保持节流同步。
13. 性能日志增加 `zoneMs`、`zoneInside/frame`、`blackHoleMs` 和 `blackHoleKills/frame`。
14. 规则测试覆盖 owner、僵持、不涨分、100 分结算、黑洞激活、吸引、吞噬和成长。
15. 真机测试 `团队占点` 标准局和大局。
16. 如果修改行为进入可测阶段，在 `测试报告/` 留下测试报告；文档-only 改动可在提交说明中注明。

## 验收标准

- 选择 `经典乱斗` 时，当前玩法完全不变。
- 选择 `团队占点` 后，战场中央出现据点。
- 据点归属颜色跟随主题阵营色。
- 圈内唯一最多阵营时，据点进度增长。
- 圈内并列或无人时，据点进度不增长。
- 据点到 100 后对应阵营获胜。
- 清场不再提前结束团队占点。
- 限时到达时仍按人数领先结算。
- 大局模式下没有明显新增卡顿。

## 后续机制接入原则

每个后续机制单独走一轮设计、实现和真机验证，不和其他机制混在同一个提交里。

### 接入前

- 先补对应设计段落，确认是否对齐原版。
- 明确是否影响胜负、移动、碰撞、渲染、音效或战报。
- 明确与 `classic` 和 `zones` 的组合关系。
- 先定义类型和快照字段，再写 UI。

### 实现时

- 新机制优先新增独立系统文件。
- 不把机制细节写进 `RpsBattlePage.ets`。
- 不在 ArkUI `build()` 中做规则计算。
- 不让未启用机制在每帧空转。
- 不新增无上限的实体、粒子或日志输出。

### 验收时

- 必测 `classic`，确认原玩法不变。
- 必测 `zones`，确认中心据点仍能正常结算。
- 必看性能日志，至少比较新增机制前后的 `fps`、`update`、`draw`、机制专属耗时。
- 如果新增机制会生成对象，要观察长局是否出现持续变慢。

后续机制口子保留，但默认关闭。黑洞、绝地求生、能量道具和团队道具已经完成规则、渲染和性能日志接入，因此从 `upcomingMechanics` 移入 `enabledMechanics`；其他模式机制继续保持 `upcoming`，等各自完成规则、渲染、HUD、性能日志和真机验收后再移动。
