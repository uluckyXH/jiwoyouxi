# 石头剪刀布大作战鸿蒙原生性能优化后续方案

日期：2026-07-09
状态：基于真机日志与 HarmonyOS 官方文档的下一阶段方案

## 一、当前判断

从最新真机日志看，`rpsBattle` 已经从最初的 30FPS timer 阶段进入 60Hz `displaySync` 阶段。标准局基本可接受，大局模式仍有偶发体感卡顿。

最新较好的大局日志：

```text
scale=large entities=36 frames=61 fps=60.9 interval=16.8ms maxInterval=31ms update=10.08ms draw=2.28ms target/frame=394 collision/frame=119 conversions/frame=0.26
```

同一次测试里也出现过转换爆发段：

```text
scale=large entities=36 frames=46 fps=45.5 interval=21.6ms maxInterval=31ms update=13.70ms draw=2.87ms target/frame=632 collision/frame=124 conversions/frame=0.59
```

结论：

- 现在不是单纯绘制慢，`draw` 多数在 2ms 到 3ms。
- 主压力仍在 `update`，尤其是转化爆发时 `target/frame`、阵营迁移、碰撞后处理、音效触发叠在一起。
- `maxInterval=31ms` 说明偶发掉到约半帧节奏，体感会感到“顿一下”。
- 之前的精确目标空间索引不应继续推进：它降低了扫描数，但真机 `update` 恶化到 21ms 到 23ms，整体掉到 32FPS 左右。

本阶段目标不是改玩法，而是在保留三阵营追捕、克制转化、碰撞、结算和音效的前提下，把大局模式的尖峰压下去。

## 二、官方文档依据

### 2.1 `displaySync` 是当前正确方向

HarmonyOS 官方 `@ohos.graphics.displaySync` 文档说明，可变帧率用于让开发者以指定帧率运行 UI 业务，通常用于自绘制 UI 且对帧率有特定诉求的场景。

当前 `rpsBattle` 是 Canvas 自绘制战场，用 `displaySync.create()`、`setExpectedFrameRateRange()`、`on('frame')`、`start()`、`stop()` 驱动游戏循环是对的。

但要注意：`ExpectedFrameRateRange` 只是期望帧率，字段包括 `min`、`max`、`expected`。实际帧率仍受设备最大帧率、系统调度、当前负载影响。因此大局模式不能只“强行要 60FPS”，还要保证每帧 `update + draw` 长期低于 16.7ms。

官方来源：

- HarmonyOS `@ohos.graphics.displaySync` 可变帧率。
- HarmonyOS `ExpectedFrameRateRange`。
- 本机 DevEco SDK：`@ohos.graphics.displaySync.d.ts`、`@ohos.arkui.UIContext.d.ts`。

### 2.2 深入定位应使用 HiDebug/Trace，而不是继续猜

HarmonyOS `hidebug.startAppTraceCapture()` 支持指定范围自动采集 Trace；官方也明确提醒 trace 采集本身有性能消耗，要在采集后及时关闭。

对当前游戏，下一步如果要继续压极限，应只在复现卡顿的短时间窗口采集：

- `hidebug.tags.ARKUI`
- `hidebug.tags.GRAPHICS`
- `hidebug.tags.AUDIO`
- `hidebug.tags.ARK`
- `hidebug.TraceFlag.MAIN_THREAD`

目标是判断 31ms 尖峰到底落在：

- ArkTS 引擎循环。
- Canvas/图形管线。
- SoundPool 音效触发。
- ArkUI 页面状态同步。
- GC 或内存分配。

官方来源：

- HarmonyOS `hidebug.startAppTraceCapture()` / `stopAppTraceCapture()`。
- 本机 DevEco SDK：`@ohos.hidebug.d.ts`。

### 2.3 Worker/TaskPool 不是当前优先方案

官方 Worker 文档说明 Worker 与主线程并行，可处理耗时操作，但不能直接操作 UI，空闲 Worker 也要及时销毁避免资源浪费。

对当前 36 个实体的小规模实时游戏，把每帧模拟整体搬到 Worker 暂不划算：

- 每帧要同步实体位置、阵营、转化、结算和音效事件。
- 主线程仍要绘制 Canvas 和处理页面状态。
- 如果不用 `ArrayBuffer`/`SharedArrayBuffer` 做数据传递，序列化成本可能抵消收益。

所以 Worker 只作为后续更大规模或数据化引擎阶段的候选，不作为当前 P0/P1。

官方来源：

- HarmonyOS Worker 文档。
- HarmonyOS TaskPool/Worker 对比文档。

### 2.4 SoundPool 需要先把参数和触发节奏收敛

当前日志里反复出现：

```text
can not find rate property
can not find parallelPlayFlag property
Failed to obtain ability in FA mode
```

本机 SDK 的 `media.PlayParameters` 明确包含 `loop`、`rate`、`leftVolume`、`rightVolume`、`priority`。当前代码传了 `loop/leftVolume/rightVolume/priority`，没传 `rate`，所以低风险改动是先补 `rate`。

`parallelPlayFlag` 在当前 SDK 类型里没有字段，但 SoundPool API 23+ 提供 `setInterruptMode()`，默认模式是 `SAME_SOUND_INTERRUPT`。因此不要直接硬塞不在类型里的字段，应先做两步验证：

- 播放参数补齐 `rate`。
- 如果目标 SDK 支持，再评估 `setInterruptMode(media.SoundInterruptMode.SAME_SOUND_INTERRUPT)` 或 `NO_INTERRUPT` 的实际日志和听感。

官方来源：

- HarmonyOS MediaKit `SoundPool` / `PlayParameters`。
- 本机 DevEco SDK：`multimedia/soundPool.d.ts`、`@ohos.multimedia.media.d.ts`。

## 三、下一阶段优化方案

### P0：分段性能日志

目标：把现在的 `update` 拆开，不再只看一个总耗时。

新增聚合指标：

- `moveMs`：寻路、追捕、逃离、速度和边界处理。
- `targetMs`：最近猎物/天敌搜索。
- `collisionMs`：空间分桶和碰撞处理。
- `convertMs`：阵营转化、缓存迁移、计数更新。
- `audioMs`：本帧音效触发入口耗时。
- `hudMs`：HUD 状态同步耗时。

验收：

- 每秒仍只打一条聚合日志，避免日志本身造成卡顿。
- 大局卡顿时能看出 31ms 尖峰来自哪一段。
- 日志开关继续保持开发期常量控制。

### P1：音效路径止噪和削峰

目标：降低 SoundPool 日志噪声和转化爆发时的音频调用压力。

建议：

- `NativeAudioService.play()` 的 `PlayParameters` 补 `rate: audio.AudioRendererRate.RENDER_RATE_NORMAL` 或 SDK 可接受的默认正常播放枚举/数值。
- 保持转化音效节流，必要时把 `RPS_BATTLE_CONVERT_SOUND_INTERVAL_MS` 从 120ms 提到 150ms 或 180ms。
- 结算音效、按钮音效不参与高频节流。
- 记录本秒 `sfxCalls/frame`，确认卡顿时音效触发是否和 `maxInterval=31ms` 同步出现。

玩法影响：不影响规则，只影响同一秒内转化音效的密度。

验收：

- `can not find rate property` 消失。
- 转化爆发期 SoundPool 日志明显减少。
- 不出现漏放结算音效、BGM 中断、页面退出后继续播放。

### P2：阵营缓存迁移改成 O(1)

目标：当前阵营缓存虽然已经减少了目标扫描，但转化时从旧阵营数组移除实体仍可能做线性查找。转化爆发时，这会叠加到 `update` 尖峰。

建议：

- 为实体维护 `factionIndexById` 或等价索引表。
- `addToFactionCache()` 记录实体在阵营数组中的下标。
- `removeFromFactionCache()` 使用 swap-remove：用最后一个实体覆盖被移除位置，再更新被移动实体的下标。
- 转化时只做 O(1) 删除和 O(1) 添加。

玩法影响：

- 克制、追捕、碰撞、结算不变。
- 阵营数组内部顺序会改变；只有完全等距目标的 tie-break 可能不同，正常对局体感基本不变。

验收：

- 转化爆发段 `update` 更稳定。
- `conversions/frame` 升高时不再同步出现明显 `update` 尖峰。
- 大局 `maxInterval` 从 31ms 尽量压到 25ms 以下。

### P3：目标搜索继续做“轻量剪枝”，不再做精确空间索引

目标：降低 `target/frame` 高峰，但不引入上一次空间索引那种高维护成本。

可尝试方案：

- 最近目标搜索仍基于阵营数组。
- 循环中先比较 `dx * dx`，只有可能优于当前最短距离时再计算完整 `dy` 和距离平方。
- 逃离目标只关心 `ENTITY_FLEE_DISTANCE` 范围内的天敌，可先用包围盒/平方距离快速跳过远目标。
- 对猎物搜索保留全阵营最近逻辑，避免改变追捕策略。

不推荐方案：

- 不重上精确目标空间索引。
- 不为 36 个实体引入复杂扩圈查询。
- 不把搜索结果缓存多帧，除非明确接受玩法行为变钝。

玩法影响：应保持不变。

验收：

- `target/frame` 可能不会大幅下降，但 `targetMs` 应下降。
- 真机 `update` 不得因剪枝逻辑变复杂而回退到 20ms 以上。

### P4：大局模式自适应帧率策略

目标：如果 P0/P1/P2 后大局仍有尖峰，用官方 `ExpectedFrameRateRange` 做可控降级，而不是让系统随机掉帧。

策略：

- 标准局：继续 `min=60 max=60 expected=60`。
- 大局默认：先试 `min=45 max=60 expected=60`。
- 如果连续多秒 `maxInterval >= 31ms` 且 `update + draw > 16.7ms`，大局切到 `min=45 max=60 expected=45`。
- 切换只影响帧驱动频率，不改模拟规则。`dt` 仍按真实时间推进。

玩法影响：

- 规则不变。
- 视觉流畅度目标从强 60 降到稳 45，体感可能更稳定，但不是更丝滑。

验收：

- 大局卡顿时不再长期出现 31ms 断续尖峰。
- 不出现游戏速度变快/变慢。
- 暂停、恢复、结算后帧率策略能重置。

### P5：绘制层只做必要削峰

当前 `draw` 不是最大瓶颈，所以绘制不作为第一优先级。但后续接素材图后必须提前约束。

建议：

- 静态背景、网格、边框尽量拆成低频绘制或缓存。
- 大局转化爆发时限制脉冲圈数量。
- 素材图预加载并复用，不能在帧循环里加载资源或创建对象。
- 继续避免每帧写大量 ArkUI `@State`。

玩法影响：只影响视觉表现，不影响规则。

## 四、不建议当前做的事

- 不再投入精确目标空间索引。已有真机数据证明对 36 个实体不划算。
- 不把每帧主循环整体搬到 Worker/TaskPool。当前数据量太小，通信和同步复杂度不值得。
- 不盲目把所有模式锁死 60FPS。60FPS 的前提是每帧稳定低于 16.7ms。
- 不在没有 Trace 的情况下继续堆复杂算法。下一步要先知道尖峰到底来自哪里。

## 五、推荐实施顺序

1. 先加 P0 分段日志，跑标准局和大局各 2 到 3 局。
2. 同时做 P1 的 `rate` 参数补齐和音效调用计数，确认 SoundPool 噪声是否下降。
3. 如果日志显示转化段尖峰明显，做 P2 阵营缓存 O(1) 迁移。
4. 如果大局仍有 31ms 尖峰，再做一次短窗口 HiDebug Trace。
5. Trace 证明主线程确实超预算后，再决定 P3 轻量剪枝或 P4 自适应帧率。

## 六、阶段验收标准

标准局：

- `fps` 稳定接近 60。
- `maxInterval` 尽量不超过 20ms。
- `update + draw` 常态低于 12ms。

大局：

- 稳态 `fps` 接近 60，或自适应后稳定在 45 到 60。
- `maxInterval` 尽量低于 25ms。
- 转化爆发段不出现持续 31ms 间隔。
- `ProcessJank` 不应连续出现。
- 音效日志不再每次播放都报缺少 `rate`。
