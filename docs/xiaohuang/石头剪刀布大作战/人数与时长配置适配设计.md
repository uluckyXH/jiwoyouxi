# 石头剪刀布大作战人数与时长配置适配设计

日期：2026-07-14
状态：方案设计，待实现与真机验证
适用范围：新版 `entry/src/main/ets/games/rpsBattle/`

## 结论

人数增加必须有上限。

原版 `games/rps` 允许三方分别通过 `+ / -` 调整人数，并支持“人数同步”。它的单方范围是 `3..60`，同步开启时三方一起变化，理论上能到 `60 / 60 / 60`，总计 180 个单位。

新版 `rpsBattle` 目前已经针对 `36` 个单位的大局做过性能优化，真机日志出现过稳定 60FPS 段。但这不能直接推导出 180 个单位也稳定。人数增加会放大以下成本：

- 移动和目标选择：随单位数线性增长，并受阵营分布影响。
- 碰撞检查：已做空间分桶，但高密度场景仍会增加邻域检查。
- Canvas 绘制：随单位数线性增长。
- 音效与转阵反馈：人数越多，单位接触越频繁，需要节流保护。

所以新版这一阶段不照搬原版 `60 / 60 / 60` 作为可直接选择的普通能力。应先做“快捷档位常驻 + 高级调整收起 + 新版性能保护”。

## 设计目标

1. 迁移原版已有的三方人数调节能力。
2. 保留“人数同步”开关。
3. 允许调整对局时长。
4. 三个数量快捷档位继续保留，作为普通玩家的主入口。
5. 高级人数调整默认收起，避免选择页变重。
6. 给人数设置硬上限，避免用户把游戏调到不可玩的密度。
7. 不改变当前已优化好的基础玩法：自动追捕、循环克制、碰撞转化、限时/清场结算。
8. 所有限制集中在规则常量和校验函数里，不把数字散落在页面组件中。

## 原版行为基线

参考文件：

- `entry/src/main/ets/games/rps/RpsPage.ets`
- `LineupPanel()`
- `CountControl()`
- `stepCount()`

原版能力：

| 能力 | 原版行为 |
| --- | --- |
| 三方人数 | 石头、剪刀、布分别有 `- / 数字 / +` |
| 人数同步 | 默认开启，调整任意一方会同步三方 |
| 独立人数 | 关闭同步后，每方独立调整 |
| 人数范围 | 每方 `3..60` |
| 时长 | 点击按钮在 `30s..120s` 间按 `15s` 步进循环 |

新版保留交互结构，但不直接保留原版最大人数。

## 新版人数限制

### 第一阶段建议值

| 常量 | 建议值 | 说明 |
| --- | ---: | --- |
| `RPS_BATTLE_MIN_COUNT_PER_FACTION` | `3` | 对齐原版下限，保证三方都能参与 |
| `RPS_BATTLE_RECOMMENDED_MAX_PER_FACTION` | `12` | 对应当前“大局”，已实际优化验证 |
| `RPS_BATTLE_MAX_COUNT_PER_FACTION` | `20` | 第一阶段高级调节单方硬上限 |
| `RPS_BATTLE_MAX_TOTAL_ENTITIES` | `60` | 第一阶段总人数硬上限 |

解释：

- `12 / 12 / 12` 是当前推荐大局，总数 36，已经有较多真机日志支撑。
- `20 / 20 / 20` 总数 60，作为下一档可测试上限，但必须继续观察真机日志。
- 不开放 `60 / 60 / 60`，因为总数 180 会明显超过当前验证范围。

### 校验规则

所有入口都必须经过同一个校验函数：

```ts
clampRpsBattleCounts(counts: RpsBattleFactionCounts): RpsBattleFactionCounts
```

规则：

1. 每方先限制在 `3..20`。
2. 如果总数小于等于 `60`，直接返回。
3. 如果总数超过 `60`，本次加人操作不生效，保持原值。
4. 同步模式下，如果下一步会超过 `20 / 20 / 20`，按钮进入禁用态。
5. 独立模式下，如果总数已到 60，只允许减少，不允许继续增加。

不建议用“自动按比例压缩”处理超限，因为玩家点了某一方的 `+`，结果三方数字一起变化，会让交互不透明。

## UI 设计

### 入口位置

选择界面保持当前顺序：

1. 阵营主题
2. 支持阵营
3. 初始阵容
4. 对局时长
5. 开始乱斗

“初始阵容”保留当前“小局 / 标准 / 大局”规模块，并在下面增加一个默认收起的“高级调整”区。

### 初始阵容

默认结构：

- 顶部：标题 `初始阵容`
- 右侧摘要：`总数 24 / 上限 60`
- 第一行：快捷档位 `小局 / 标准 / 大局`
- 第二行：`高级调整` 折叠入口

点击快捷档位会直接写入三方人数，并保持“人数同步”当前开关状态不变。它是普通入口，也是最快开局路径。

快捷档位：

| 档位 | 人数 | 行为 |
| --- | --- | --- |
| 小局 | `5 / 5 / 5` | 快速局 |
| 标准 | `8 / 8 / 8` | 默认 |
| 大局 | `12 / 12 / 12` | 推荐最大普通局 |

### 高级调整

高级调整默认收起。入口文案建议：

- 收起态：`高级调整`
- 展开态：`收起调整`
- 辅助信息：`同步 / 总数 24 / 60`

展开后结构：

- 第一行：`人数模式` 两段选项：`一起调 / 单独调`
- 第二行：三方人数控制卡
- 第三行：人数上限提示，例如 `每方 3-20，总数最多 60`

三方人数控制卡：

- 显示主题图标和名称。
- 中间显示当前人数。
- 左右是 `-` 和 `+` 图标按钮。
- 选中主题后名称、图标、颜色同步变化。
- 到达下限时 `-` 禁用。
- 到达单方上限或总数上限时 `+` 禁用或点击无效。

高级调整不做额外二级页面，仍在当前选择页内完成。它只是把原版人数能力折叠起来，不影响默认开局效率。

### 人数模式

默认开启。

选择 `一起调` 时：

- 点任意阵营 `+`，三方同时 `+1`。
- 点任意阵营 `-`，三方同时 `-1`。
- 上限等价于每方 20，总数 60。

选择 `单独调` 时：

- 每方独立调整。
- 单方仍受 `3..20` 限制。
- 总数仍受 60 限制。
- 如果总数已到 60，所有 `+` 禁用，只有 `-` 可用。

### 对局时长

保留原版的轻量循环，但用更明确的分段按钮呈现，避免用户点按钮不知道下一档是什么。

建议档位：

| 档位 | 时长 |
| --- | ---: |
| 快速 | `30s` |
| 标准 | `60s` |
| 持久 | `90s` |
| 长局 | `120s` |

默认 `60s`。

这一阶段不做任意输入，避免配置面板变重。

## 数据结构调整

当前 `RpsBattleOptions` 是：

```ts
export interface RpsBattleOptions {
  supportFaction: RpsBattleFaction;
  scale: RpsBattleScale;
  initialCount: number;
  durationSec: number;
}
```

建议改为：

```ts
export interface RpsBattleOptions {
  supportFaction: RpsBattleFaction;
  initialCounts: RpsBattleFactionCounts;
  durationSec: number;
}
```

`scale` 可以继续保留为 UI 快捷档位状态，但引擎启动时不再只接收一个 `initialCount`。这样才能支持独立人数。

建议新增：

```ts
export interface RpsBattleLineupConfig {
  counts: RpsBattleFactionCounts;
  syncCounts: boolean;
  durationSec: number;
}
```

页面状态：

```ts
@State lineupCounts: RpsBattleFactionCounts = rpsBattleCountsFromValue(8);
@State syncCounts: boolean = true;
@State durationSec: number = 60;
```

规则函数集中放在 `RpsBattleRules.ets` 或新增 `RpsBattleLineupConfig.ets`：

- `rpsBattleCountsFromValue(value: number)`
- `rpsBattleTotalCount(counts: RpsBattleFactionCounts)`
- `rpsBattleCanIncreaseCount(counts, faction, syncCounts)`
- `rpsBattleCanDecreaseCount(counts, faction, syncCounts)`
- `rpsBattleStepCount(counts, faction, step, syncCounts)`
- `rpsBattleClampDuration(durationSec)`

## 引擎适配

`RpsBattleEngine.reset()` 当前按单一 `initialCount` 生成三方均衡单位。改造后：

1. 从 `options.initialCounts.rock` 生成石头逻辑阵营。
2. 从 `options.initialCounts.scissors` 生成剪刀逻辑阵营。
3. 从 `options.initialCounts.paper` 生成布逻辑阵营。
4. 生成后同步维护阵营缓存和 `countsCache`。

注意：

- 阵营主题不进入引擎。
- 引擎只认识 `rock / scissors / paper`。
- 生成布局要根据总人数调整随机散布，避免 60 个单位开局过度重叠。
- 开局倒计时期间不消耗正式时长。

## 性能保护

### 日志观察指标

实现后继续沿用 `RpsBattlePerf` 日志，重点看：

| 指标 | 目标 |
| --- | --- |
| `fps` | 尽量稳定接近 60 |
| `interval` | 接近 `16ms` |
| `maxInterval` | 不频繁超过 `31ms` |
| `update` | 建议低于 `10ms`，60 人局允许短时更高 |
| `draw` | 建议低于 `4ms` |
| `targetMs` | 若升高，说明追捕目标选择成为瓶颈 |
| `collisionMs` | 若升高，说明密度过高或分桶参数需调整 |
| `sfx/frame` | 不应随人数明显升高 |

### 放开上限的条件

只有满足以下条件，才考虑从总数 60 继续放开：

1. `20 / 20 / 20` 真机连续 3 局没有明显卡顿。
2. 性能日志多数时间 `fps >= 58`。
3. `maxInterval=31ms` 只是偶发，不连续出现。
4. 转阵高峰期音效没有明显拖慢 UI 线程。
5. 用户体感认可。

下一档可以考虑：

- 每方上限 `24`
- 总人数上限 `72`

不建议一次性跳到原版 `60 / 60 / 60`。

## 实施步骤

1. 新增人数和时长规则常量。
2. 新增人数校验与步进函数，并补单元测试。
3. 将 `RpsBattleOptions.initialCount` 改为 `initialCounts`。
4. 改造 `RpsBattleEngine.reset()` 支持三方不同人数。
5. 改造选择面板：快捷档位常驻，高级调整默认收起。
6. 高级调整内实现人数模式、三方 `+/-`、人数上限提示。
7. 增加时长档位选择。
8. 改造重开逻辑：沿用当前主题、支持阵营、人数、同步状态和时长。
9. 更新性能日志的规模字段，能区分快捷档位和自定义人数。
10. 真机测试 12/12/12、20/20/20、非均衡 20/20/3、非均衡 20/3/20。
11. 根据日志决定是否保持 60 总上限或继续放开。

## 验收标准

- 默认局仍为 `8 / 8 / 8 · 60s`。
- `小局 / 标准 / 大局` 快捷档位可用。
- 高级调整默认收起。
- 展开高级调整后可看到人数模式和三方人数控制。
- 选择 `一起调` 时，调整任意一方会同步三方。
- 选择 `单独调` 时，三方可独立调整。
- 每方不能低于 3。
- 每方不能高于 20。
- 总人数不能高于 60。
- 时长可在 `30s / 60s / 90s / 120s` 间选择。
- 开始、倒计时、战斗、暂停、重开、结算不退化。
- 大局 `12 / 12 / 12` 保持当前流畅度。
- `20 / 20 / 20` 至少能完成对局，若日志不稳定，保持为高级上限不继续放开。

## 暂不做

- 任意输入人数。
- 原版 `60 / 60 / 60` 普通开放。
- 按设备性能自动放开人数上限。
- 人数超限后的自动比例压缩。
- 因人数变化引入新的玩法预设。
