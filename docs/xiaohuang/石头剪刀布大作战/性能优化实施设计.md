# 石头剪刀布大作战性能优化实施设计

日期：2026-07-09
状态：P1/P2 引擎优化已实现；P3 精确空间索引实测失败已回退；P4 显示同步帧驱动已实现

## 一、目标

在不改变基础玩法的前提下，降低 `rpsBattle` 大局模式的卡顿感，让游戏继续保持：

- 三阵营自动追捕。
- 石头剪刀布循环克制。
- 碰撞后转化阵营。
- 限时或清场结算。
- 音效、暂停、继续、返回大厅逻辑不退化。

本设计只针对当前已观察到的性能瓶颈，不做视觉重构，不接新素材，不做多线程模拟。

## 二、当前真机基线

来自 2026-07-09 真机日志。

### 标准局

```text
scale=standard entities=24 fps=28.9~29.6 interval=33.7~34.1ms maxInterval=35~38ms update=12.14~12.30ms draw=1.69~1.77ms target/frame=1152 collision/frame=276
```

### 大局

```text
scale=large entities=36 fps=29.2 interval=34.1ms maxInterval=37ms update=21.73ms draw=2.00ms target/frame=2592 collision/frame=630
```

### 判断

- `draw` 只有 1.7ms 到 2.0ms，当前 Canvas 绘制不是主瓶颈。
- `update` 在标准局约 12ms，大局约 22ms，是主要瓶颈。
- `target/frame` 与 `entities * entities * 2` 完全吻合，说明目标搜索仍在做每单位两次全量扫描。
- `collision/frame` 与 `entities * (entities - 1) / 2` 完全吻合，说明碰撞检测仍在做全体两两检查。

因此第一轮优化重点放在 `RpsBattleEngine`，目标是先减少每帧扫描次数。

## 三、核心方案

### 3.1 第一阶段：阵营缓存 + 增量计数

目标：先把目标搜索从“扫描全体实体”改成“只扫描目标阵营实体”。

当前状态：2026-07-09 已实现，待真机日志复测。

当前逻辑：

- 每个单位找猎物时扫描全部实体。
- 每个单位找天敌时扫描全部实体。
- 标准局 24 个实体：`24 * 24 * 2 = 1152` 次目标扫描。
- 大局 36 个实体：`36 * 36 * 2 = 2592` 次目标扫描。

优化后逻辑：

- 引擎维护三个阵营缓存：
  - `rockEntities`
  - `scissorsEntities`
  - `paperEntities`
- 找猎物只扫描被克制阵营缓存。
- 找天敌只扫描克制自己的阵营缓存。
- 实体转化时，从旧阵营缓存移到新阵营缓存。
- `counts` 随转化增量更新，不再每次结算判断都全量统计。

平衡局预期：

| 规模 | 当前 target/frame | 阵营缓存后预估 |
| --- | ---: | ---: |
| 标准局 24 个 | 1152 | 约 384 |
| 大局 36 个 | 2592 | 约 864 |

收益：

- 目标搜索扫描量理论上降低到约 1/3。
- 胜负判断和 HUD 计数减少重复统计。
- 玩法行为基本不变，只是搜索范围从“全体里筛阵营”变成“直接看目标阵营”。

风险：

- 转化时必须同时维护实体本身 `faction`、阵营缓存、counts，三者不能不同步。
- 如果某阵营数量严重膨胀，扫描量会随阵营人数变化，但仍优于扫描全体。

验收：

- 标准局 `target/frame` 从 1152 降到约 300 到 500。
- 大局 `target/frame` 从 2592 降到约 700 到 1100。
- `update` 明显下降。
- 三阵营转化和结算结果不变。

### 3.2 第二阶段：碰撞空间分桶

目标：把碰撞检测从“全体两两检查”改成“只检查附近实体”。

当前状态：2026-07-09 已实现，待真机日志复测。

当前逻辑：

- 标准局 24 个实体：`24 * 23 / 2 = 276` 次碰撞检查。
- 大局 36 个实体：`36 * 35 / 2 = 630` 次碰撞检查。

优化后逻辑：

- 每帧按实体坐标放入均匀网格。
- 格子尺寸先取 `ENTITY_RADIUS * 3`，当前约 42。
- 每个实体只和当前格、相邻 8 格里的实体检查碰撞。
- 用 `id` 或索引顺序避免重复检查同一对实体。

收益：

- 实体数量增长后，碰撞检查次数不再按全体平方增长。
- 后续新增更大规模或事件模式时有性能余量。

风险：

- 格子尺寸太小可能漏碰撞，太大则收益下降。
- 边界格和相邻格遍历要稳定，不能出现偶发穿透或重复转化。

验收：

- 标准局 `collision/frame` 明显低于 276。
- 大局 `collision/frame` 明显低于 630。
- 单位不会穿透、异常重叠或卡边。
- 转化次数和战场节奏保持合理。

### 3.3 第三阶段：目标空间索引验证 + 热路径微优化

目标：在不改变基础玩法的前提下，继续降低大局 60Hz 下的 `update` 成本。

当前状态：2026-07-09 精确空间索引实测失败，已回退；仅保留热路径微优化。

触发原因：

- 显示同步后，大局可以跑到约 60FPS，但 `update=11.90~12.47ms`。
- `draw=1.90~2.05ms`，绘制不是主瓶颈。
- `target/frame=678~724` 仍是引擎主要扫描来源。
- `update + draw` 已接近 60Hz 的 `16.7ms` 帧预算，偶发 `maxInterval=32ms`。

已验证失败项：

- 每帧开始按阵营构建目标空间桶。
- 最近猎物/天敌查询从“扫描目标阵营数组”改成“从所在格向外扩圈查询”。
- 查询停止条件使用未访问区域的最小可能距离，保证仍然找最近目标，不改克制规则和转化规则。
- 单位移动后如果跨格，只同步它自己的目标桶位置，尽量保持逐个更新语义。

真机结果：

```text
scale=large fps=31.1~32.1 interval=31.2~31.8ms maxInterval=31~50ms update=21.52~23.16ms target/frame=434~624
```

判断：

- `target/frame` 确实下降，但 `update` 从约 12ms 恶化到 22ms。
- 对 36 个实体来说，精确空间索引的扩圈、桶维护和停止条件计算成本大于扫描收益。
- 该方案不能保留。

保留项：

- `normalized()` 临时对象已删除，改成局部数值计算。
- 逃离逻辑先用平方距离判断是否进入逃离范围，减少不必要开方。

收益：

- 热路径对象分配减少，降低 GC 抖动风险。
- 玩法内容不变：阵营、数量、碰撞、转化、结算、音效不变。

验收：

- 回退后大局 `update` 应从空间索引版 `21~23ms` 回落到约 `12ms`。
- `fps` 应恢复接近 60，而不是停留在 32 左右。
- 游戏规则、清场/限时结算、碰撞转化表现不退化。

### 3.4 第四阶段：显示同步帧驱动验证

当前状态：2026-07-09 已实现 `displaySync` 驱动，待真机复测。

背景：

- P1/P2 后，大局 `update` 已从约 `21.73ms` 降到约 `8.97ms`。
- 用户仍反馈体感有卡顿，因此问题更可能从“单帧计算过重”转向“30FPS timer 帧节奏不够顺”。
- 当前 `setInterval(33ms)` 已替换为 `@kit.ArkGraphics2D` 的 `displaySync`，目标帧率为 60。

实现要点：

- `RpsBattlePage` 持有 `displaySync.DisplaySync` 驱动。
- `startLoop()` 创建并 `start()` 显示同步器，设置 `ExpectedFrameRateRange` 为 `60/60/60`。
- `stopLoop()` 只停止显示同步器，暂停、结算、返回大厅都会停循环。
- `aboutToDisappear()` 额外 `off('frame')` 解绑回调，避免页面销毁后仍持有组件引用。
- `step()` 使用回调时间戳计算 `dt`，并兼容纳秒、微秒、毫秒三种可能单位。
- 性能日志新增 `loop=displaySync targetFps=60`，方便区分帧驱动版本。

风险：

- 60Hz 会让每秒 update/draw 次数翻倍，如果真机日志显示 `update + draw` 接近或超过 16ms，仍可能掉帧。
- 如果大局模式依然卡，应优先继续压 `update/draw`，而不是继续提高帧率。
- 需要重点观察音效触发期间的 `maxInterval` 是否出现尖峰。

## 四、推荐实施顺序

### PR1：阵营缓存与增量计数

改动文件：

- `RpsBattleEngine.ets`
- 必要时补充 `RpsBattleTypes.ets`

实现要点：

- 新增阵营缓存结构。
- `reset()` 生成实体时同步加入缓存。
- `convertEntity()` 转化时迁移缓存并更新 counts。
- `nearestEntity()` 改成接收候选数组，而不是扫描全体实体。
- `counts()` 返回当前 counts 拷贝，避免外部直接改内部对象。
- `resultAt()` 使用当前 counts。

预期日志：

```text
standard target/frame: 1152 -> 约 384
large target/frame: 2592 -> 约 864
```

### PR2：碰撞空间分桶

改动文件：

- `RpsBattleEngine.ets`

实现要点：

- 新增临时网格桶。
- 每帧碰撞前重建桶。
- 只检查当前格和相邻格。
- 保持 `lastCollisionCheckCount` 统计真实检查次数。

预期日志：

```text
standard collision/frame: 276 -> 明显下降
large collision/frame: 630 -> 明显下降
```

### PR3：热路径对象分配优化

改动文件：

- `RpsBattleEngine.ets`

实现要点：

- 消除 `normalized()` 热路径临时对象。
- 逃离逻辑先用平方距离过滤，减少不必要开方。
- 保持外部接口不变。

预期日志：

```text
large update: 空间索引版 21~23ms -> 回落到约 12ms
```

不再保留：

- 精确目标空间索引。
- 每帧目标桶构建。
- 移动后目标桶同步。

### PR4：显示同步帧驱动

改动文件：

- `RpsBattlePage.ets`

实现要点：

- 使用 `displaySync` 替代 `setInterval(33ms)`。
- 目标帧率先设为 60。
- 保持玩法、HUD 节流、音效节流和结算逻辑不变。

复测重点：

- 日志应出现 `loop=displaySync targetFps=60`。
- 如果真机能稳定调度，`fps` 应接近 60，`interval` 应接近 16 到 17ms。
- 如果 `fps` 仍接近 30，说明系统或设备把实际调度降到了 30，需要继续看 `maxInterval` 和体感。
- 如果 `update/draw` 变高导致 `maxInterval` 尖峰，下一步做渲染降级或动态帧率。

## 五、验收标准

### 功能验收

- 选择阵营、选择规模、开始游戏正常。
- 单位移动、追捕、逃离、碰撞转化正常。
- 三阵营克制关系不变。
- 清场结算、限时结算、平局逻辑不变。
- 暂停、继续、重新选择、返回大厅正常。
- 音效触发不变。

### 性能验收

标准局目标：

- `fps` 接近 60。
- `maxInterval` 大部分时间不超过 20ms。
- `update + draw` 稳定低于 16.7ms。
- `target/frame` 明显低于初始 1152。

大局目标：

- `fps` 接近 60，或在动态帧率策略下稳定接近目标帧率。
- `maxInterval` 大部分时间不超过 25ms，尽量避免 32ms 掉帧。
- `update + draw` 稳定低于 16.7ms。
- `target/frame` 明显低于 60Hz 基线 678 到 724。
- `collision/frame` 在 PR2 后明显低于 630。

## 六、暂不做

- 不先把 FPS 提到 60。
- 不先接新素材。
- 不先做 Worker/TaskPool 主循环模拟。
- 不为了性能删除碰撞、追捕、逃离、转化这些核心玩法。

## 七、结论

当前最该先做的是 `RpsBattleEngine` 的阵营缓存与增量计数。它能直接打掉最大的一块扫描成本，风险比空间分桶低，也不会明显改变玩法手感。

阵营缓存完成并复测后，再根据日志决定第二刀是否马上做碰撞空间分桶。如果 `target/frame` 下降明显但 `update` 仍偏高，第二刀就做空间分桶；如果 `update` 已经足够低但 `maxInterval` 仍抖，再评估帧驱动。
