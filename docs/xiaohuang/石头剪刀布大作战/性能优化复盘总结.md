# 石头剪刀布大作战性能优化复盘总结

日期：2026-07-09
状态：大局模式第一轮性能优化收口

## 一、结论

本轮优化目标是在不改变基础玩法的前提下，让新版 `rpsBattle` 大局模式从明显卡顿逐步稳定到可玩的 60Hz 体验。

当前保留的有效优化：

- 使用 `displaySync` 替代 30FPS timer 驱动。
- 运行中不再每帧向 ArkUI `@State` 写完整实体数组。
- HUD 降频同步。
- 阵营缓存和增量计数。
- 碰撞空间分桶。
- 碰撞分桶只清理、遍历有实体的桶。
- 碰撞只检查同桶和 4 个唯一邻桶方向，避免重复邻域遍历。
- 阵营缓存迁移使用 O(1) 下标表和 swap-remove。
- SoundPool 播放参数补齐 `rate`，转化音效保留节流。
- 保留性能日志，但只做每秒聚合输出。

最终真机日志里，大局模式出现过稳定段：

```text
scale=large entities=36 fps=64.0 interval=15.6ms maxInterval=16ms update=0.09~0.16ms draw=0.52~0.77ms
```

注意：`update/draw/targetMs` 低到 0.xms 后，`Date.now()` 的毫秒级精度已不适合继续做精确分段判断。最终阶段最可信的指标是 `fps`、`interval`、`maxInterval` 和体感。

## 二、优化过程

### 2.1 初始问题：30FPS + 每帧计算偏重

初始大局日志：

```text
scale=large entities=36 fps=29.2 interval=34.1ms maxInterval=37ms update=21.73ms draw=2.00ms target/frame=2592 collision/frame=630
```

判断：

- `draw` 约 2ms，不是主瓶颈。
- `update` 约 22ms，是核心问题。
- `target/frame=2592`，说明每个单位每帧分别全量扫描猎物和天敌。
- `collision/frame=630`，说明碰撞是全体两两检查。

第一步方向是压引擎扫描量，而不是先改 UI 或素材。

### 2.2 去掉高频 ArkUI 状态同步

早期运行中不能每帧把完整实体数组 clone 到 `@State`。这会让 ArkUI 状态系统和布局刷新被拖进游戏主循环。

处理方式：

- Canvas 绘制直接从 `engine.entitiesForRender()` 读取实体引用。
- HUD 只定时同步人数、时间、转化数。
- 暂停、结算、返回大厅时停止循环。

收益：

- 避免每帧触发大对象状态更新。
- 后续性能日志能更准确地反映引擎和 Canvas 的成本。

注意：

- 游戏运行态实体不要放进 `@State` 高频刷新。
- 只把页面真正需要响应式显示的少量数据放进 `@State`。

### 2.3 阵营缓存和增量计数

原逻辑：

- 找猎物扫描全部实体。
- 找天敌扫描全部实体。
- 结算和 HUD 可能重复统计人数。

优化：

- 引擎维护 `rockEntities`、`scissorsEntities`、`paperEntities`。
- 找猎物只扫描被克制阵营。
- 找天敌只扫描克制自己的阵营。
- `countsCache` 随转化增量更新。

效果：

- 大局 `target/frame` 从 2592 降到数百级。
- `update` 从约 22ms 降到约 9ms 到 12ms 区间。

注意：

- 转化时必须同时维护实体 `faction`、阵营缓存、人数缓存。
- 缓存不同步会直接导致胜负判断、HUD、追捕目标错误。

### 2.4 碰撞空间分桶

原逻辑：

```text
36 * 35 / 2 = 630
```

每帧检查全部实体对。

优化：

- 按坐标把实体放入均匀网格。
- 只检查当前格和相邻格。
- 后续进一步改成只遍历有实体的桶。
- 再进一步改成同桶内部 + 右、左下、下、右下 4 个唯一邻桶方向。

效果过程：

```text
collision/frame: 630 -> 约 80~160
collisionMs: 4~5ms -> 约 3~4ms
```

注意：

- 格子不能太小，否则可能漏掉跨格碰撞。
- 格子不能太大，否则分桶收益下降。
- 遍历邻桶时要避免重复检查同一对实体。
- 当前做法不会改变碰撞规则，只减少重复遍历和空桶成本。

### 2.5 `displaySync` 帧驱动

原先 timer 约 33ms，天然接近 30FPS。

优化：

- 改用 `@kit.ArkGraphics2D` 的 `displaySync`。
- 设置 `ExpectedFrameRateRange` 为 60。
- 使用回调时间戳计算 `dt`。
- 页面销毁、暂停、结算时停止并解绑。

效果：

```text
standard fps=61.9 interval=16.1ms
large fps=56~61 interval=16~18ms
```

注意：

- `displaySync` 只是期望帧率，不保证系统一定给 60。
- 60FPS 的前提是 `update + draw` 稳定低于 16.7ms。
- 如果引擎还没压下来，盲目上 60FPS 会把每秒计算量翻倍。

### 2.6 SoundPool 参数和音效节流

问题日志：

```text
can not find rate property
can not find parallelPlayFlag property
Failed to obtain ability in FA mode
```

处理：

- `PlayParameters` 补 `rate: audio.AudioRendererRate.RENDER_RATE_NORMAL`。
- 转化音效保留 120ms 节流。
- 性能日志记录 `audio` 和 `sfx/frame`。

结果：

- `can not find rate property` 消失。
- `audio` 多数为 0ms 到 0.04ms。
- 音效不是当前主瓶颈。

注意：

- 当前 SDK 类型里没有 `parallelPlayFlag` 字段，不要为了压日志硬塞非类型字段。
- `parallelPlayFlag` 目前不作为性能主因处理。
- 结算音效和按钮音效不要走高频节流。

## 三、失败方案

### 3.1 精确目标空间索引失败

方案：

- 每帧按阵营建立目标空间桶。
- 最近目标从所在格向外扩圈搜索。
- 用停止条件保证仍然找最近目标。

真机结果：

```text
fps=31.1~32.1 update=21.52~23.16ms target/frame=434~624
```

判断：

- `target/frame` 降了。
- 但 `update` 从约 12ms 恶化到 22ms。
- 对 36 个实体来说，扩圈、桶维护、停止条件计算的成本高于收益。

结论：

- 不保留。
- 当前规模不适合复杂精确空间索引。

### 3.2 成对预计算最近目标失败

方案：

- 每帧对不同阵营实体对只算一次距离。
- 同时更新双方最近猎物和天敌。

预期：

- 减少重复距离计算。
- 保持最近目标规则。

真机结果：

```text
fps=39.9~54.6 update=11.15~14.61ms move=7.22~11.27ms targetMs=5.82~9.66ms
```

判断：

- `target/frame` 数字下降。
- 但 `targetMs` 明显恶化。
- ArkTS 真机上数组写入、双向更新、分支逻辑成本更高。

结论：

- 已回退。
- 不能只看扫描次数，必须看真机耗时。

### 3.3 逃跑半径剪枝收益不足

方案：

- 天敌只在 `ENTITY_FLEE_DISTANCE` 内影响逃跑。
- 对天敌搜索加入半径剪枝。

结果：

- 仍然要扫描目标阵营数组。
- 阵营数量不均衡时 `targetMs` 仍会上升。

结论：

- 可以作为小优化思路，但不是决定性方案。
- 当前不继续沿这个方向堆复杂逻辑。

## 四、性能日志怎么读

当前核心日志：

```text
RpsBattlePerf loop=displaySync targetFps=60 scale=large entities=36 frames=...
fps=...
interval=...
maxInterval=...
update=...
draw=...
move=...
targetMs=...
collisionMs=...
convertMs=...
audio=...
hud=...
sfx/frame=...
target/frame=...
collision/frame=...
conversions/frame=...
totalConversions=...
```

优先级：

1. 先看 `maxInterval`：是否出现 31ms 或更高。
2. 再看 `fps` 和 `interval`：是否持续低于目标。
3. 再看 `update + draw`：是否接近或超过 16.7ms。
4. 如果 `update` 高，再拆 `move/targetMs/collisionMs/convertMs`。
5. 如果 `audio/hud/draw` 很低，就不要把优化方向放错。

注意：

- `Date.now()` 精度只有毫秒级，分段值低到 0.xms 后只能看趋势。
- 每秒聚合日志比每帧日志安全，避免日志本身影响帧率。
- 真机日志优先级高于理论推导。

## 五、当前保留实现的注意点

### 5.1 阵营缓存

- `addToFactionCache()` 必须写入 `factionIndexById`。
- `removeFromFactionCache()` 使用 swap-remove。
- swap-remove 会改变阵营数组顺序，但不改变最近目标结果，除非出现完全等距目标。
- 如果未来逻辑依赖阵营数组顺序，必须重新评估。

### 5.2 碰撞分桶

- 只清理上一帧活跃桶，避免每帧清全图空桶。
- 只遍历活跃桶，避免空桶循环。
- 只检查唯一邻桶方向，避免同一对实体重复检查。
- 如果未来实体半径、速度或格子尺寸变化，要重新验证是否漏碰撞。

### 5.3 目标搜索

- 当前保持“按阵营数组扫描最近目标”的保守方案。
- 不再使用精确空间索引或成对预计算。
- 后续如果要继续优化目标搜索，必须先用真机日志证明目标搜索重新成为主瓶颈。

### 5.4 音效

- `rate` 参数保留。
- 高频转化音效继续节流。
- `parallelPlayFlag` 日志暂不处理，除非后续确认它影响性能或音效体验。

### 5.5 性能日志

- 开发期可以保持 `RPS_BATTLE_ENABLE_PERF_LOG = true`。
- 准备发体验版或正式版时应关闭。
- 如果继续调优，建议下一步改用更高精度计时或 HiDebug Trace，而不是继续依赖 `Date.now()` 分段。

## 六、后续建议

短期：

- 先保持当前实现，多跑几局大局。
- 如果连续大局 `maxInterval` 保持 16ms 左右，可以把性能优化阶段收口。
- 关闭正式体验里的性能日志。

中期：

- 接素材图前，先做资源预加载和复用设计。
- 不允许在帧循环里加载图片、创建重资源或频繁构造大对象。
- 大局如果偶发卡顿但规则稳定，可考虑自适应帧率，而不是继续堆复杂算法。

长期：

- 如果未来实体数量从 36 扩到 80 或 100，再考虑数据导向结构、TypedArray 或 Worker。
- Worker 方案必须同时解决数据传输、UI 绘制、音效事件和生命周期同步，否则很容易收益为负。

## 七、经验原则

1. 先看真机日志，不凭理论判断。
2. 降扫描次数不等于降耗时，ArkTS 里的额外数组写入、分支和对象访问可能更贵。
3. 不改玩法的优化优先：缓存、降重复、降状态同步、降空循环。
4. 复杂算法必须可回退，并且只以真机 `update/maxInterval/fps` 验收。
5. `draw` 不高时不要过早重构渲染。
6. 音效日志多不等于音效是主瓶颈，要看 `audio` 耗时和帧尖峰是否同步。
7. 性能日志是开发工具，最终要关闭或降级。
