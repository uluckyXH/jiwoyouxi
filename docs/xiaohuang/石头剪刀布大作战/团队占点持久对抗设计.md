# 石头剪刀布大作战团队占点持久对抗设计

日期：2026-07-21
状态：P1 结算和援军已实现，P2 占点 AI 第一版已实现，P3/P4 待讨论
适用范围：新版 `entry/src/main/ets/games/rpsBattle/` 的 `团队占点` 模式

## 背景

改动前的 `团队占点` 已经接入中心据点、黑洞、绝地求生、能量道具和团队道具，但真机体验仍然容易滑向“谁吞噬得快谁赢”。

原因不是单个数值，而是模式目标权重还偏向基础乱斗：

- 单位 AI 仍以追猎物、躲天敌为主要目标。
- `clear` 清场仍然可以提前结束团队占点。
- 当前据点只有一个全局 `score + owner`，更像“最后控制进度的人拿走胜利”，不是三方长期经营领地。
- 黑洞、分裂和团队道具会加剧人数波动，战局容易快速滚雪球。
- 复活是事件补偿，不是稳定的“战线续航”系统。

所以如果只继续提高复活概率、加快道具刷新，能延长一些战斗，但不能真正把模式从“吞噬清场”拉回“领地争夺”。

## 实施记录

2026-07-21 已完成 P1 最小改动：

- `团队占点` 不再因为只剩一个阵营或清场而立即结束，结算优先来自据点到 `100` 或时间到。
- `经典乱斗` 继续保留清场结算，不受本轮规则改变影响。
- 绝地求生从一次性复活改为冷却援军：阵营剩余 `3` 个以内且数量下降时排复活印记。
- 阵营团灭后，只要绝地求生开着、冷却结束、对局未结算，仍可再次排援军回场。
- 同阵营援军冷却为 `5s`，同一时间最多一个复活印记。
- 复活延迟为 `2.2s`，复活单位速度倍率为 `1.18`。
- 复活走 `RpsBattleLastStandSystem`，性能日志继续使用 `lastStandMs`、`lastStandEvents/frame`、`lastStandRevives/frame`。
- 复活可使用最多 `3` 个援军预备槽，避免满员大局下团灭阵营完全无法回场；非复活分裂仍受 `RPS_BATTLE_MAX_TOTAL_ENTITIES` 保护。
- 复活生成点会避开激活中的黑洞影响圈：优先使用复活印记位置，若危险则回退到阵营出生点，出生点仍危险时使用远离黑洞的战场角落。
- `团队占点` 已加入轻量占点 AI：一部分单位稳定向据点争夺或驻守，天敌靠近时仍优先保留躲避，避免破坏石头剪刀布追逃核心。

## 设计目标

1. `团队占点` 的主要胜负压力来自据点分数和时间，而不是清场。
2. 单位行为要更明显地围绕据点移动、驻守和反抢。
3. 弱势方可以持续获得有限援军，让战线不断被拉回据点附近。
4. 不改变 `经典乱斗` 的清场爽感。
5. 不引入 WebView / HTML / 非原生方案，继续使用 ArkTS 规则系统和 Canvas 渲染。
6. 继续保持性能可测，新增逻辑不能每帧创建大量对象或刷新大量 ArkUI 状态。

## 当前机制问题

### 1. 清场优先级太高

历史实现中，`RpsBattleEngine.resultAt()` 在 `aliveCount <= 1` 且没有等待复活印记时，会直接以 `clear` 结束。

这对经典乱斗是合理的，但对团队占点会产生两个问题：

- 单位更快吞噬对手时，玩家还没感受到据点争夺，对局就结束了。
- 据点分数没有成为“主线目标”，更像清场之外的附加结算。

### 2. 单位没有足够占点意图

基础移动逻辑主要围绕三方克制：

```text
追自己的猎物
躲自己的天敌
没有明确目标时游走
```

团队占点需要一部分单位主动进入据点、留在据点、反抢据点。否则中心据点只是单位路径上的偶然交汇点。

### 3. 单一据点分数不够像领地

当前据点结构是：

```text
owner
capturingFaction
score
pressure
```

只要某一方唯一最多，就成为 owner 并推动同一个 `score` 增长。这个模型简单，但领地感不足：

- 三方各自累计了多少占点成果不直观。
- 后手抢到 owner 后，容易像“拿走共同进度”。
- 玩家更难理解为什么这局是“占领赢”，而不是“最后谁站在圈里谁赢”。

### 4. 弱势续航还是事件型

历史版本中，绝地求生更接近一次性事件。当前已改为剩余 `3` 个以内可触发冷却援军，但仍要通过真机观察复活频率和战线拉扯是否自然。

## 推荐方向

推荐按四层处理，先完成 P1/P2 的低风险版本，不一次性做复杂领地系统。

```text
P1 结算和援军：团队占点不再轻易清场结束，弱势方有有限持续援军
P2 占点 AI：单位有占点、驻守、反抢意图
P3 分数模型：从单一 owner 分数改为三方据点进度
P4 多点领地：再考虑多个据点或资源点
```

## P1：结算和援军

### 规则

团队占点应从“清场优先”改成“据点 / 时间优先”。

建议第一版：

- `经典乱斗` 保持当前清场结算。
- `团队占点` 中，如果时间未结束、据点未到 `100`，不因为某阵营清零立即结束。
- 当某阵营人数低于阈值时，进入援军队列。
- 援军从该阵营出生点或战场边缘安全点进入。
- 如果据点达到 `100`，仍然立即结算。
- 如果时间到，按据点分数或人数/分数混合规则结算。

### 援军触发

建议替代当前“一次性复活”的团队占点专用续航规则：

| 参数 | 建议值 |
| --- | --- |
| 触发条件 | 阵营存活数 `<= 3` |
| 清零处理 | 清零也不立即结束，进入援军倒计时 |
| 援军延迟 | `2.0s..3.0s` |
| 每次援军数 | `1` 或 `2` |
| 援军冷却 | 同阵营 `5s` 内最多触发一次 |
| 最低保障 | 阵营为 `0` 时优先补到 `1` |
| 性能上限 | 不突破 `RPS_BATTLE_MAX_TOTAL_ENTITIES` |

当前实现补充：复活允许使用最多 `3` 个援军预备槽，用来处理满员大局中阵营团灭后没有空位的问题；这不是无限扩容，仍然是小规模兜底。

这比“无限复活”更可控：它不是给每个阵营无限堆人数，而是让弱势方能重新回到场上参与据点。

### 清场结束兜底

为了避免极端死循环，保留兜底：

- 如果只剩一个阵营，且另外两方援军都无法生成，才允许清场结束。
- 无法生成的条件包括：时间已到、总人数上限长期满、模式关闭援军、或进入结算阶段。

第一版可以更简单：

```text
团队占点不按 clear 结束，只按 zone/time 结束
```

这样最容易验证“模式是否从吞噬清场转向占点”。

## P2：占点 AI

当前第一版已实现：在 `RpsBattleEngine.applyMovement()` 中按 `entity.id` 稳定分配轻量行为权重，不每帧随机、不额外创建实体状态。

实现规则：

- 约 `45%` 单位作为争夺者：本方不是据点 owner 时，向中心据点叠加较强移动意图。
- 约 `25%` 单位作为守卫者：本方占领或正在占领时，向据点附近聚集。
- 少量非 owner 单位提供支援意图，其余继续保持原有追猎、躲避和游走。
- 单位已在本方据点圈内时，占点意图降低，减少圈内无意义抖动。
- 天敌进入危险距离时，占点意图降低，追逃克制仍然优先。

这不是新的玩法状态机，只是在原生 ArkTS 移动系统中增加一个低成本向量，目的是让战局更自然地围绕据点拉扯。

单位需要有明确的占点行为，而不是所有单位都一直追猎物。

### 角色分配

不要每帧随机分配角色。用稳定规则按 `entity.id` 分组：

| 角色 | 比例 | 行为 |
| --- | --- | --- |
| 争夺者 | 约 `45%` | 朝据点移动，优先进入圈内 |
| 守卫者 | 约 `25%` | 本方占优时留在据点附近，防止被反抢 |
| 游击者 | 约 `30%` | 保留当前追猎/躲避逻辑 |

这样不会破坏石头剪刀布核心克制，同时让战场重心稳定回到据点。

### 占点力

在移动系统中增加一个轻量向量，不替代原有追逃，只叠加：

```text
zoneIntent = normalize(zoneCenter - entityPosition) * zoneIntentStrength
```

建议强度：

| 状态 | 占点力 |
| --- | --- |
| 本方不是 owner | 高 |
| 本方是 owner 但据点压力不足 | 中 |
| 本方明显占优 | 低，更多单位游击 |
| 单位附近有天敌 | 降低占点力，优先躲避 |

关键点：占点力要被天敌躲避压制，否则单位会不合理地直冲据点被吞。

## P3：三方据点进度

当前 `score + owner` 可以保留为原版模式，但如果要强调领地，建议团队占点新版改为三方独立分数：

```ts
zoneScores: RpsBattleFactionCounts
```

规则：

- 据点内唯一压力最高的一方获得进度。
- 只给当前 `capturingFaction` 的分数增长。
- 三方分数各自保留。
- 任一方分数到 `100` 获胜。
- 僵持时不涨分。

示例：

```text
石头 42
剪刀 35
布 18
```

好处：

- 玩家能看到每个阵营真实占点成果。
- 反抢不会“继承别人的进度”，更像三方抢地盘。
- 时间结束时可以直接按最高据点分结算。

### UI 表达

HUD 不再只显示：

```text
据点 石头 73
```

改成更清楚的三方进度：

```text
据点  石头 42  剪刀 35  布 18
```

Canvas 据点外圈可以画三段进度弧：

- 石头一段。
- 剪刀一段。
- 布一段。
- 当前正在占领的一方外圈更亮。

这会显著强化“领地积分”感。

## P4：多点领地

不建议现在马上做多据点。当前手机竖屏空间有限，单位数量也还在性能保护范围内。多据点会引入：

- AI 分兵。
- 多个压力统计。
- 多个 HUD 状态。
- 更复杂的胜负解释。

如果 P1 到 P3 后仍然觉得占点不够，可以再做轻量三点布局：

| 据点 | 位置 | 作用 |
| --- | --- | --- |
| 中央据点 | 中心 | 主要得分，权重 `1.0` |
| 左侧资源点 | 左中 | 慢速得分，权重 `0.45` |
| 右侧资源点 | 右中 | 慢速得分，权重 `0.45` |

第一版多点不要做复杂“连线领地”或网格占领，先做三个圆形点即可。

## 黑洞和道具的调整

如果目标是让团队占点更持久，黑洞和道具都要服务于据点，而不是加速清场。

### 黑洞

建议：

- 黑洞默认仍可开启，但出现时间延后到 `14..22s`。
- 黑洞生成继续避开中心据点。
- 黑洞在团队占点中的成长上限可进一步降低。
- 黑洞更偏向“打散站位”，不要变成持续吞噬核心。

可选规则：

```text
如果黑洞在据点附近，吞噬成长倍率降低
```

这样保留危险事件，但减少它把占点局面直接打成清场局。

### 道具

道具刷新要诱导争夺空间：

- 道具优先刷新在据点外圈附近，不直接刷在出生点。
- 护盾和分裂比加速更有助于持久对抗。
- 团队分裂继续受人数上限限制，避免一帧暴涨。

当前 `加速 -> 护盾 -> 分裂` 循环是合适的，后续如果占点 AI 接入，可以考虑：

```text
护盾 -> 分裂 -> 加速
```

让防守和续航权重更高。

## 推荐实施顺序

### 第一轮：最小改动验证

目标：让团队占点不再很快被清场结束。

要做：

1. `zones` 模式不再按 `clear` 立即结算，优先 `zone/time`。
2. 低人数援军改为可冷却触发，不再每阵营只成功一次。
3. Canvas 继续复用复活印记倒计时和复活脉冲。
4. 性能日志复用 `lastStandMs`、`lastStandEvents/frame`、`lastStandRevives/frame`。

验收：

- 三方被吞到 `0` 后仍有机会回场。
- 对局结束更多来自据点到 `100` 或时间到。
- `fps`、`maxInterval` 不明显退化。

### 第二轮：占点 AI

目标：让单位主动围绕据点行动。

要做：

1. 给实体分配稳定角色：争夺、守卫、游击。已完成第一版。
2. 移动系统增加轻量 `zoneIntent`。已完成第一版。
3. 真机观察单位是否更常驻留据点。

验收：

- 圈内压力更稳定。
- 不会所有单位无脑冲圈被克制方吞光。
- 对局观感从“追逐乱跑”变成“围绕据点来回拉扯”。

### 第三轮：三方据点进度

目标：把占点胜利解释清楚。

要做：

1. `RpsBattleControlZone` 增加三方 `scores`。
2. 任一方分数到 `100` 以 `zone` 结算。
3. 时间到按三方分数最高结算。
4. HUD 和 Canvas 改成三方分段进度。

验收：

- 玩家能看懂每个阵营的占点成果。
- 后手抢点不会继承前手进度。
- 结算原因更像“领地胜利”。

## 不推荐的方案

### 只提高复活和道具

这会让对局变长，但不一定更像占点。单位仍会继续按吞噬逻辑滚雪球，只是滚得更久。

### 直接做多个据点

多据点会让 UI 和 AI 同时复杂化。当前真正的问题是目标权重，不是据点数量不够。

### 禁止转化

转化是石头剪刀布大作战的核心爽点，不能为了占点把它拿掉。应该降低“清场结束”的权重，而不是删除吞噬。

### 无限复活

无限复活会让战局失去结算压力。更好的方式是：限时、据点分数、援军冷却和人数上限共同约束。

## 当前建议结论

我建议下一步先做 **P1：团队占点持久化最小改动**：

- `团队占点` 先不按清场立即结束。
- 弱势方改为可冷却援军，而不是一次性复活。
- 据点到 `100` 和时间到仍然是核心结算。
- 先不做多据点，也不改经典乱斗。

这轮改动最能验证你的判断：如果清场不再抢走结算，并且弱势方能持续回到场上，玩家会不会明显感觉“这局是在抢地盘”。
