# 石头剪刀布大作战新玩法纵向配置卡片 UI 设计

日期：2026-07-17
状态：已实现，待 ArkTS 构建与真机验收
适用范围：新版 `entry/src/main/ets/games/rpsBattle/` 的开局配置页

## 结论

当前开局页把主题、阵营、玩法、机制、人数和时长连续堆在同一张大面板中；玩法摘要、机制行和高级人数又各自有一层浅色容器，导致层级相近、阅读路径不清楚。

建议改为一列自上而下的配置卡：

```text
玩法方案 -> 本局机制 -> 阵营主题 -> 支持阵营 -> 战斗规模 -> 对局时长 -> 开始乱斗
```

这不是向导式强制流程。玩家仍可直接修改任意一项；编号只用于建立视觉阅读顺序。所有已有配置、默认值和战斗行为保持不变，本轮只重组开局 UI。

## 实施记录

2026-07-17 已完成开局配置页的首轮实现：

- 配置页已改为模式、机制、主题、支持阵营、规模和时长的纵向卡片流。
- 团队占点机制卡保留中心据点锁定、黑洞开关和后续机制状态；经典乱斗不显示空机制卡。
- 开始乱斗改为固定底部动作区，并显示本局模式、可调机制、规模和时长摘要。
- 高级人数调整收进战斗规模卡，计数器改为三行布局，`+`、`-` 点击区域为 `40vp`。
- `RpsBattlePage.ets` 已拆出配置屏、顶部栏、玩法机制卡和倒计时覆盖层，页面与设置面板均控制在 800 行以内。

本次没有改动战斗引擎、模式规则、人数校验、音频和 Canvas 渲染；构建与真机结果见 `测试报告/2026-07-17-rpsBattle纵向配置卡片UI.md`。

## 当前问题

当前 `RpsBattleSetupPanel.ets` 的结构有三个主要问题：

1. 页面最外层是一张大卡，内部再连续出现主题格、玩法格、玩法摘要框、机制框、高级人数框、人数控制框和时长格，视觉上形成多层同类容器。
2. `玩法方案` 是决定本局规则的主决策，却排在主题和支持阵营之后；玩家需要先经过展示身份设置，才能看到真正影响玩法的选项。
3. `阵容规模`、`高级调整` 和展开后的三方人数控制是同一个决策的不同层级，但当前被拆成彼此相邻的独立块，容易让人误以为是三套不同设置。

改版重点是让每张卡只回答一个问题，并让当前选择结果在卡内直接可见。

## 设计目标

- 让“这一局玩什么”成为最先看到、最清晰的决策。
- 把模式相关机制放在模式之后，避免机制脱离上下文。
- 保留主题、支持阵营、规模、同步人数、独立人数和时长的全部已有能力。
- 不把卡片再套进卡片；卡内选项是扁平的选择控件、分段控件或列表行。
- 保持暖色纸张、深棕文字和主题阵营色，不新增素材加载、Canvas 绘制或重型动画。
- 继续使用 HarmonyOS 原生 ArkTS / ArkUI；不改变引擎、规则、音频和性能主循环。

## 信息架构

| 顺序 | 配置卡 | 玩家要决定的内容 | 现有状态 | 改版后的呈现 |
| --- | --- | --- | --- | --- |
| 01 | 玩法方案 | 本局的目标和模式 | `selectedModeKey` | 2 x 2 玩法选择控件，卡底展示目标和当前机制摘要 |
| 02 | 本局机制 | 当前模式带哪些可调增强机制 | `mechanicSelection` | 仅展示当前模式的机制；核心锁定、已实现可开关、后续不可用 |
| 03 | 阵营主题 | 使用哪一组阵营名称、图标和颜色 | `selectedThemeKey` | 两列主题选择控件 |
| 04 | 支持阵营 | 玩家支持哪一方 | `selectedFaction` | 三列阵营选择控件，主题切换后立即刷新名称和颜色 |

### 条件展示规则

- `经典乱斗` 没有可调机制时，不渲染“本局机制”卡，玩法卡内只保留核心机制摘要。
- `团队占点` 显示“本局机制”卡：中心据点为核心锁定，黑洞默认开启且可关闭，未迁移机制显示“后续”。
- `小心叛徒`、`众生平等` 在机制尚未迁移完成前仍显示机制卡，但所有未实现项保持不可操作的“后续”状态；不伪造可以生效的开关。
- 高级人数默认收起；展开内容在“战斗规模”卡内原位出现，不额外生成新的浮层或嵌套卡。
- 当经典乱斗隐藏机制卡时，后续卡号自动前移，保证可见的配置步骤连续。

## 卡片详细设计

### 页面标题区

标题区不再重复使用大面板标题样式，只保留紧凑的页面识别和当前玩法摘要：

```text
石头剪刀布大作战
团队占点 · 占点到100或限时领先
```

- 标题保持现有顶部导航中的游戏名，不增加营销式大标题。
- 副行跟随 `selectedModeKey` 更新，帮助玩家确认当前正在配置的玩法。
- 顶部导航的“大厅”入口保持不变。

### 01 玩法方案

这是首张配置卡，也是视觉优先级最高的一张。

- 保持现有四个模式和 2 x 2 布局：经典乱斗、团队占点、小心叛徒、众生平等。
- 每个模式控件仍显示图标、名称和一行副标题，但使用扁平选择按钮，不再被另一个描边容器包裹。
- 选中模式下方使用同一卡内的分隔线和文本摘要，不再独立渲染 `ModeSummary` 小卡。
- 摘要固定为“目标 / 已启用 / 后续”三类；没有后续项时不显示对应行。

模式切换继续沿用现有规则：切换后将 `selectedMechanics` 重置为该模式的默认机制选择，不改变主题、支持阵营、人数或时长。

### 02 本局机制

机制卡只在当前模式存在机制信息时显示。它不是“全部规则库”，只展示已选模式相关的机制。

每行由左到右组成：

```text
机制标记 | 名称与一行说明 | 状态或开关
```

三种状态保持现有语义：

| 类型 | 右侧呈现 | 可操作性 |
| --- | --- | --- |
| 核心机制 | `核心` | 不可关闭 |
| 已实现增强机制 | 原生开关，默认开 | 可在开局前关闭 |
| 后续机制 | `后续` | 不可点击 |

黑洞继续遵守已有设计：团队占点默认开启、玩家可在开局前关闭；中心据点不可关闭。机制卡的重排不改变该规则。

### 03 阵营主题

主题仅影响阵营展示身份，不改变玩法规则。

- 使用两列等宽选择控件。
- 选中态采用主题阵营强调色，未选中态为浅纸色和细边框。
- 主题名称过长时沿用现有动态字号策略，不能因屏幕宽度缩放整页字体。

### 04 支持阵营

支持阵营独立成卡，避免与主题选择混在一组。

- 固定三列，列宽稳定。
- 每列展示阵营符号、名称和“克制对象”短文本。
- 切换主题后保留当前阵营 ID，名称、图标和颜色随主题配置刷新。
- 这只影响玩家代入和结算反馈，不改变该阵营的初始人数或克制规则。

### 05 战斗规模

规模相关内容收敛到一张卡内，分为“快捷规模”和“高级调整”两个层级。

```text
快捷规模：小局 / 标准 / 大局
高级调整：收起或展开
展开后：一起调 / 单独调 + 三方计数器
```

设计规则：

- 快捷规模下方显示总人数和上限，当前数值随 `lineupCounts` 更新。
- `高级调整` 是整行可点击的展开控制，右侧使用方向图标或展开状态；不把“展开”做成难以点击的小文字按钮。
- 展开后直接在卡内追加分隔线、人数模式分段控件和三行计数器。计数器使用稳定高度，`+`、`-` 点击热区不小于 `40vp`。
- “一起调”和“单独调”是互斥模式，继续复用现有同步人数逻辑与总人数上限校验。
- 展开或收起只改变该卡高度，不改变其他卡的选中状态，也不重置人数。

### 06 对局时长

- 保留现有时长选项及默认值。
- 使用一行等宽的分段选择控件，而不是另一层带阴影的小卡。
- 选中态与规模、主题、模式的选中态使用同一套描边和填充语义。

### 固定底部动作区

开始操作不放在最后一张设置卡内部，而是独立为页面底部动作区：

```text
团队占点 · 黑洞开启 · 标准 · 60 秒
[开始乱斗]
```

- 在 `Stack` 根层覆盖一个固定底部栏，`Scroll` 的底部预留足够空白，避免最后一张卡被遮挡。
- 摘要只展示用户最需要复核的四项：模式、可调机制状态、规模、时长；不塞入所有阵营详情。
- 按钮使用现有主按钮色和文案，不改变 `startGame()` 调用时机。
- 卡片滚动时动作区不移动，减少用户反复滑到底部才能开局的成本。

## 视觉与交互规范

### 层级

- 页面背景继续使用当前暖色底。
- 每个配置卡是唯一一级表面：浅色底、1px 细边、`8vp` 圆角、无重阴影或仅极轻阴影。
- 卡内的模式格、主题格、规模格、时长格属于选择控件，不再使用与外层卡相同的“浮起卡片”阴影。
- 卡与卡间距建议 `12vp`，卡内水平内边距建议 `14vp`，保证手机端扫描节奏。
- 标题使用 `14-16fp`，说明使用 `11-12fp`，不使用视口比例字号。

### 动效

只保留与状态变化有关的轻量 ArkUI 动效：

- 切换模式时，模式摘要和机制卡以短距离淡入淡出更新。
- 展开高级人数时，卡内内容做短时高度过渡。
- 使用 `this.getUIContext().animateTo` 或项目已有动效 token；动画不能承载数据重置、人数校验或开局逻辑。

不做粒子、全屏转场、持续呼吸动画或资源型插画，避免让配置页显得嘈杂，也避免增加进入游戏前的加载开销。

### 无障碍与适配

- 所有点击区域不小于 `40vp x 40vp`。
- 模式、主题、时长等长文本必须在固定控件内完整显示；必要时使用现有动态字号，不做横向溢出。
- 小屏设备以单列卡片滚动为基线；平板和折叠屏仅增加内容最大宽度，不改变配置顺序。
- 固定底部栏需避开系统手势区，并为滚动内容留出底部安全空间。

## 实现边界与拆分建议

本轮 UI 改造不应继续向已接近上限的页面和设置面板堆叠 Builder。按 `docs/鸿蒙原生开发规范.md`，实现时应拆分游戏私有组件：

```text
games/rpsBattle/
  RpsBattleSetupScreen.ets          配置页滚动区和固定开始动作区
  RpsBattleSetupPanel.ets           主题、支持阵营、规模和时长配置卡
  RpsBattleSetupCardHeader.ets      配置卡编号、标题和说明
  RpsBattleModeSetupCard.ets        玩法方案、规则摘要和当前模式机制
  RpsBattleTopBar.ets               游戏内顶部导航和运行态操作
  RpsBattleCountdownOverlay.ets     开战倒计时覆盖层
```

约束：

- `RpsBattlePage.ets` 继续只维护页面生命周期、战斗状态和 `startGame()`；设置界面的布局不再继续增加到页面中。
- 规则、机制默认值、人数上下限和模式解析继续留在现有 `Rules.ets`、`ModeConfig.ets`、`Types.ets`，不为了 UI 重排复制逻辑。
- 每张配置卡只接收必要的 `@Prop` 和回调，避免将完整引擎或大状态对象传给多个组件。
- 不新增 Web、HTML、Node 运行时代码或资源预加载逻辑。

## 验收标准

实现后应验证以下内容：

1. 配置页以“玩法方案”开头，后续卡按本文顺序纵向排列。
2. 不再存在一张外层大卡包住所有配置卡的结构，也不出现卡片套卡片。
3. 团队占点下可看见中心据点锁定、黑洞开关和后续机制；关闭黑洞后开始游戏的运行规则保持现有正确行为。
4. 经典乱斗不会显示空的机制卡。
5. 主题切换、支持阵营、规模、同步/独立人数、三方加减和时长的现有行为不回退。
6. 切换模式仍只重置该模式的机制默认选择，不重置其他配置。
7. 小屏和常规手机屏中，文字、三列阵营、计数器和固定开始按钮均不重叠、不溢出。
8. `RpsBattlePerf` 的引擎更新与 Canvas 绘制指标不因配置页重排增加每帧开销；配置页只在用户点击时更新 ArkUI 状态。
9. 完成 ArkTS 构建、现有规则测试和真机手工验证，并在 `测试报告/` 记录构建环境、设备、截图或录屏路径。

## 实施顺序

1. 先拆配置 UI 组件，保留全部现有状态和回调，完成纵向卡片布局。
2. 再接固定底部动作区和配置摘要，不触碰 `startGame()` 的参数构造。
3. 完成构建与真机验证后，再根据实际屏幕截图微调间距、文案和字号。
4. 玩法机制继续按既定节奏逐项迁移；每新增一个已实现增强机制，直接复用“本局机制”卡中的开局前开关语义。

## 暂不做

- 不新增玩法机制，不改团队占点、黑洞、经典乱斗或未来模式的规则。
- 不重新设计战斗 HUD、Canvas 战场、暂停页或结算页。
- 不接模式插图、视频、网络资源或首屏加载动画。
- 不把配置流程做成必须逐步点击“下一步”的强制向导。
- 不开放尚未实现的机制开关。
