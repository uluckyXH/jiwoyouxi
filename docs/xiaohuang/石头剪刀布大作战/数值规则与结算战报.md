# 石头剪刀布大作战数值规则与结算战报

日期：2026-07-08
状态：根据现有代码梳理

## 一、基础数值

核心数值来自 `GameEngine.ets`。

| 数值 | 当前值 | 说明 |
| --- | --- | --- |
| 基础半径 | `13` | 生成单位时会额外随机 `0..3` |
| 基础速度 | `74` | 初始速度方向随机 |
| 单位速度 | `74..98` | `BASE_SPEED + random(0, 24)` |
| 最大速度 | `168` | 普通状态速度上限 |
| 碰撞推开系数 | `0.32` | 单位重叠时用于分离 |
| 帧间隔 | `33ms` | 页面主循环 `setInterval` |
| dt 限制 | `0.001..0.05s` | 防止后台恢复后模拟跳变 |
| 开局布局保持 | `1000ms` | 开局后先稳定布局再开始模拟 |

## 二、战场边界

边界由页面宽高和 UI 预留空间计算：

- 窄屏顶部预留 `118px`，宽屏顶部预留 `82px`。
- 窄屏底部预留 `82px`，宽屏底部预留 `72px`。
- 左右基础边距 `14px`。
- 开启缩圈后，边界额外加 `shrinkPadding`。

单位碰到边界会被限制在场内，并反向弹回，速度保留约 `88%`。

## 三、胜负判定

每帧更新后执行 `checkFinish()`。

### 3.1 清场胜利

如果存活阵营数小于等于 `1`：

- 唯一存活阵营获胜。
- 如果没有唯一领先者，则无人获胜。
- 结算原因：`清场`。

### 3.2 据点胜利

开启据点争夺时：

- 任一据点分数达到 `100`。
- 且据点 owner 合法。
- owner 阵营获胜。
- 结算原因：`据点达标`。

### 3.3 限时胜利

非死斗模式下：

- 当前时间超过 `durationSec`。
- 如果存在唯一领先阵营，该阵营获胜。
- 结算原因：`时间到`。

`leadingType()` 遇到并列会返回 `-1`。因此限时到点并列时不会产生明确胜者。

### 3.4 我要打十个胜利

“我要打十个”期间，如果劣势方把优势方清零：

- 劣势方立即获胜。
- 结算原因：`以弱胜强`。
- 同时记录“以弱胜强！”事件。

## 四、计分和大厅成绩

当前 `RpsPage` 在结算时调用：

```text
recordScore(max(石头人数, 剪刀人数, 布人数))
```

也就是说大厅记录的分数不是传统得分，而是结算时最大阵营人数。

重构建议：

- 如果新版继续记录最高分，建议定义更直观的得分模型。
- 可考虑：胜利人数、转化次数、耗时、翻盘倍率、事件奖励。
- 至少要在结算页解释“本局得分”来源。

## 五、赔率规则

下注赔率在页面层计算，不在引擎层计算。

公式：

```text
total = count0 + count1 + count2
prey = counts[(type + 1) % 3]
predator = counts[(type + 2) % 3]
strength = max(1.2, counts[type] + prey * 0.3 - predator * 0.18)
odds = clamp(total / strength * 0.72, 1.18, 4.8)
```

解读：

- 人数多，强度高，赔率低。
- 猎物多，说明该阵营更容易扩张，赔率低。
- 天敌多，说明该阵营更危险，赔率高。
- 赔率只在开局和人数配置变更时更新，不随局内实时变化动态变化。

## 六、战报数据

战报由 `BattleReportRecorder` 记录。

### 6.1 开局记录

每局开始时记录：

- 配置选项。
- 开局时间。
- 三方初始人数。
- 所有初始单位 id 和阵营。
- 第一条走势点：`开局`。

### 6.2 过程记录

战报会记录：

- 每秒一次人数走势点。
- 重要事件走势点。
- 阵营归零事件。
- 每次转化的贡献者。
- 被转化过的单位 id。

走势点最多保留 `80` 个。战报面板只显示最后 `9` 个走势点。

### 6.3 结算记录

结算时记录：

- 最终赢家。
- 结算原因。
- 最终人数。
- 存活单位 id。
- 对局耗时。
- 最终走势点：`结算`。

### 6.4 战报内容

战报摘要包含：

- 本局模式：死斗或限时。
- 初始阵容。
- 最多 `5` 条关键事件。
- 最终赢家、耗时和原因。

MVP 包含：

- 转化王：单个单位造成的最多转化次数。
- 全程幸存：从开局活到最后且未被转化的单位统计。
- 逆转：如果赢家开局不是最大阵营，且最终人数高于开局人数，会记录逆转描述。

## 七、事件音效

页面根据 `GameEventKind` 播放音效：

| 事件类型 | 音效 key | 触发例子 |
| --- | --- | --- |
| `Attack` | `attack` | 转化、反杀 |
| `Warning` | `warning` | 黑洞出现、叛徒、灭霸预警 |
| `Shrink` | `shrink` | 缩圈推进 |
| `Snap` | `snap` | 响指完成 |
| `Win` | `win` | 获胜或无人获胜 |
| `PowerUp` | `power` | 道具、复活、悬赏 |

胜利音效会暂停 BGM。

## 八、HUD 信息

游戏中 HUD 展示：

- 三方当前人数。
- 局势：剩余时间、死斗中、已暂停、据点分数或获胜状态。
- 积分：下注积分 `bank`。
- 赛制：多局比分、死斗或时长。
- 事件提示：最近事件文本，关闭通知后隐藏。
- 怒气条：我要打十个期间显示。

重构建议：

- HUD 应从“信息堆叠”改成“观战播报”。
- 关键事件提示要比普通转化更突出。
- 人数走势可以做成轻量条形图，强化三方变化。

## 九、验收点

重构时至少要覆盖这些规则：

- 石头、剪刀、布的循环克制正确。
- 单位会追猎物、躲天敌。
- 护盾能抵挡一次转化。
- 限时、死斗、据点三种结束路径正确。
- 黑洞吞噬是死亡，不是转化。
- 叛徒标记 `2s` 后改阵营。
- 绝地分裂和复活每阵营最多触发一次。
- 灭霸响指每阵营减半，且每局只触发一次。
- 道具过期、拾取和团队共享正确。
- 我要打十个期间是击杀优势方，不是转化。
- 多局赛制会给胜方下一局减员，但不低于 `3`。
- 下注胜负只影响本地积分，不影响模拟。
- 战报能记录赢家、原因、关键事件、转化王和走势。
