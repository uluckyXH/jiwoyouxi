# 石头剪刀布大作战重构代码映射与拆包建议

日期：2026-07-08
状态：重构前技术梳理

## 一、现有职责分布

### 1.1 页面层

`entry/src/main/ets/games/rps/RpsPage.ets`

当前承担：

- 页面生命周期。
- Canvas 创建和尺寸同步。
- 配置 UI。
- 游戏 HUD。
- 倒计时、暂停、结算、战报弹窗。
- 主循环 `setInterval(33ms)`。
- 音频初始化和事件音效分发。
- 多局赛制状态。
- 下注积分和赔率。
- 上帝之手触摸分发。

问题：

- 单文件超过 1400 行。
- 页面层同时处理 UI、赛制、下注、音频和引擎驱动。
- 配置面板过长，玩家首屏压力大。

### 1.2 引擎层

`entry/src/main/ets/model/GameEngine.ets`

当前承担：

- 单位生成、移动、追捕、躲避。
- 边界、障碍、碰撞和转化。
- 缩圈、黑洞、叛徒、灭霸、绝地、道具、据点。
- 悬赏系统和我要打十个系统调度。
- 胜负判断。
- 战报记录。
- 渲染状态组装。

问题：

- 引擎同时包含基础模拟和大量事件逻辑。
- 特殊事件之间耦合在同一 update 流程里。
- 随机逻辑直接使用 `Math.random()`，不利于回放和测试。

### 1.3 共享模型

`GameTypes.ets` 和 `GameRules.ets`

当前优点：

- 定义了 `GamePhase`、`FieldMode`、`GameEventKind`、`PowerUpKind`。
- 定义了 `NativeGameOptions`、`Entity`、`GameSnapshot`、`NativeRenderState`。
- `beats()`、`preyType()`、`predatorType()` 简洁明确。
- `countEntities()`、`aliveTypeCount()`、`leadingType()` 可直接复用。

重构建议：

- 类型可以搬进新版包或提取为 RPS 专用规则模块。
- `beats()` 必须单测覆盖。

### 1.4 子系统

| 文件 | 职责 | 可复用性 |
| --- | --- | --- |
| `BountySystem.ets` | 悬赏头名判断和奖励 | 规则可复用，表现需重做 |
| `TenAgainstOneSystem.ets` | 劣势方暴走反杀 | 强烈建议保留并重点打磨 |
| `BattleReport.ets` | 战报统计和摘要 | 可复用思路，数据结构可升级 |
| `CanvasRenderer.ets` | Canvas 几何渲染 | 不建议直接复用视觉 |
| `ThemeConfig.ets` | 阵营身份和颜色 | 可作为皮肤系统参考 |
| `ReportPanel.ets` | 战报 UI | 可参考，但新版建议主题化 |
| `RageBar.ets` | 我要打十个倒计时 | 可参考 |
| `AudioService.ets` | 音频加载和播放 | 服务层可继续复用 |

## 二、建议新包结构

后续如果新建代码包，建议不要覆盖 `games/rps`。可以暂定：

```text
entry/src/main/ets/games/rpsBattle/
```

建议拆分：

```text
RpsBattlePage.ets             页面入口，连接大厅、生命周期、状态机
RpsBattleTypes.ets            类型、枚举、常量
RpsBattleRules.ets            纯规则：克制、计数、胜负、赔率
RpsBattleEngine.ets           模拟主循环，不依赖 ArkUI
RpsBattleEvents.ets           事件调度：黑洞、叛徒、绝地、响指、道具
RpsBattleModes.ets            预设、模式配置、多局规则
RpsBattleTheme.ets            颜色、尺寸、动效 token
RpsBattleRenderer.ets         Canvas 或组件化渲染
RpsBattleHud.ets              顶部状态和人数走势
RpsBattleControls.ets         暂停、上帝之手、模式内按钮
RpsBattleOverlays.ets         倒计时、事件、结算、战报弹窗
RpsBattleReport.ets           战报模型和统计
```

如果想先做最小版本，至少拆成：

```text
RpsBattlePage.ets
RpsBattleTypes.ets
RpsBattleRules.ets
RpsBattleEngine.ets
RpsBattleRenderer.ets
```

## 三、推荐状态机

现有 `GamePhase` 可以保留概念，但新版建议拆细：

| 状态 | 含义 |
| --- | --- |
| `Setup` | 模式选择和配置 |
| `Countdown` | 开局倒计时和支持阵营选择 |
| `IntroEvent` | 强事件入场，例如我要打十个、响指 |
| `Playing` | 正常模拟 |
| `Paused` | 暂停 |
| `Resolving` | 胜负已确定，等待表现完成 |
| `Result` | 结算和战报 |

这样可以避免强事件覆盖层、胜利延迟和暂停逻辑互相抢状态。

## 四、规则层测试建议

新版规则层应能脱离 ArkUI 测试。建议覆盖：

- `beats(0,1)`、`beats(1,2)`、`beats(2,0)` 为真，其余为假。
- `leadingType()` 对并列返回 `-1`。
- 限时结束唯一领先才有赢家。
- 据点分数到 `100` 后 owner 获胜。
- 护盾抵挡一次转化。
- 下注赔率随人数变化符合预期范围。
- 多局胜方惩罚不低于 `3` 人。
- 黑洞吞噬不计为转化。
- 我要打十个期间碰撞结果为击杀优势方。

## 五、重构迁移顺序

### 阶段 1：规则内核

目标：

- 新建包。
- 搬迁或重写基础类型。
- 实现三阵营生成、追捕、躲避、碰撞转化。
- 实现限时、死斗、清场结算。

验收：

- 不接大厅也能通过本地页面预览或临时入口跑一局。
- 三方人数变化正确。
- 结算稳定。

### 阶段 2：新版观战体验

目标：

- 重做战场视觉。
- 重做 HUD。
- 设计更清楚的追击、转化、被追提示。
- 做倒计时、暂停、结算。

验收：

- 玩家不用读说明也能看懂三方关系。
- 小屏不遮挡战场。
- 事件提示不盖住核心单位。

### 阶段 3：核心模式

目标：

- 经典乱斗。
- 据点争夺。
- 翻盘事件模式。

验收：

- 每个模式都有明确目标和结算文案。
- 不把旧版全部开关一次性塞进首版配置页。

### 阶段 4：高光事件

目标：

- 我要打十个。
- 黑洞。
- 叛徒。
- 绝地求生。
- 能量道具。

验收：

- 每个事件都有触发提示、视觉表现、音效、战报记录。
- 事件可以开关，且组合后不破坏胜负判断。

### 阶段 5：战报和长期记录

目标：

- 结果页解释胜利原因。
- 战报记录关键事件、数量走势、转化王、逆转。
- 大厅成绩记录改为更合理的分数。

验收：

- 战报不是流水账，能指出本局关键转折。
- 结果页能回答“为什么赢”。

## 六、保留和舍弃建议

强烈保留：

- 三阵营循环克制。
- 自动追捕和转化。
- 据点争夺。
- 我要打十个。
- 战报。

建议保留但重做表现：

- 黑洞。
- 叛徒。
- 绝地求生。
- 道具。
- 上帝之手。

可以延后：

- 多主题身份。
- 下注积分。
- 灭霸响指。
- 全量自定义开关。

## 七、主要风险

- 事件太多会稀释核心玩法，新版首版要收敛。
- 随机事件如果不可预期又没有提示，会让玩家觉得结果不公平。
- 单位数量过多时 Canvas 和碰撞成本会上升，需要考虑空间分桶或限制单位数量。
- 如果继续使用旧版 `Math.random()`，回放和战报复盘无法做到确定。
- 大厅入口和旧版共存时，必须明确游戏 id，避免覆盖旧成绩。

## 八、命名建议

新版内部名可以选一个稳定 id：

| 候选 | 说明 |
| --- | --- |
| `rpsBattle` | 清楚表达新版大作战 |
| `rpsArena` | 偏竞技场 |
| `rpsRework` | 只适合作临时开发名 |

推荐用 `rpsBattle`，避免和旧 `rps` 混淆。
