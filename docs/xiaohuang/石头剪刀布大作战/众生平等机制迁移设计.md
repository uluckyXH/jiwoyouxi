# 石头剪刀布大作战众生平等机制迁移设计

日期：2026-07-24
状态：已实现，待真机验收
适用范围：新版 `entry/src/main/ets/games/rpsBattle/` 的 `equality` 模式

## 1. 目标与边界

“众生平等”是旧版石头剪刀布大作战的一个预设模式，不是泛泛的数值平衡玩法。它的身份由三部分共同组成：

1. 死斗清场：不按时长结算，直到只剩一个阵营。
2. 单次随机减员：战局中途让每个阵营随机失去一半存活单位，制造重新洗牌的机会。
3. 静态障碍：开局生成少量障碍，让追捕路径和碰撞关系出现变化。

新版要迁移这三个规则核心，但不能直接照搬旧版的触发时机、全屏素材和障碍密度。手机端画布更小、单局更短，若仍在开局 9 至 23 秒后才触发减员，很多对局已经清场，模式核心根本不会出现。

本设计只覆盖 `众生平等`。不把黑洞、缩圈、绝地求生、道具或团队占点机制混入该模式，也不修改其他模式的胜负规则。

## 2. 原版规则核对

原版来源：

- 预设：`entry/src/main/ets/model/Presets.ets`
- 运行时：`entry/src/main/ets/model/GameEngine.ets`
- 类型：`entry/src/main/ets/model/GameTypes.ets`

| 项目 | 原版行为 | 保留结论 |
| --- | --- | --- |
| 预设开关 | `deathmatch=true`、`thanos=true`、`obstacles=true` | 三项规则语义保留 |
| 胜负 | 不走时长结算，存活阵营仅剩一个时结束 | 保留，作为模式核心 |
| 减员次数 | 每局只会触发一次 | 保留 |
| 减员对象 | 分阵营独立处理当前存活单位 | 保留 |
| 减员数量 | 每个阵营随机移除 `floor(存活数 / 2)` | 保留 |
| 最低存活 | 一个阵营仅剩 1 个单位时不会被该事件移除 | 保留 |
| 原版时机 | 开局后随机 9 至 23 秒，提前 3 秒预警，展示后约 1.65 秒完成减员 | 改为手机短局节奏 |
| 响指表现 | 角色图片覆盖层和专用响指音效 | 不迁移角色图或原专用音频；改为原生 Canvas 反馈和通用事件音 |
| 障碍数量 | 小画布 3 个，宽画布 5 个 | 改为手机低密度 |
| 障碍效果 | 静态矩形，单位碰撞后反弹并衰减速度 | 改为更大的静态圆形石块，保留反弹并改善角落脱困 |

原版的减员是随机清场，不是转换阵营、复活、补兵或强制平衡人数。新版也不额外加入这些效果，避免把模式改成另一种玩法。

## 3. 设计结论

### 3.1 模式目标

```text
目标：死斗清场
核心：基础追捕、碰撞转阵、死斗清场
增强：响指减员、战场障碍
```

“死斗清场”决定了众生平等的胜负方式，因此属于模式核心，不在机制区提供关闭入口。

“响指减员”和“战场障碍”属于原版预设附带的增强机制。按照新版已经确立的机制分层，它们默认开启，但开局前允许玩家单独关闭：

| 机制 | 默认状态 | 是否可关闭 | 关闭后的结果 |
| --- | --- | --- | --- |
| 死斗清场 | 核心 | 否 | 不适用 |
| 响指减员 | 开 | 是 | 本局不预警、不清算、不绘制事件效果 |
| 战场障碍 | 开 | 是 | 本局不生成障碍、不做障碍碰撞、不绘制障碍 |

机制选择只在开局配置页生效。进入倒计时或对局后不允许切换，避免运行时状态和胜负预期发生变化。

### 3.2 表达与主题

旧版使用了特定影视角色名称、图片和专用音效。新版不使用角色名、肖像、图片、原专用音频或带有明显指向性的文案。

保留“响指减员”作为玩法说明，运行时统一称为“均衡清算”：

```text
配置项：响指减员
预警文案：均衡清算将在 1.8 秒后开始
结算文案：均衡清算完成
```

视觉表现使用已有游戏主题色和 Canvas 基础图形：

- 预警：细描边、短倒计时、低面积脉冲，不遮挡整块战场。
- 执行：战场中央一次短暂环形脉冲；各阵营附近出现短时 `-N` 数量提示。
- 障碍：低饱和中性色圆形石块，边缘使用当前模式强调色的细线，不与三阵营颜色混淆。
- 声音：预警复用已预加载的通用警示音，清算完成复用通用事件音；不加载或播放原版响指资源。
- 不加载全屏图片，不依赖额外网络或资源加载，也不做长时间覆盖层。

## 4. 手机端规则

### 4.1 均衡清算时序

原版首次触发可能晚于手机端整局时长。新版改为固定的早期随机窗口，确保玩家先看到基础追捕，再得到一次明确的翻盘事件。

```text
开局
  └─ 3.0 至 4.0 秒：进入预警
       └─ 1.8 秒预警倒计时
            └─ 4.8 至 5.8 秒：执行一次均衡清算
                 └─ 0.8 秒收束动画，状态进入已完成
```

规则：

1. 每局最多执行一次，不因暂停恢复、画布尺寸变化或剩余单位变少而重新排期。
2. 在预警或执行前，如果已经满足死斗清场条件，则直接结束对局，不强行触发事件。
3. 清算时按当前存活数对三个阵营分别计算 `floor(count / 2)`。
4. 同一阵营被移除的实体由可注入随机源均匀抽取；抽取后立刻复用引擎现有的阵营缓存和 swap-remove 删除路径。
5. 清算不转换阵营、不复活、不增兵、不改变支持阵营，也不修改已有道具或其他模式状态。

示例：

| 清算前阵营存活数 | 本次移除数 | 清算后存活数 |
| --- | --- | --- |
| 0 | 0 | 0 |
| 1 | 0 | 1 |
| 2 | 1 | 1 |
| 5 | 2 | 3 |
| 12 | 6 | 6 |

该规则与原版“每个阵营保留至少一半向上取整”的结果一致。

### 4.2 战场障碍

障碍必须制造路线选择，而不能在小屏上变成墙、出生点封锁或单位抖动来源。

#### 数量与尺寸

设 `shortSide = min(战场宽度, 战场高度)`：

| 画布短边 | 障碍数量 | 单个半径 |
| --- | --- | --- | --- |
| `<= 360` | 2 个 | `clamp(shortSide * 0.095..0.13, 28, 42)` |
| `> 360` | 3 个 | `clamp(shortSide * 0.095..0.13, 28, 50)` |

这里的数值使用 Canvas 实际画布坐标，不把设备像素密度写死为某个固定值。宽屏或平板后续可以单独做画布适配，不在本次手机模式里恢复旧版 5 个障碍。

#### 生成约束

每个障碍必须同时满足：

1. 与战场边界保持至少一个实体直径的间距。
2. 不落在任一阵营开局安全区。安全区半径取 `max(56, shortSide * 0.18)`。
3. 与已生成障碍保持足够间距，避免圆形相交或只留一条不可通行的窄缝。
4. 不覆盖战场中心的小范围可交战区域，避免开局把三方直接隔开。
5. 生成尝试次数有上限，例如每个障碍最多 24 次；超出后从预定义的安全候选点中取第一个可用位置。

因此，障碍生成必须是有界、可复现、可测试的纯规则，不能在渲染循环里持续随机重试。

#### 碰撞与反弹

单位和障碍均使用圆形碰撞体：

1. 使用“圆心距离小于两个半径之和”判断相交。
2. 相交后沿两个圆心的法线把单位完整推出障碍，避免单位停留在内部。
3. 将速度按法线反射并乘以 `0.82` 衰减系数；反弹后再应用边界约束。
4. 当圆心恰好重合时，按当前速度选择确定性法线，不能产生 `NaN`、零向量抖动或重复翻转。

障碍碰撞发生在单位移动和边界约束之后、阵营之间的空间桶碰撞之前。这样单位不会因为障碍内重叠而重复触发转阵。

## 5. UI 与战斗呈现

### 5.1 开局配置

众生平等模式的机制卡片从“后续”调整为：

```text
死斗清场       核心
响指减员       开 / 关
战场障碍       开 / 关
```

规则摘要同步更新：

```text
目标：死斗清场
机制：响指减员、战场障碍
```

关闭一个增强机制后，摘要只展示当前启用项；两个都关闭时展示：

```text
目标：死斗清场
机制：基础追捕、碰撞转阵
```

### 5.2 战斗 HUD

众生平等是死斗，不显示虚假的“剩余 60.0s”。HUD 主状态改为：

```text
死斗清场
```

均衡清算状态按阶段显示：

| 阶段 | HUD/Canvas 文案 | 表现 |
| --- | --- | --- |
| 等待 | `死斗清场` | 无额外提示 |
| 预警 | `均衡清算 1.8s` 至 `0.0s` | 顶部细条和局部脉冲 |
| 已执行 | `均衡清算完成`，短暂显示 | 0.8 秒后回到死斗状态 |

预警与动画均由 Canvas 绘制和引擎时间驱动。页面层只接收节流后的 HUD 快照，不在每一帧触发 ArkUI `@State` 重组。

### 5.3 结算

胜负判定沿用死斗清场：

- 只有一个阵营仍有单位：该阵营获胜，原因 `清场`。
- 无阵营存活：平局，原因 `清场`。
- 均衡清算本身不是结算原因，只记录为本局事件。

## 7. 性能预算与观测

当前新版已有阵营缓存、空间桶和性能日志。众生平等迁移应复用这些基础，不额外引入随单位数平方增长的路径。

| 项目 | 目标 |
| --- | --- |
| 清算更新 | 非触发帧为常数级状态判断；触发帧按当前实体数线性处理一次 |
| 单位与障碍检查 | 最多 3 个障碍，`实体数 × 障碍数` 的小常数检查 |
| 内存 | 障碍开局一次性生成；清算候选与结果复用短生命周期数组，不在每帧分配 |
| ArkUI | 不因预警或障碍碰撞每帧写入页面 `@State` |
| 渲染 | 障碍和脉冲直接走现有 Canvas 绘制路径 |

新增 `RpsBattlePerf` 观测字段建议：

```text
snapMs
obstacleMs
snapRemoved/frame
obstacleHits/frame
```

关闭相应机制时，相关状态更新、碰撞和 Canvas 绘制必须完全跳过；日志中对应耗时应接近 `0.00ms`。

### 8.3 真机验收

在小屏和常见手机宽度下分别验证小、中、大阵容：

1. 默认开局时，约第 3 至 4 秒出现清晰预警，约第 5 至 6 秒只执行一次清算。
2. 每个阵营的减员数符合 `floor(count / 2)`，单个单位不会被清算事件直接清除。
3. 关闭响指减员后，整局没有预警、清算文案和清算帧开销。

5. 开启障碍后，出生点、中心交战区和边缘没有被封死；单位不会卡在障碍内部或持续高频抖动。
6. 暂停、恢复、重新开始、返回大厅不会重复触发清算或遗留障碍状态。
7. HUD 始终显示死斗状态，不显示与该模式冲突的剩余时间。
8. 观察 `RpsBattlePerf`：默认大局不因清算或障碍导致持续性掉帧，清算仅出现单次短峰值。

## 9. 实施顺序

机制按可独立验证的顺序迁移，避免一次同时改动引擎、渲染和多个规则：

### P1：均衡清算

1. 扩展模式规则、机制开关和快照类型。
2. 实现 `RpsBattleSnapSystem.ets` 与 Hypium 测试。
3. 接入引擎缓存删除路径、HUD 和 Canvas 预警/收束效果。
4. 完成构建、真机日志和测试报告。

### P2：战场障碍

1. 实现 `RpsBattleObstacleSystem.ets` 与 Hypium 测试。
2. 接入安全生成、移动碰撞、Canvas 绘制和性能指标。
3. 验收小屏出生点、圆形反弹和持续帧率。

### P3：组合平衡

1. 在小、中、大阵容下比较“仅清算”“仅障碍”“两个都开”的对局时长与结束方式。
2. 只调整清算触发窗口、障碍数量和尺寸上限，不修改原版减半公式。
3. 根据真机测试补充测试报告和本设计的最终参数记录。

## 10. 暂不纳入

- 不接入原版角色图片、名称、配音或其他受版权约束的素材。
- 不增加第二次清算、持续减员、全屏吸入或全局减速。
- 不为清算加入复活、补兵、护盾或阵营强制平衡。
- 不恢复旧版宽画布的 5 个障碍配置。
- 不改变经典乱斗、团队占点和小心叛徒的机制包。

这样能在保留原版众生平等玩法辨识度的前提下，适应手机端短局、保持 Canvas 性能边界，并为后续逐项实现与真机调参留下清晰入口。
