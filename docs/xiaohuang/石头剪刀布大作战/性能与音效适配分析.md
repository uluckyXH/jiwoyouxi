# 石头剪刀布大作战性能与音效适配分析

日期：2026-07-09
状态：新版 `rpsBattle` 基础玩法第二轮性能调研

## 一、当前结论

当前卡顿已经不是最早那条“每帧把完整实体数组写入 ArkUI 状态”的问题为主了。首轮止血后，运行中实体不再每帧 clone 到 `@State`，HUD 也已经降频同步，Canvas 绘制由页面循环直接调用。

现在剩余卡顿更可能来自两类成本：

- 引擎每帧算法仍然偏重：最近目标搜索和碰撞检测都是平方级扫描。
- Canvas 每帧绘制仍然偏重：静态战场背景、网格、外框和所有单位符号都在同一帧里完整重画。

这类小型实时游戏后续要继续做鸿蒙原生优化，优先方向不是盲目提高帧率，也不是立刻把整套游戏循环丢到后台线程，而是先把主线程每帧工作量压下来，再评估更贴近系统渲染节奏的帧回调方案。

## 二、当前实现状态

当前新版相关文件：

- `RpsBattlePage.ets`：页面状态机、主循环、Canvas 容器、HUD 状态同步、音效生命周期。
- `RpsBattleEngine.ets`：单位移动、目标搜索、碰撞、转化、胜负判断。
- `RpsBattleRenderer.ets`：Canvas 战场绘制。
- `RpsBattleHud.ets`：人数、倒计时、支持阵营、转化数。
- `RpsBattleSetupPanel.ets`：阵营和规模选择。
- `RpsBattleOverlay.ets`：暂停和结果层。

当前主循环已在 2026-07-09 从 `setInterval(33ms)` 切到 `@kit.ArkGraphics2D` 的 `displaySync`，目标约 60 FPS。每次循环主要做：

1. 计算 `dt` 和累计时长。
2. `engine.update()` 推进模拟。
3. `renderer.draw()` 绘制当前实体引用。
4. 每 `250ms` 左右同步一次 HUD。
5. 根据转化和结算触发已节流的音效。

当前已经完成的止血项：

- 高频实体状态不再每帧写入 ArkUI `@State`。
- 运行中不再每帧 `snapshot()` clone 完整实体数组。
- HUD 降频刷新。
- BGM、转化音效、结算音效已经接入，并对转化音效做了节流。

## 三、官方资料调研

本轮只参考华为 HarmonyOS 官方资料，调研时间为 2026-07-09。

### 3.1 ArkTS 高性能编程

官方文档：[ArkTS高性能编程实践](https://developer.huawei.com/consumer/cn/doc/harmonyos-guides/arkts-high-performance-programming)

与本游戏直接相关的点：

- 性能敏感循环中应减少重复属性访问，把循环内不会变化的值提到循环外。
- 纯数值计算场景可评估 `TypedArray`，尤其是实体位置、速度、半径等数据量增加后。
- 避免稀疏数组、混合类型数组和频繁异常路径。
- 避免在热路径里频繁创建临时对象，减少 GC 压力。

对 `rpsBattle` 的落地含义：

- `applyMovement()`、`resolveCollisions()`、`nearestEntity()` 是热路径，循环内要少做对象创建和重复函数调用。
- 当前 `normalized()` 每次返回临时向量对象，后续可改成直接用局部数值计算或复用结构。
- 如果后续单位规模扩大，可把实体数据从对象数组逐步改成结构化数值数组，例如 `x[]`、`y[]`、`vx[]`、`vy[]`、`faction[]`。

### 3.2 前台帧率与帧率体验

官方文档：

- [前台帧率](https://developer.huawei.com/consumer/cn/doc/harmonyos-guides/standard-foreground-frame-rate)
- [帧率](https://developer.huawei.com/consumer/cn/doc/harmonyos-guides/performance-frame-rate)

官方按场景区分帧率建议：变化区域大的持续型动效建议按 60Hz 目标理解，小区域微动效可低到 30Hz 及以下。`rpsBattle` 的战场是持续运动场景，标准目标应是稳定优先。

对 `rpsBattle` 的落地含义：

- 现在不要为了“看起来更顺”先把 timer 从 33ms 改到 16ms。60 FPS 会让每秒 update/draw 次数翻倍，如果算法和绘制没有先降成本，卡顿会更明显。
- 当前阶段可以继续以 30 FPS 做玩法验证，但每帧 update + draw 要稳定落在预算内。
- 等 P0/P1 优化后，再评估 45/60 FPS 或动态帧率，而不是现在直接加速循环。

### 3.3 前台绘制渲染

官方文档：[前台绘制渲染](https://developer.huawei.com/consumer/cn/doc/harmonyos-guides/standard-foreground-render)

与本游戏直接相关的点：

- 布局层级要扁平，避免冗余嵌套。
- 精准控制组件更新范围。
- 重复利用的渲染内容应缓存复用。
- 动效不可见时应停止。
- 降低复杂视效计算，脏区尽量只包含变化内容。

对 `rpsBattle` 的落地含义：

- 战场静态背景、网格、边框不应和单位运动绑定在同一个全量绘制成本里。
- 后续接素材图时，必须预加载、复用，不允许在帧循环中加载或构造资源。
- 大局模式要有视觉降级开关，比如关闭单位阴影、降低脉冲圈复杂度、减少网格密度。
- 暂停、结算、返回大厅时必须停止循环，避免不可见动效继续占资源。

### 3.4 线程模型、TaskPool 与 Worker

官方文档：

- [线程模型](https://developer.huawei.com/consumer/cn/doc/harmonyos-guides/thread-model-stage)
- [TaskPool简介](https://developer.huawei.com/consumer/cn/doc/harmonyos-guides/taskpool-introduction)
- [Worker简介](https://developer.huawei.com/consumer/cn/doc/harmonyos-guides/worker-introduction)
- [TaskPool和Worker的对比](https://developer.huawei.com/consumer/cn/doc/harmonyos-guides/taskpool-vs-worker)

官方线程模型里，主线程负责 UI 绘制、交互事件、生命周期回调，也接收 TaskPool/Worker 消息。TaskPool 和 Worker 都适合处理耗时或密集任务，但工作线程不能直接操作 UI，并且线程间通信涉及序列化、转移或共享机制。

对 `rpsBattle` 的落地含义：

- 不建议现在把每帧模拟整体搬到 Worker。当前实体量不大，跨线程传输每帧实体状态可能抵消收益，还会让音效、HUD、暂停恢复、结算同步更复杂。
- TaskPool 更适合相对独立、可取消、可调度的任务。对本游戏可用于战报统计、录像摘要、离线 AI 策略计算等，不适合先承接 30 FPS 主循环。
- Worker 更适合长时间占用线程且需要开发者管理生命周期的任务。只有当后续实体规模大幅增加，并且能用 `ArrayBuffer`/`SharedArrayBuffer` 等方式稳定降低传输成本时，才值得做独立模拟线程验证。

### 3.5 UIContext FrameCallback

官方文档：

- [Class (FrameCallback)](https://developer.huawei.com/consumer/cn/doc/harmonyos-references/arkts-apis-uicontext-framecallback)
- [Class (UIContext)](https://developer.huawei.com/consumer/cn/doc/harmonyos-references/arkts-apis-uicontext-uicontext#postframecallback12)

`FrameCallback` 用于设置下一帧渲染时执行的任务，需配合 `UIContext.postFrameCallback` 或 `postDelayedFrameCallback` 使用，API 12 起支持，Stage 模型可用。

对 `rpsBattle` 的落地含义：

- 这是替代裸 `setInterval(33ms)` 的候选方案，理论上更贴近 UI 渲染节奏。
- 需要先做一个小分支验证：用 `FrameCallback.onFrame(frameTimeInNano)` 驱动游戏 tick，并在仍处于 `playing` 时重新 post 下一帧。
- 验证重点不是“API 能不能跑”，而是：生命周期能否干净停止、暂停/恢复是否稳定、预览器和真机行为是否一致、是否比 timer 更少抖动。

## 四、剩余卡顿的核心原因

### 4.1 目标搜索仍是 O(n²)

当前每个单位每帧都会分别找最近猎物和最近天敌。每次查找都扫描全体实体。

标准局 `8/8/8` 是 24 个单位，大致每帧接近 `2 * 24 * 24` 次目标检查；大局 `12/12/12` 是 36 个单位，大致每帧接近 `2 * 36 * 36` 次目标检查。后续人数、事件、特效、素材一加，余量会继续变少。

### 4.2 碰撞检测仍是 O(n²)

`resolveCollisions()` 每帧检查所有实体对。36 个实体时还不算灾难，但它和目标搜索叠在一起，形成稳定的每帧 CPU 压力。

### 4.3 统计和结算仍有重复扫描

`resultAt()` 内部会统计人数，HUD 同步时也会取 counts。虽然 HUD 已降频，但转化后人数完全可以增量维护，避免多处全量扫描。

### 4.4 Canvas 每帧重画静态内容

`RpsBattleRenderer.draw()` 每帧都会重画背景、顶光、底部阴影、网格、外框和实体。背景类内容不随实体变化，应该被缓存或至少被拆分。

### 4.5 单位符号绘制成本偏高

每个单位用 `fillText()` 绘制手势符号。文字绘制比简单图形更重，后续如果接素材，也需要从“每帧构造/文字绘制”转成“预加载后复用绘制”。

## 五、优化路线

### P0：先加性能计数

目标：确认卡顿到底落在 update、draw，还是 timer 间隔抖动。

当前状态：2026-07-09 已在 `RpsBattlePage.ets` 和 `RpsBattleEngine.ets` 落地开发期聚合日志，日志开关为 `RPS_BATTLE_ENABLE_PERF_LOG`。

建议记录：

- `frameIntervalMs`：实际帧间隔。
- `updateMs`：引擎更新耗时。
- `drawMs`：Canvas 绘制耗时。
- `targetScanCount`：目标搜索检查次数。
- `collisionCheckCount`：碰撞检查次数。
- `entityCount`：实体总数。
- `conversionCount`：累计转化数。

日志建议每秒输出一次，开发期开启，默认不要污染正式体验。

当前日志示例：

```text
RpsBattlePerf scale=large entities=36 frames=31 fps=30.2 interval=33.1ms maxInterval=42ms update=1.35ms draw=5.10ms target/frame=2592 collision/frame=630 conversions/frame=0.16 totalConversions=12
```

### P1：阵营缓存和增量计数

目标：用最低风险先减少重复扫描。

建议改动：

- 引擎维护三个阵营数组或索引缓存：`rockList`、`paperList`、`scissorsList`。
- 转化时把实体从原阵营缓存移到新阵营缓存。
- 人数在转化时增量更新，不再频繁 `countEntities()`。
- 胜负判断直接看缓存数量。

收益：

- 最近目标搜索不用先扫描无关阵营。
- HUD 和结算减少重复全量统计。
- 对玩法规则影响小，适合先做。

### P2：空间分桶优化碰撞

目标：把碰撞检测从“全体两两检查”改成“同格和邻格检查”。

建议方案：

- 使用均匀网格，格子尺寸可先取 `ENTITY_RADIUS * 2.5` 到 `ENTITY_RADIUS * 3.5`。
- 每帧先清空桶，再按实体坐标入桶。
- 碰撞只检查当前格和相邻 8 格。
- 保留原有转化规则，保证结果行为不变。

收益：

- 单位数量上来后，碰撞检查次数会明显下降。
- 大局模式和后续更高人数模式会有更稳定的 CPU 余量。

### P3：目标搜索也使用空间信息

目标：减少最近猎物和最近天敌的全图扫描。

建议方案：

- 先基于阵营缓存做一版。
- 如果 P1 后仍卡，再用空间网格从近到远搜索目标。
- 设置搜索上限半径，没找到时保留巡游行为。

注意：

- 搜索上限会改变单位追逐倾向，需要试玩确认手感。
- 这一步比 P1/P2 更可能改变战场节奏，建议放在碰撞分桶之后。

### P4：渲染分层和静态缓存

目标：降低 draw 成本，为后续素材图留空间。

建议方案：

- 把战场背景、网格、边框视为静态层，只在尺寸变化或主题变化时重绘。
- 动态层只绘制单位、转化脉冲和必要提示。
- 在接素材前建立资源预加载策略，单位图标按阵营复用。
- 大局模式降低视觉细节，优先保证玩法顺。

如果当前 Canvas 组件不方便做真正离屏缓存，至少可以先减少静态绘制复杂度：降低网格密度、减少半透明叠层、减少每帧路径数量。

### P5：验证显示同步替代 timer

目标：降低 `setInterval` 和系统渲染节奏不一致带来的抖动。

当前状态：2026-07-09 已实现，待真机复测。

已落地方案：

- 使用 `displaySync.create()` 建立显示同步驱动。
- 用 `setExpectedFrameRateRange({ min: 60, max: 60, expected: 60 })` 请求 60Hz 调度。
- 使用 `displaySync.IntervalInfo.timestamp` 计算 `dt`。
- 暂停、结算、返回大厅时 `stop()`。
- 页面销毁时 `off('frame')` 解绑回调。
- 性能日志新增 `loop=displaySync targetFps=60`。

复测判断：

- 如果 `fps` 接近 60、`interval` 接近 16 到 17ms，说明帧节奏已经切到显示同步。
- 如果 `fps` 仍在 30 左右，说明系统实际调度没有按 60Hz 跑，需要结合设备刷新率和前台帧率策略继续查。
- 如果 `maxInterval` 偶发很高，同时日志附近有大量 SoundPool 输出，要继续看音效触发是否造成主线程抖动。
- 如果 `update + draw` 在大局下接近 16ms，说明 60Hz 预算吃紧，下一步应做动态帧率或渲染降级。

这一步是在 P1/P2 后执行的。它不改变玩法，只改变帧驱动和 `dt` 来源。

### P6：谨慎评估 TaskPool / Worker

目标：只在数据证明主线程模拟成为瓶颈后再做。

可行方向：

- TaskPool：战报统计、结算分析、离线策略计算。
- Worker：未来大规模单位模拟，且实体数据已经能用紧凑数值数组传递或共享。

暂不建议：

- 每帧把完整对象数组发给 Worker。
- Worker 内直接参与 UI 或音效控制。
- 在没有性能计数前做线程化重构。

## 六、推荐实施顺序

### PR1：性能计数与日志

- 加开发期开关。
- 记录 update、draw、实际帧间隔、扫描次数。
- 标准局和大局各试玩 60 秒。

验收：

- 能明确判断当前瓶颈是 update 还是 draw。
- 日志不会影响正式玩法。

### PR2：阵营缓存和增量计数

- 引擎维护阵营缓存。
- 转化时同步更新缓存和 counts。
- HUD/结算不再重复全量统计。

验收：

- 转化规则不变。
- 清场和限时结算不变。
- 大局 update 平均耗时下降。

### PR3：碰撞空间分桶

- 用均匀网格减少碰撞检查。
- 记录优化前后 `collisionCheckCount`。

验收：

- 大局碰撞检查次数明显下降。
- 单位不会穿边界或异常重叠。
- 玩法结果仍符合石头剪刀布克制规则。

### PR4：渲染降本

- 拆静态/动态绘制策略。
- 降低大局视觉复杂度。
- 为后续素材预加载留接口。

验收：

- `drawMs` 下降。
- 暂停、结算、返回大厅后没有持续绘制。
- 后续接素材不会引入帧内加载。

### PR5：显示同步小分支验证

- 用官方 `displaySync` 做帧驱动实验。
- 和 timer 方案用相同计数对比。
- 只在真机表现更稳时继续保留。

验收：

- 暂停/恢复/结算/退出生命周期干净。
- 帧间隔抖动降低。
- 没有回调泄漏。

## 七、不建议立刻做

- 不建议在未完成引擎降本前直接把 FPS 提到 60。当前已先做 P1/P2，再用 `displaySync` 验证 60Hz 体感。
- 不建议先大规模接素材。渲染分层和预加载策略没打稳前，素材会放大卡顿。
- 不建议先做完整多线程模拟。当前更像算法和绘制成本问题，线程化会引入通信、生命周期和同步复杂度。
- 不建议为了减少卡顿删除核心碰撞或转化逻辑。优化必须保证游戏仍然正常进行。

## 八、验收指标建议

开发期建议关注：

- 标准局 `8/8/8`：移动连续，无明显顿挫。
- 大局 `12/12/12`：允许画面忙，但不能持续卡顿。
- 暂停/继续：没有循环泄漏，没有音乐叠播。
- 返回大厅：循环停止，音频释放或暂停。
- 结算：只结算一次，只播放一次结算音。

建议目标：

| 指标 | 标准局目标 | 大局目标 |
| --- | --- | --- |
| update 平均耗时 | `< 4ms` | `< 8ms` |
| draw 平均耗时 | `< 6ms` | `< 10ms` |
| HUD 更新频率 | `<= 4Hz` | `<= 4Hz` |
| ArkUI 高频状态写入 | 尽量为 0 | 尽量为 0 |
| 实际帧间隔 | 接近目标帧间隔 | 抖动可控 |

这些指标是开发期目标，不是最终发布承诺。真机型号差异较大，最终以真机试玩、性能日志和 DevEco 性能工具结果为准。
