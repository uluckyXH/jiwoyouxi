# 石头剪刀布大作战玩法方案卡片与战役入口 UI 设计

日期：2026-07-15
状态：UI 已实现，机制分层持续迁移
适用范围：新版 `entry/src/main/ets/games/rpsBattle/`

## 结论

下一阶段不要把原版“游戏机制”和“精彩事件”的十几个开关直接铺到新版选择页。

新版选择页当前已经承载了：

- 阵营主题。
- 支持阵营。
- 初始阵容。
- 高级人数调整。
- 对局时长。
- 开始乱斗。

如果继续平铺 `上帝之手 / 障碍 / 缩圈 / 据点 / 黑洞 / 叛徒 / 绝地 / 响指 / 道具 / 团队道具 / 我要打十个`，选择页会回到原版配置面板的问题：信息密度高、主路径不清楚、玩家不知道应该先选什么。

建议先增加一个“玩法方案”区域，用 2x2 卡片承接原版预设。玩家先选“这局想玩什么”，后续再逐个实现对应玩法。

## 当前 UI 判断

当前新版 UI 的优点：

- 主路径已经清楚：主题 -> 支持方 -> 阵容 -> 时长 -> 开始。
- 人数高级调整默认收起，避免普通开局变重。
- 主题和阵营表现已经统一到当前暖色纸张风格。

当前风险：

- 设置页已经接近上限，继续加复杂配置会让一屏内容过长。
- 玩法机制会影响胜利目标、HUD、Canvas 表现和结算，不能只作为普通开关插在最后。
- 据点、叛徒、响指这类机制需要玩家先知道“这一局的玩法核心”，否则战斗中突然发生会显得乱。

因此玩法入口应该是“方案卡片”，不是“开关列表”。

## UI 放置位置

选择界面推荐顺序：

```text
阵营主题
支持阵营
玩法方案
初始阵容
对局时长
开始乱斗
```

放在 `支持阵营` 后面的原因：

1. 主题和支持阵营是身份选择，优先级最高。
2. 玩法方案决定胜利目标和事件组合，应该早于人数和时长。
3. 人数和时长是局部调参，应该跟随玩法方案。

## 玩法方案卡片

区域标题：

```text
玩法方案
选择本局规则核心
```

布局：

- 2x2 网格。
- 每张卡固定高度，建议 `70-78vp`。
- 卡片圆角保持 `8vp`，和当前选项块一致。
- 不用大段说明，不做展开式大卡。
- 选中态使用当前主按钮深色底或主题强调色。
- 未选中态使用浅纸色底和细边框。

卡片内容三层：

1. 顶部：短图标或符号。
2. 中间：方案名。
3. 底部：2 到 3 个关键词。

建议文案：

| key | 名称 | 卡片关键词 | 原版来源 |
| --- | --- | --- | --- |
| `classic` | 经典乱斗 | 基础追捕 · 清场 | `classic` |
| `zones` | 团队占点 | 据点 · 团队道具 | `zones` |
| `traitor` | 小心叛徒 | 缩圈 · 叛徒 | `traitor` |
| `equality` | 众生平等 | 响指 · 障碍 | `equality` |

图标建议先用文本符号，不接素材：

| key | 图标 |
| --- | --- |
| `classic` | `⚔` |
| `zones` | `◎` |
| `traitor` | `!` |
| `equality` | `=` |

说明：

- `traitor` 不建议用过多剧情化文案，卡片先保持短。
- `equality` 原版叫“众生平等”，机制是响指和障碍，卡片要让玩家知道它是强事件局。
- 图标后续接素材图时可替换，不影响布局。

## 选中后的规则摘要

玩法卡片下面显示一条轻量摘要，不再给每张卡展开大量文本。

结构：

```text
目标：限时领先 / 清场 / 占点到100
机制：据点、团队道具、绝地求生
```

每个方案摘要：

### 经典乱斗

```text
目标：限时领先或清场
机制：基础追捕、碰撞转阵
```

当前新版已经具备，可作为默认。

### 团队占点

```text
目标：占点到100或限时领先
机制：中心据点、团队道具、绝地求生
```

原版开启：

- `controlZones`
- `blackHole`
- `lastStand`
- `powerUps`
- `teamPowerUps`

新版第一阶段建议先做：

- `controlZones`
- HUD 据点进度。
- Canvas 中心据点圈。

此处是第一阶段计划；当前 `blackHole`、`lastStand`、`powerUps`、`teamPowerUps` 已分阶段接入。

### 小心叛徒

```text
目标：死斗清场
机制：缩圈、叛徒、黑洞
```

原版开启：

- `shrink`
- `blackHole`
- `traitor`
- `deathmatch`

新版第一阶段建议先做 UI 入口，不立刻实现全部机制。真正落地时先做 `traitor`，再做 `shrink`，最后做 `blackHole`。

### 众生平等

```text
目标：死斗清场
机制：响指减员、障碍
```

原版开启：

- `thanos`
- `obstacles`
- `deathmatch`

新版第一阶段建议改造表达，不直接绑定有版权风险的素材。机制可命名为“平衡响指”或后续替换为鸡窝主题事件，但卡片名可以先沿用原版“众生平等”。

## 与现有配置的关系

### 阵营主题

玩法方案不改变主题。主题只影响展示：

- 阵营名。
- 图标。
- 颜色。
- HUD 和结算文本。

### 支持阵营

玩法方案不改变支持阵营。支持阵营仍用于玩家代入和结算反馈。

### 初始阵容

玩法方案可以给默认建议，但不强制覆盖用户已调整的人数。

第一版规则：

- 切换玩法方案不重置人数。
- 切换玩法方案不重置人数模式。
- 如果后续某玩法需要强制人数范围，再单独加提示。

### 对局时长

第一版规则：

- 经典乱斗默认 `60s`。
- 团队占点默认仍 `60s`。
- 小心叛徒、众生平等后续可能使用死斗，但 UI 第一阶段先不自动切死斗。

原因：

- 当前新版还没有死斗 UI。
- 自动改时长容易让玩家困惑。
- 死斗应作为后续“赛制”能力单独接入。

## 模式机制包和高级规则

每个玩法模式都应该有自己的“默认机制包”。玩家选择模式时，系统先按该模式加载默认机制；高级玩家可以在默认收起的规则区里微调。

这比把所有机制平铺出来更合理：

- 普通玩家只需要选择模式，不需要理解十几个开关。
- 每个模式都有清楚的设计意图，机制组合不会显得随机。
- 高级玩家仍然可以关闭某个不喜欢的机制。

## 当前四模式机制分层口径

2026-07-21 讨论后，先按“每个模式有自己的机制包”继续迁移，不把所有机制做成全局自由组合。后续如果要做自由组合，应单独设计 `自定义乱斗` 或高级沙盒模式，不混入四个原版模式的基础 UI。

机制分三层：

| 层级 | 说明 | 是否可关闭 | 示例 |
| --- | --- | --- | --- |
| 模式核心机制 | 决定这个模式为什么成立，关闭后模式目标会失效 | 不可关闭 | 团队占点的中心据点 |
| 基础玩法核心 | 石头剪刀布大作战的底层循环克制，不属于增强机制 | 不可关闭 | 基础追捕、碰撞转阵 |
| 增强机制 | 原版预设默认带的玩法事件或附加系统 | 已实现后默认开，可关闭 | 黑洞、绝地求生、能量道具 |

因此“经典乱斗不是没有机制”，而是没有可选增强机制。它的基础追捕和碰撞转阵属于底层核心，不在 UI 上作为可关闭开关呈现。

| 模式 | 胜负目标 | 核心机制 | 增强机制 | 当前新版状态 |
| --- | --- | --- | --- | --- |
| 经典乱斗 | 限时领先或清场 | 基础追捕、碰撞转阵 | 无 | 已实现；不显示额外机制卡 |
| 团队占点 | 占点到 100 或限时领先 | 中心据点 | 黑洞、绝地求生、能量道具、团队道具 | 中心据点已实现且锁定；黑洞、绝地求生、能量道具、团队道具已实现且可关 |
| 小心叛徒 | 死斗清场 | 死斗清场 | 缩圈、叛徒、黑洞 | 当前只展示入口和后续机制；未进入运行时 |
| 众生平等 | 死斗清场 | 死斗清场 | 响指减员、战场障碍 | 已实现；增强机制默认开启且可单独关闭 |

### 原版预设对照

| 原版模式 | 原版开启项 | 新版迁移说明 |
| --- | --- | --- |
| `classic` | `deathmatch` | 新版已实现基础追捕、碰撞转阵和限时/清场结算；无额外增强机制 |
| `zones` | `controlZones`、`blackHole`、`lastStand`、`powerUps`、`teamPowerUps` | 新版已实现中心据点、黑洞、绝地求生、能量道具和团队道具；增强机制默认开且可关闭 |
| `traitor` | `shrink`、`blackHole`、`traitor`、`deathmatch` | 新版先保留模式入口和机制占位；真正实现时建议先做叛徒，再做缩圈，最后接入该模式下的黑洞 |
| `equality` | `thanos`、`obstacles`、`deathmatch` | 新版已实现死斗、一次均衡清算和低密度障碍；不使用原版角色素材，增强机制默认开启且可关闭 |

### UI 呈现规则

- `经典乱斗` 不显示“本局机制”卡，避免玩家误以为基础追捕和碰撞转阵可以关闭。
- `团队占点` 显示“本局机制”卡，因为中心据点改变胜负目标，必须明确告诉玩家；黑洞和绝地求生作为已实现增强机制显示开关。
- 尚未迁移到运行时的机制只展示为“后续”，不允许玩家点击开启。
- 已实现增强机制默认跟随模式开启，但只要不是核心机制，就允许玩家开局前关闭。
- 战斗中不提供机制开关，避免运行时状态不一致。
- 后续真机测试时可以先验证默认组合，再验证自定义组合。

### 默认机制包

| 模式 | 默认机制 | 说明 |
| --- | --- | --- |
| `classic` | 基础追捕、碰撞转阵 | 当前基础玩法，不额外加随机事件 |
| `zones` | 中心据点、黑洞、绝地求生、能量道具、团队道具 | 对齐原版团队占点预设 |
| `traitor` | 缩圈、黑洞、叛徒、死斗 | 对齐原版小心叛徒预设 |
| `equality` | 平衡响指、障碍、死斗 | 对齐原版众生平等预设，但响指表现可换成鸡窝主题 |

注意：默认机制包表达的是“最终对齐原版时的目标状态”。当前未实现的其他模式机制不能被玩家开启运行，只能显示为“后续”或禁用态。

### 选择模式后的展示

选择某个玩法后，摘要区从单行机制文本升级为轻量机制列表：

```text
目标：占点到100或限时领先
已启用：中心据点 / 黑洞 / 绝地求生 / 能量道具 / 团队道具
可关闭：黑洞 / 绝地求生 / 能量道具 / 团队道具
```

等某个后续机制真正实现后，再从“后续”移动到“已启用”。如果该机制属于当前模式默认机制包，则默认选中。

### 高级规则入口

不建议把所有机制一次性开放为可点击开关。已实现且验证过的增强机制可以先在玩法方案摘要下方提供轻量开关；未实现机制仍只展示为 `后续`：

```text
高级规则
障碍 / 缩圈 / 据点 / 黑洞 / 叛徒 / 绝地 / 道具
```

展开后只展示当前模式相关机制，不展示全量机制。例如选择 `团队占点` 后：

```text
团队占点机制
[锁] 中心据点   核心
[开关] 黑洞     已接入
[开关] 绝地求生 已接入
[x] 能量道具    可关闭
[x] 团队道具    可关闭，依赖能量道具
```

实现完成后的目标态：

```text
团队占点机制
[x] 中心据点
[x] 黑洞
[x] 绝地求生
[x] 能量道具
[x] 团队道具
```

玩家可以取消某个已实现机制，例如关闭黑洞，只玩“据点 + 团队道具”。这合理，但必须满足两个条件：

- 关闭后不会破坏模式核心目标。例如 `团队占点` 的 `中心据点` 是核心机制，不建议允许关闭；否则就不再是团队占点。
- 该组合已经完成规则和性能验证。未验证组合先不要开放。

### 开关分级

机制开关分三类：

| 类型 | 示例 | UI 行为 |
| --- | --- | --- |
| 核心机制 | 团队占点的中心据点、经典乱斗的碰撞转阵 | 默认开启，不允许关闭 |
| 默认机制 | 团队占点的黑洞、绝地、团队道具 | 实现后默认开启，已完成验证的机制允许关闭 |
| 后续机制 | 尚未迁移的其他模式机制 | 展示为后续或禁用，不可点击 |

当前阶段不要把所有高级规则都做成可点击开关，原因：

- 当前优先是保证模式卡片、中心据点和已接入增强机制稳定。
- 道具底层还没有逐项迁移，不能给玩家开启入口。
- 黑洞和绝地求生已经接入运行时，可以作为已实现增强机制提供开局前开关。
- 提前开放未实现机制会制造“点了但没效果”的风险。

## 数据结构建议

新增类型：

```ts
export type RpsBattleModeKey = 'classic' | 'zones' | 'traitor' | 'equality';

export interface RpsBattleModePreset {
  key: RpsBattleModeKey;
  name: string;
  icon: string;
  subtitle: string;
  objective: string;
  mechanics: string[];
  enabledMechanics: string[];
  upcomingMechanics: string[];
}
```

后续高级规则需要再补机制定义：

```ts
export type RpsBattleMechanicKey =
  'controlZone' | 'blackHole' | 'lastStand' | 'powerUps' | 'teamPowerUps' |
  'shrink' | 'traitor' | 'snap' | 'obstacles';

export type RpsBattleMechanicState = 'core' | 'enabled' | 'upcoming';

export interface RpsBattleMechanicOption {
  key: RpsBattleMechanicKey;
  label: string;
  summary: string;
  state: RpsBattleMechanicState;
  defaultEnabled: boolean;
  canDisable: boolean;
}
```

含义：

- `core`：模式核心机制，默认开启，不允许关闭。
- `enabled`：已实现的默认机制，默认开启；是否允许关闭由 `canDisable` 控制。
- `upcoming`：后续机制，展示但不可点击。

`团队占点` 第一阶段示例：

```ts
[
  { key: 'controlZone', label: '中心据点', state: 'core', defaultEnabled: true, canDisable: false },
  { key: 'blackHole', label: '黑洞', state: 'enabled', defaultEnabled: true, canDisable: true },
  { key: 'lastStand', label: '绝地求生', state: 'enabled', defaultEnabled: true, canDisable: true },
  { key: 'powerUps', label: '能量道具', state: 'enabled', defaultEnabled: true, canDisable: true },
  { key: 'teamPowerUps', label: '团队道具', state: 'enabled', defaultEnabled: true, canDisable: true }
]
```

新增配置文件：

```text
entry/src/main/ets/games/rpsBattle/RpsBattleModeConfig.ets
```

原因：

- 不直接依赖旧版 `model/Presets.ets`。
- 新版可以逐步迁移机制，不需要一次接入全部旧版字段。
- UI 卡片和规则摘要可以先稳定下来。

页面状态：

```ts
@State selectedModeKey: RpsBattleModeKey = 'classic';
```

设置面板入参：

```ts
@Prop selectedModeKey: RpsBattleModeKey = 'classic';
selectMode: (modeKey: RpsBattleModeKey) => void = () => {};
```

## 视觉细节

玩法方案卡片应延续当前选择页风格：

- 浅纸色底。
- 细边框。
- 8vp 圆角。
- 选中态深色底。
- 标题 13 到 14 号加粗。
- 关键词 10 到 11 号，不超过一行。

不要做：

- 大英雄图。
- 大面积说明文字。
- 卡片里再嵌卡片。
- 把所有机制开关直接放到卡片里。

## 实施顺序

### P1：UI 卡片落地

目标：

- 增加 `玩法方案` 2x2 卡片。
- 默认选中 `经典乱斗`。
- 卡片切换后更新规则摘要。
- 不改变战斗底层规则。

验收：

- 四张卡可选中。
- 选中态明确。
- 摘要文案随选择变化。
- 点击开始仍进入当前基础乱斗。

### P2：团队占点

目标：

- 先实现中心据点。
- HUD 展示据点归属和进度。
- 据点达到 100 后结算。

原因：

- 团队占点是最像“新玩法”的方案。
- 它改变胜利目标，比随机事件更适合作为第一批战役玩法。

### P3：小心叛徒

目标：

- 实现叛徒标记和倒计时叛变。
- 再接缩圈和黑洞。

### P4：众生平等（已完成代码实现，待真机验收）

完成项：

- 已实现安全生成的低密度障碍和单位反弹。
- 已实现一次性阵营减员事件与 Canvas 预警/结算反馈。
- 已使用“均衡清算”替代原版角色相关的运行时表达。

## 验收标准

- 玩法方案区域出现在支持阵营之后、初始阵容之前。
- 四个方案卡片在小屏下 2x2 显示，不横向溢出。
- 卡片文字不出现省略号。
- 选中后规则摘要清楚说明目标和机制。
- 切换玩法方案不重置主题、支持阵营、人数、时长。
- 现有基础乱斗、倒计时、暂停、重开、结算不退化。

## 暂不做

- 不直接实现全部原版开关。
- 不做全量可点击的高级规则折叠区。
- 未迁移机制只允许展示为“后续”，不允许开启运行。
