## 一、目标与范围

本次重写有三个层次，按优先级：

1. **玩法复刻**：当前规则（SRS 墙踢、7-bag、计分、锁定延迟、幽灵落点）是正确的，重写必须保证这些行为 1:1 不变，并用规则测试守住。
2. **性能优化**：解决长按左右、快速旋转、硬降、高等级下的卡顿，方法是拆文件 + Canvas 棋盘 + 状态机 + 按需重绘。
3. **鸿蒙原生质感**：延续鸡窝的奶油玻璃 / HarmonyOS 6 风格，不做像素掌机复刻；交互打磨见独立文档。

不在本次范围：

- 不改核心规则（不加新方块、不换计分公式、不换 SRS）。
- 不引入 WebView、HTML、跨端壳。
- 不做联机、排行、云存档。

## 二、现状诊断

### 2.1 违反开发规范的结构问题

核心文件 `entry/src/main/ets/games/tetris/TetrisPage.ets` 当前 1286 行，单文件包含：

- 规则（碰撞、旋转、墙踢表、7-bag、消行、计分）。
- UI（顶栏、棋盘、左右状态卡、控制面板、暂停弹窗、结算弹窗）。
- 主题（颜色、尺寸全散落在组件里）。
- 输入分发（控制面板触摸命中计算）。
- 游戏循环（setInterval）。

直接踩中规范第 3 节红线：

- 「文件超过 1000 行只能作为临时过渡，改动该文件时必须同步制定拆分计划」——已超。
- 「页面超过 800 行必须拆分」——已超。
- 「同一文件同时包含规则和 3 个以上 UI 区块时，必须拆分」——已超。
- 「游戏暂停、重开、结算、返回大厅要有清晰状态机，不用多个布尔值互相猜状态」——当前用 `paused / gameOver / lockActive / settled` 四个布尔互相组合。

### 2.2 性能根因（详见性能优化分析.md）

1. `tick` 进了棋盘 ForEach key，每次移动 / 旋转 / 落地都让 200 个格子被当新节点重建。
2. `colorAt()` 每个格子都重算当前方块和幽灵落点，幽灵落点内部又循环调碰撞检测。
3. 渲染期反复创建数组（`gridIndexes` / `previewIndexes` / `previewCells`）和字符串（`queueKey` / `boardColumnsTemplate`）。
4. `hardDrop()` 用 while 循环调 `moveInternal()`，十几格硬降 = 十几次状态更新。
5. 50ms 主循环 + 80ms 输入循环叠加，每次都触发全量棋盘重建。

结论：规则计算本身没坏，坏在「规则和渲染耦合 + 渲染成本过高」。所以重写的重点是**怎么组织**，不是**重新发明规则**。

## 三、玩法规格（两者结合决策）

玩法采用「现代化体验基本盘 + 同步参考项目的玩法味道」。下表每一项都给出依据，作为验收和回归测试基准。

| 规则点 | angular-tetris | 现有鸿蒙实现 | 本次采用 | 依据 |
|---|---|---|---|---|
| 棋盘 | 10×20 | 10×20 | **10×20** | 一致 |
| 方块表示 | 4×4 矩阵预定义 4 形态 | 相对坐标+运行时旋转 | **4×4 矩阵预定义** | 同步参考项目，代码与性能都更好，旋转零计算 |
| 出块 | 纯随机 | 7-bag | **7-bag** | 纯随机是 demo 简化，7-bag 更公平，现代标配 |
| 旋转踢墙 | 简单向左推 | 完整 SRS | **SRS（I 型专属表）** | 简单踢墙贴墙旋转失败率高，SRS 手感更好 |
| 锁定延迟 | 无 | 500ms+15 次 | **500ms+15 次** | 无锁定延迟手感发硬，是体验基本盘 |
| 幽灵落点 | 有 | 有 | **有** | 一致 |
| hold | 完整可操作 | 仅展示 | **可操作 hold** | 同步参考项目，每块限用一次 |
| next 预览 | 1 个 | 3 个 | **3 个** | 信息更足，不影响休闲感 |
| 计分 | 固定 [100,300,700,1500] | ×等级+连消+软硬降 | **固定档 + 软硬降分**（见 3.6） | 折中：休闲节奏，保留操作反馈 |
| 速度档 | 6 档 [700→160] 每 20 行 +1 | 10 档每 10 行 +1 | **6 档（同步参考）** | 更舒缓休闲，节奏贴近参考项目 |

坐标约定：`y` 向下为正（屏幕坐标）。旋转矩阵预定义，不再用运行时公式。

### 3.1 棋盘与方块

- 棋盘 `BOARD_COLS=10`、`BOARD_ROWS=20`。
- 7 种方块 I/O/T/S/Z/J/L，颜色沿用：
  - I `#06b6d4`、O `#eab308`、T `#8b5cf6`、S `#22c55e`、Z `#ef4444`、J `#3b82f6`、L `#f97316`。
- 每种方块预定义 4 个旋转形态的 4×4 布尔矩阵（参考 angular-tetris `SHAPES_*[Deg0/Deg90/180/270]`）。运行时按 `(kind, rotation)` 直接索引，渲染期不做旋转计算。
- 方块的占地格通过 `(row, col)` 遍历 4×4 矩阵得到，绝对坐标 = `(piece.y + row, piece.x + col)`。

### 3.2 生成与队列

- 7-bag：每轮打乱 `[0..6]`，逐个取出，取完再生成下一轮。
- next 队列展示前 3 个，底层维持至少 5 个。
- 生成位置 `x=4, y=0, rotation=0`；生成即碰撞则游戏结束。

### 3.3 移动与旋转

- 左右移动一格；软降一格。
- 旋转只做顺时针（`rotation = (rotation + 1) % 4`）。
- 墙踢：O 型不踢；I 型用专属 I-kick 表；其余用标准 SRS kick 表。完整 8 组 transition 与现有 `standardWallKicks` / `iWallKicks` 一致，搬迁时原样保留。
- 落地状态下做左右移动或旋转成功时，重置锁定延迟（move reset）。

### 3.4 锁定延迟

- 方块落地后启动 500ms 锁定计时。
- 移动 / 旋转可重置锁定计时，但最多 15 次；超过则不再重置。
- 锁定计时到期 → 锁定方块 → 消行 → 生成下一块。

### 3.5 幽灵落点

- 从当前方块位置向下找到不碰撞的最低位置，半透明显示。
- 游戏结束时不显示当前块和幽灵块。

### 3.6 消行与计分（折中方案）

- 满行清除，上方下落。
- 主消行分采用参考项目的**固定档**，不乘等级（休闲、不滚雪球）：
  - 单 / 双 / 三 / Tetris = `100 / 300 / 500 / 800`。
- 操作反馈分保留（删了软硬降无意义）：
  - 软降每格 +1。
  - 硬降每格 +2。
- 连消加分保留（连消是经典爽点）：`combo > 1` 时 `(combo - 1) × 50`（不乘等级）。
- 速度等级独立于计分：见 3.7。

### 3.7 速度等级（同步参考项目节奏）

- 6 档，对应下落间隔 `SPEED_DELAYS = [700, 600, 450, 320, 240, 160]`（毫秒）。
- 每消 20 行升 1 档，封顶第 6 档（即封顶 160ms）。
- 等级数字对外仍展示为「等级 1..6」，方便 HUD 显示。

### 3.8 Hold（恢复可操作）

- 每个方块限用一次 hold（参考项目行为）。
- 操作后，新方块回到顶部；hold 内为空时从 next 取，不为空时与当前交换。
- hold 后该块不可再 hold，直到下一块锁定。
- hold 入口在控制层实现（PR3 交互打磨阶段补触摸命中；数据通路在 PR1 就绪）。

### 3.9 控制

- 控制面板统一 onTouch 分发（真机 uiInput 下子控件单独 onClick 不稳定，见开发状态.md）。
- 点按：左 / 右 移动一格，下 软降一格，旋转 顺时针一次。
- 左右长按 180ms 后每 80ms 连续移动。
- 下长按 350ms 触发硬降。
- 系统返回键在游戏内返回大厅（交由 `exitToHub`），与全局约定一致。

## 四、技术方案

### 4.1 文件拆分（对齐规范第 3 节）

新增目录文件，全部落在 `entry/src/main/ets/games/tetris/`：

```text
TetrisPage.ets        页面入口：组装子组件、生命周期、状态机驱动、连接 recordScore/exitToHub
TetrisTypes.ets       类型、枚举、常量（PieceKind、TetrisPiece、Offset、BOARD_COLS/ROWS、DROP_INTERVALS、颜色表）
TetrisRules.ets       纯规则函数：碰撞、旋转、墙踢、消行、计分、幽灵 Y、7-bag（无 ArkUI 依赖，可单元测试）
TetrisTheme.ets       游戏私有颜色、尺寸、动效参数（cell 颜色、棋盘底色、面板色、圆角）
TetrisBoard.ets       Canvas 棋盘组件：drawBoard(ctx)，响应尺寸变化
TetrisHud.ets         左右状态卡：分数 / 行数 / 等级 / 状态提示
TetrisPreview.ets     next 队列 + hold 预览（小方块绘制）
TetrisControl.ets     底部控制面板：统一 onTouch 分发、长按 / 点按判定
TetrisOverlay.ets     暂停 / 结算覆盖层
```

最小可接受结构（规范第 3 节）是 Page + Types + Rules，本次直接做到完整拆分。规则和类型不放 UI，UI 不调底层存储。

### 4.2 规则层 TetrisRules.ets

把现有 Page 里的规则方法提取为纯函数，输入明确、输出明确、无 UI 副作用：

- `collides(board, piece): boolean`
- `cellsOf(piece): Offset[]`（预计算旋转，缓存到 PieceDef）
- `rotate(piece): Piece` + `tryRotateWithKick(board, piece, direction): Piece | null`
- `wallKicks(kind, from, to): ReadonlyArray<Offset>`（标准表 + I 表，原样搬迁）
- `ghostY(board, piece): number`
- `lockPiece(board, piece): number[]`（返回新棋盘）
- `clearLines(board): { board: number[], cleared: number }`
- `scoreForLines(cleared, level): number`
- `drawBag()` / `refillBag(bag): number[]`（7-bag）
- `dropIntervalForLevel(level): number`

设计要点：

- 方块形状预定义为矩阵，避免渲染期反复旋转（开源参考.md 的建议）。每种方块的 4 个旋转形态在模块加载时预计算成 `Record<PieceKind, Offset[][]>`，运行时直接按 `rotation` 索引。
- 函数全部基于 `board: number[]` 和 `piece` 入参，不持有 ArkUI 状态，便于在 node mjs 或 Hypium 里测。

### 4.3 渲染层 TetrisBoard.ets（Canvas）

棋盘从 `Grid + 200 个 GridItem` 改为单个 `Canvas`，参考 suika 的写法：

- `RenderingContextSettings(true)` + `CanvasRenderingContext2D`。
- `Canvas(context).onReady(() => drawBoard()).onAreaChange(...)` 同步尺寸。
- `drawBoard(ctx)` 一次绘制：背景 → 网格线 → 固定块 → 幽灵块 → 当前块。
- 不再用 `colorAt(index)` 每格查表，而是遍历预计算好的 `visibleBoard: Int8Array`（0 空 / 1-7 固定 / 负值表幽灵 / 当前块用独立标记），整盘一次画完。

绘制触发（按需 redraw，不轮询全量）：

- 移动 / 旋转 / 软降 / 硬降 / 锁定 / 消行 / 生成 / 重开 → 状态机进入对应态后调一次 `drawBoard`。
- 自然下落：由状态机的「下落调度」驱动，到点下落一格后 redraw 一次。
- 暂停 / 结算时不绘制活动块。

尺寸：`cellSize` 由 Page 根据可用宽高算出后传入，Canvas 宽 = `cellSize × BOARD_COLS`，高 = `cellSize × BOARD_ROWS`，保证像素对齐（`Math.floor`）。

### 4.4 状态机（替代布尔互猜）

引入明确的游戏状态枚举，规范第 7 节要求：

```text
Spawning   生成新块（极短，立即转 Falling）
Falling    方块自由下落
Locking    方块落地，锁定延迟计时中
Clearing   消行动效/结算中（可短暂时长，完成后回 Spawning）
Paused     暂停
GameOver   结算
```

转换示例：

- `Falling` + 到下落时间 + 能下落 → 下落，仍 `Falling`。
- `Falling` + 不能下落 → `Locking`，启动锁定计时。
- `Locking` + 移动/旋转成功且未超 move reset → 重置计时，回 `Falling`（若离地）或留 `Locking`。
- `Locking` + 计时到期 → 锁定 → 消行 → `Clearing` → `Spawning`。
- 任意态 + 暂停按钮 → `Paused`（记录来源态，继续时回来源态）。
- `Spawning` + 生成即碰撞 → `GameOver`。

好处：渲染分支、输入接收、定时器启停都只看单一 `state`，不再 `if (paused && !gameOver && lockActive ...)`。

### 4.5 性能要点（结合性能优化分析.md 阶段 1 + 阶段 2）

1. **visibleBoard 预计算**：状态变化后统一生成 `Int8Array(200)`，一次写入固定块 + 幽灵 + 当前块，Canvas 和（过渡期的 Grid）都只读这个数组。
2. **稳定 key**：过渡期若仍用 Grid，ForEach key 固定为 `index`，去掉 `tick`。
3. **hardDrop 一次性**：直接用 `ghostY` 算出落点，一次性 set piece + 加分 + 锁定，不再 while 循环 moveInternal。
4. **常量化**：`GRID_INDEXES`、`PREVIEW_CELLS_BY_KIND`、`BOARD_COLUMNS_TEMPLATE` 提为模块级常量。
5. **按需重绘**：去掉 50ms 全量轮询刷新；下落用「到点调度」，锁定用独立计时，只在状态变化时 redraw。
6. **TypedArray**：棋盘用 `Int8Array`，密集数值、无装箱。

### 4.6 生命周期与存储

- `aboutToDisappear` 清理：下落定时器、锁定定时器、输入长按定时器（规范第 7 节）。
- 分数只通过 `recordScore(score)` 上报，不直接读写 preferences。
- `session` prop 变化（@Watch）触发重开，保持与 Index.ets 现有 `tetrisSession` 机制兼容。
- 最高分展示用 `Math.max(bestScore, score)`，与现有一致。

### 4.7 接入保持不变

- `GameRegistry.ets` 的 tetris 条目不变。
- `Index.ets` 对 `TetrisPage` 的 props（`bestScore / session / exitToHub / recordScore`）不变。
- 重写只动 `games/tetris/` 目录内部，不影响大厅、存档、转场。

## 六、开发计划与 PR 拆分

遵循协作开发说明.md 的「小步 PR」节奏，按规范第 11 节每 PR 单一目标。建议 3 个 PR：

### PR1：规则层提取 + 拆文件 + 规则测试（行为不变）

目标：把规则搬出 Page，建立可测基线，不动渲染结构。

- 方块 4 旋转形态预计算。
- 补规则回归测试（见下「测试方案」）。
- Page 仍用 Grid，但先做低风险性能修复：去掉 `tick` 进 key、加 `visibleBoard`、`hardDrop` 一次性、常量化。

验收：玩法行为与重写前完全一致（规则测试 + 真机手感对比）；`build-app.sh` 通过；规则测试通过。

目标：换渲染、收敛状态，解决卡顿主因。

- 新建 `TetrisBoard.ets`（Canvas + drawBoard）。
- 引入游戏状态机枚举，替换布尔组合。
- 下落 / 锁定改为按需计时 + 按需 redraw，去掉 50ms 全量轮询。
- 棋盘 `Int8Array` + 一次性绘制。

验收：长按左右 / 快速旋转 / 硬降 / 等级 6+ 真机流畅；视觉与 PR1 一致；行为测试仍通过。

风险：Canvas 尺寸在横竖屏 / 不同机型像素对齐；暂停 / 结算时活动块显隐与现状一致。

### PR3：交互与视觉打磨

验收：见交互打磨设计.md 的验收清单。
