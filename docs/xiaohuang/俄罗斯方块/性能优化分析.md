# 俄罗斯方块性能优化分析

日期：2026-06-16

## 结论

当前俄罗斯方块的卡顿大概率不是规则计算本身导致的，而是棋盘渲染方式和派生数据计算方式导致 ArkUI 频繁重建大量节点。

最核心的问题有两个：

1. 棋盘用 `Grid + 200 个 GridItem` 渲染，并且每次移动、旋转、落地缓冲、生成方块都会更新 `tick`。棋盘格子的 `ForEach` key 包含 `tick`，所以每次 `tick` 变化都会让 200 个格子被视为新节点重建。
2. 每个格子的 `colorAt(index)` 都会重新计算当前方块和幽灵落点。幽灵落点又会循环调用碰撞检测。一次棋盘刷新会重复做大量本来只需要算一次的工作，并产生很多临时数组和对象。

所以用户感知到的“不流畅”，主要表现会是：

- 左右移动长按时明显卡顿。
- 快速点按旋转、软降时掉帧。
- 硬降时容易瞬间卡一下。
- 等级提高后，因为自然下落更频繁，卡顿更明显。

## 代码现状

核心文件：

- `entry/src/main/ets/games/tetris/TetrisPage.ets`

当前实现特点：

- 10 x 20 棋盘由 ArkUI `Grid` 渲染。
- 每个格子是一个 `GridItem`，内部再放一个 `Row` 作为色块。
- 当前方块、幽灵落点、固定方块全部通过 `colorAt(index)` 在渲染期合成。
- 游戏循环是 `setInterval(..., 50)`，每 50ms 检查一次是否需要自然下落或处理锁定延迟。
- 左右长按通过独立 `setInterval(..., 80)` 连续触发移动。
- `tick` 被当作手动刷新信号，在移动、旋转、生成、落地缓冲等路径中递增。

## 主要性能问题

### 1. `tick` 进入棋盘 key，导致 200 个格子频繁重建

位置：

- `TetrisPage.ets:218-226`
- `TetrisPage.ets:750`
- `TetrisPage.ets:797`
- `TetrisPage.ets:832`
- `TetrisPage.ets:888`

当前棋盘 key：

```ts
(index: number) => `${index}-${this.tick}-${this.board[index]}`
```

只要 `tick` 变化，所有格子的 key 都变化。ArkUI 会倾向于把这些格子当成不同节点，而不是复用旧节点更新背景色。

这对俄罗斯方块特别不利，因为当前方块每移动一格都会触发 `tick += 1`。长按左右时每 80ms 触发一次移动，等于持续重建整个 200 格棋盘。

建议：

- 棋盘格子 key 固定为 `index`。
- 不要把 `tick` 放入 key。
- 如果确实需要强制刷新，使用单独的可观察渲染数据，例如 `@State visibleBoard: number[]`，而不是靠破坏 key。

### 2. `colorAt()` 在每个格子里重复计算幽灵落点

位置：

- `TetrisPage.ets:1095-1110`
- `TetrisPage.ets:1121-1137`

当前每个格子都会执行：

```ts
this.cellsFor(this.current)
this.cellsFor(this.ghostPiece())
```

而 `ghostPiece()` 会从当前 y 位置向下循环测试，直到碰撞。每次测试都会调用 `collides()`，`collides()` 又会调用 `cellsFor()` 并创建数组。

粗略估算一次棋盘刷新：

- 200 个格子都会调用 `colorAt()`。
- 每个格子至少创建当前方块 cells 数组。
- 每个格子还会重新计算一次完整幽灵落点。
- 幽灵落点通常要检查多行，内部又反复创建 4 格数组和临时对象。

实际上当前方块和幽灵方块在同一次渲染里是不变的，只需要算一次。

建议：

- 在状态变化后统一生成 `visibleBoard`。
- 先复制固定棋盘，再把幽灵方块和当前方块叠进去。
- 每个格子的渲染只做一次数组读取，例如 `this.visibleBoard[index]`。
- 当前方块 cells、幽灵 cells 用索引数组或布尔数组表达，避免在 `colorAt` 里做 `find()`。

### 3. 渲染期反复创建数组和字符串

位置：

- `TetrisPage.ets:1139-1145`
- `TetrisPage.ets:1147-1153`
- `TetrisPage.ets:1164-1166`
- `TetrisPage.ets:1168-1199`
- `TetrisPage.ets:1201-1207`

当前这些函数会在 build / 子 build 过程中反复执行：

- `gridIndexes()` 每次创建 200 个数字的数组。
- `previewIndexes()` 每次创建 16 个数字的数组。
- `previewCells(kind)` 每个预览格都重新计算一次居中后的 4 格坐标。
- `queueKey()` 在 next 队列 UI 中多次 join 字符串。
- `boardColumnsTemplate()` 每次重新拼接 `1fr 1fr ...`。

这些单项都不大，但和 `tick` 高频刷新叠加后，会制造明显的临时对象和 GC 压力。

建议：

- 把 `GRID_INDEXES`、`PREVIEW_INDEXES`、`PREVIEW_SLOTS`、`BOARD_COLUMNS_TEMPLATE` 提为文件级常量。
- 预先计算 `PREVIEW_CELLS_BY_KIND`。
- `nextQueue` 变化时再计算队列 key，不要在渲染期多次 join。

### 4. `hardDrop()` 会连续触发多次状态更新

位置：

- `TetrisPage.ets:872-881`
- `TetrisPage.ets:780-798`

当前硬降：

```ts
while (this.moveInternal(0, 1, false)) {
  moved += 1;
}
```

`moveInternal()` 每成功移动一格都会：

- 创建新的 `current` 对象。
- 更新 `this.current`。
- `tick += 1`。

一次硬降可能移动十几格，理论上会连续触发十几次状态更新，最后再锁定、消行、生成下一块。

建议：

- 硬降只计算最终 y 和移动距离。
- 一次性设置 `current` 到最终位置。
- 一次性更新分数和渲染数据。
- 然后锁定生成下一块。

### 5. 50ms 游戏循环持续轮询，和输入计时器叠加

位置：

- `TetrisPage.ets:701-739`
- `TetrisPage.ets:588-607`

主循环每 50ms 运行一次，即使当前还没到自然下落时间，也会检查落地状态。长按左右时另有 80ms 的移动定时器。

这个设计本身可以接受，但在当前渲染成本很高的情况下，主循环和输入循环叠加会放大卡顿：

- 主循环触发落地缓冲时会 `tick += 1`。
- 输入循环触发移动时会 `tick += 1`。
- 任一 `tick` 都会导致棋盘全量重建。

建议：

- 第一阶段先降低单次刷新成本，不急着改循环模型。
- 第二阶段可以把自然下落改成“按下一次下落时间调度”的定时器，而不是固定 50ms 轮询。
- 锁定延迟可单独用 `lockDeadline` 判断，只有状态变化时刷新 UI。

### 6. UI 层级偏重，但不是第一瓶颈

棋盘外层、左右面板、控制面板使用了较多圆角、阴影、边框和半透明背景。这会增加绘制成本，但它们大部分不是每个输入都必须重建的内容。

当前第一瓶颈仍然是棋盘 200 格频繁重建和重复派生计算。先优化棋盘，收益最大。

## 推荐优化路线

### 阶段 1：不改视觉结构，先做低风险性能修复

目标：保留当前页面结构，把明显浪费去掉。

建议改动：

1. 移除 `tick` 对节点 key 的影响。
2. 用常量替代 `gridIndexes()`、`previewIndexes()`、`previewSlots()`、`boardColumnsTemplate()`。
3. 预计算所有方块的预览格坐标，替代 `previewCells(kind)` 在渲染期重复计算。
4. 新增 `visibleBoard` 或等价渲染数组，状态变化时统一计算：
   - `0`：空格
   - `1-7`：固定方块
   - `8-14`：幽灵方块
   - `15-21`：当前方块
5. `colorAt(index)` 改成简单查表。
6. `hardDrop()` 改成只提交一次状态更新。

预期收益：

- 长按左右移动的卡顿应明显下降。
- 硬降瞬间卡顿应明显下降。
- 低端机上的 GC 抖动会减少。

风险：

- 需要确保当前方块覆盖幽灵方块，幽灵方块覆盖空格但不覆盖固定方块。
- 需要确保游戏结束时当前方块/幽灵方块是否显示与现状一致。

### 阶段 2：把棋盘改成 Canvas 绘制

目标：把 200 个 ArkUI 节点降为 1 个 `Canvas`，游戏主区域只在需要时直接绘制。

仓库里已有可参考的 Canvas 写法：

- `entry/src/main/ets/games/suika/SuikaPage.ets`
- `entry/src/main/ets/games/eggSling/EggSlingPage.ets`
- `entry/src/main/ets/games/rps/RpsPage.ets`

建议结构：

- ArkUI 继续负责顶部栏、左右信息、控制按钮、暂停/结算弹窗。
- 棋盘区域改成 `Canvas`。
- 当前方块、幽灵落点、固定棋盘、网格线全部在 `drawBoard(ctx)` 中绘制。
- 每次移动、旋转、锁定、消行、生成方块后调用一次 `drawBoard()`。
- 分数、行数、等级等文本仍走 `@State` 更新。

预期收益：

- 棋盘不再创建/销毁 200 个 UI 子节点。
- 渲染逻辑完全可控。
- 后续做消行动效、锁定闪烁、轻微压缩动画更顺手。

风险：

- 需要处理 Canvas 尺寸和页面布局同步。
- 需要确保横竖屏、不同机型下棋盘像素对齐。
- 需要真机确认 Canvas 文本/矩形绘制清晰度。

### 阶段 3：拆出规则层并补测试

目标：降低后续优化和动效打磨时改坏规则的风险。

建议新增：

- `entry/src/main/ets/games/tetris/TetrisRules.ets`
- `entry/src/main/ets/games/tetris/TetrisTypes.ets`
- `scripts/test-tetris-rules.mjs`

优先覆盖：

- 碰撞检测。
- 左右移动边界。
- 顺时针旋转和墙踢。
- 软降/硬降落点。
- 锁定和消行。
- 7-bag 队列。

这样后续即使把棋盘换成 Canvas，也能确认玩法行为没有变。

## 建议的第一轮改代码清单

第一轮建议只做阶段 1，不直接上 Canvas，便于快速确认卡顿是否来自当前判断。

具体任务：

1. 增加文件级常量：
   - `GRID_INDEXES`
   - `PREVIEW_INDEXES`
   - `PREVIEW_SLOTS`
   - `BOARD_COLUMNS_TEMPLATE`
   - `PREVIEW_CELLS_BY_KIND`
2. 把棋盘 `ForEach` key 改成稳定 key：
   - `index => `${index}``
3. 把 next/hold 预览 key 中的 `tick` 去掉。
4. 新增 `@State visibleBoard: number[] = []`。
5. 新增 `refreshVisibleBoard()`：
   - 基于 `board` 复制 200 格。
   - 计算一次 `ghostPiece()`。
   - 计算一次 `cellsFor(current)`。
   - 按优先级写入渲染数组。
6. 所有改变棋盘显示的路径调用 `refreshVisibleBoard()`：
   - `restart()`
   - `spawn()`
   - `moveInternal()`
   - `rotateWithDirection()`
   - `holdPiece()`
   - `softDrop()`
   - `hardDrop()`
   - `lockAndSpawn()`
   - `finish()`
7. `colorAt(index)` 只读 `visibleBoard[index]`。
8. `hardDrop()` 改成一次性落到 ghost 位置，不再循环调用 `moveInternal()`。

如果第一轮后真机仍明显卡，则进入阶段 2，把棋盘改成 Canvas。

## 验收标准

功能验收：

- 进入游戏不是旧局残留。
- 左右点按移动一格。
- 左右长按 180ms 后连续移动，每 80ms 左右。
- 下键点按软降一格并加 1 分。
- 下键长按 350ms 硬降并锁定。
- 旋转只做顺时针旋转，墙边可正常墙踢。
- 幽灵落点位置正确。
- 消行、计分、等级、连消行为不变。
- 暂停、继续、再来一局、回大厅正常。

性能验收：

- 真机长按左右移动时，棋盘响应稳定，不出现明显一顿一顿。
- 快速连续点击旋转时，输入不丢、不延迟明显。
- 硬降不出现肉眼可见的长卡顿。
- 等级 6 以上自然下落频率提高后仍能正常操作。

## 后续备注

本分析基于静态代码阅读，还没有接入真机 profiler 或帧耗时日志。即便如此，当前渲染路径里“高频 key 变化 + 渲染期重复计算幽灵块”的问题已经足够明确，建议优先修掉。
