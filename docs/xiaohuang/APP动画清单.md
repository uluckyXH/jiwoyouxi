# APP 动画清单

更新时间：2026-06-10

## 当前结论

当前 App 级动画已经进入“够用、自然、先收口”的状态。后续不建议继续为了装饰感增加动画，优先把已有动画保持稳定，并在游戏功能成熟后复用现有组件。

目前继续优化 App 的重点应从“继续加动画”转向：

- 稳定布局。
- 控制文件拆分。
- 真机验收。
- 游戏成熟后接入统一弹层。
- 避免不同游戏各写一套交互风格。

## 当前保留效果

### 进入游戏

- 大厅向左退场。
- 游戏页从右侧入场。
- 顶部身份胶囊短暂显示游戏图标和游戏名。
- 最近游玩立即记录，不依赖动画结束。

### 返回大厅

- 游戏页向右退场。
- 大厅从左侧回来。
- 方向和进入相反。
- 如果身份胶囊仍在显示，返回时主动隐藏。

### 大厅交互

- 游戏卡片、Hero、收藏行、设置项按下轻微缩放。
- Tab 切换有轻微淡入和纵向位移。
- 按压反馈是视觉反馈，不是震动。

### 弹层

- 统一弹层组件已经可用。
- 目前只在扫雷试点接入。
- 后续其他游戏稳定后再逐个接入，不提前改游戏内部。

## 暂缓清单

| 能力 | 暂缓原因 | 后续条件 |
| --- | --- | --- |
| 真正 `geometryTransition` 共享元素 | 当前身份胶囊已经够用，真正共享元素要求页面结构配对，风险更高 | 游戏页标题/身份区统一后再试点 |
| `Navigation + NavPathStack` | 当前 `Index.ets` 薄路由足够，迁移会影响返回、生命周期和存档 | 游戏数量明显增加或需要标准页面栈时 |
| 全局真实震动 `HapticService` | 不是当前必要项，且要考虑设备支持和系统设置 | App 操作风格稳定后单独接入 |
| 所有游戏内部微动效 | 各游戏仍在开发，过早接入容易返工 | 单个游戏玩法和 UI 稳定后再接 |
| 常驻粒子/复杂装饰动画 | 小游戏主循环可能掉帧，也容易显得花 | 只在结算或短反馈中谨慎使用 |

## 后续复用策略

- 新增游戏不用自己写进入/退出转场，统一由 `Index.ets` 和 `GameSceneTransition.ets` 承担。
- 新增大厅可点击卡片或按钮，优先使用 `MotionPressable.ets`。
- 新增暂停、结算、确认弹层，优先使用 `CoopModalTransition.ets`。
- 游戏进入时的顶部身份提示统一使用 `GameEntryBadge.ets`。
- 新动效参数先放到 `CoopMotion.ets`，不要在页面里散落时长和位移数字。

## 验收清单

每次改 App 级交互或动画时，至少检查：

- 从大厅进入 6 个游戏，方向和节奏一致。
- 游戏内点“大厅”能回大厅。
- 游戏内按系统返回能回大厅。
- 快速重复点击游戏卡片不会卡死或打开错误游戏。
- 进入游戏身份胶囊显示正确游戏，不残留。
- 胶囊不挡顶部按钮，不拦截点击。
- 大厅卡片按压有反馈，但不会导致网格重排。
- Tab 切换不闪屏，不跳布局。
- 弹层遮罩和面板出现自然，不遮挡必要按钮。
- 扫雷暂停、继续、重新开始、胜负结算行为不变。
- 最近游玩、收藏、最高分不受动画影响。
- 小屏无文字重叠，无按钮压到系统手势区。
- `./scripts/build-app.sh --stacktrace` 构建通过。
