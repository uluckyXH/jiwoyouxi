# 合成大西瓜玩法对齐设计

## 目标

先把「合成大西瓜」的核心玩法对齐参考 H5 项目，素材图暂时不作为阻塞项。

本阶段目标不是复制参考项目，而是用 HarmonyOS 原生 ArkTS / Canvas 重新实现同类体验：

- 玩法节奏接近参考项目。
- 输入、生成、碰撞、合成、失败线、计分规则清楚稳定。
- 美术先用圆形、颜色、文字或鸡窝主题占位，后续可以替换成正式素材。
- 不引入 ArkWeb、WebView 或 H5 游戏壳。

## 参考来源

- 本地参考项目：`reference-projects/liyupi-daxigua`
- 当前鸿蒙实现：`entry/src/main/ets/games/suika/SuikaPage.ets`

参考项目仅用于研究玩法，不直接复制代码和素材。

## 当前实现概况

当前 `SuikaPage.ets` 已经具备基础可玩闭环：

- 11 级物体：葡萄到西瓜。
- 顶部可拖动瞄准线。
- 松手后投放当前物体。
- 简单重力、墙体、底部反弹。
- 圆形碰撞分离。
- 相同等级碰撞后合成下一等级。
- 顶部失败线持续超出后结算。
- 分数和最高分上报。

当前主要差距：

- 单文件承担了 UI、规则、物理、渲染，后续要拆模块。
- 生成规则和参考项目不完全一致。
- 计分规则和参考项目差异较大。
- 合成反馈较弱，缺少碎片、弹跳、分数浮动、最大水果庆祝。
- 失败线提示只有红虚线，缺少临界提醒。
- 自写物理比较简化，堆叠手感还需要专项调参。

## 参考项目玩法拆解

### 物体等级

参考项目使用 0 到 10 共 11 个等级：

| 等级 | 参考名称 | 行为 |
| --- | --- | --- |
| 0 | 葡萄 | 可生成，可合成 |
| 1 | 樱桃 | 可生成，可合成 |
| 2 | 橘子 | 可生成，可合成 |
| 3 | 柠檬 | 可生成，可合成 |
| 4 | 猕猴桃 | 可生成，可合成 |
| 5 | 西红柿 | 小等级随机池边界附近 |
| 6 | 桃 | 通过合成出现 |
| 7 | 菠萝 | 通过合成出现 |
| 8 | 椰子 | 通过合成出现 |
| 9 | 半个西瓜 | 合成后生成最终西瓜 |
| 10 | 大西瓜 | 最终目标，不继续普通升级 |

我们可以先继续用当前颜色圆形占位。正式主题可以改成鸡蛋等级，但内部等级编号保持 0 到 10。

### 生成规则

参考项目规则要点：

- 第一颗固定为等级 0。
- 开局前几颗使用固定序列，默认类似 `0, 0, 1, 2, 2, 3`。
- 后续只随机生成较小等级，不直接生成大物体。
- 当前物体在顶部生成时先不参与物理碰撞。
- 玩家拖动改变当前物体的水平位置。
- 松手后启用碰撞体和刚体，下落进入容器。
- 下一颗物体延迟约 0.5 秒出现，避免连续投放过快。

我们需要对齐：

- `currentLevel` 初始固定为 0。
- 新增 `spawnCount` 和 `START_LEVELS = [0, 0, 1, 2, 2, 3]`。
- 固定序列用完后，从小等级池随机生成，建议先用 `0..4`。
- 当前预览物体必须水平跟手，但不能被已有物体碰撞影响。
- 松手后才成为正式下落物体。

### 输入规则

参考项目输入是横向拖动和松手投放：

- `touch start`：当前物体移动到触点 x。
- `touch move`：当前物体持续跟随 x。
- `touch end/cancel`：释放当前物体。

我们需要对齐：

- 保留单指横向拖动。
- 触点 x 要限制在容器左右边界内，并根据当前物体半径留边。
- 已暂停、结算、投放冷却期间不接受投放。
- 如果没有当前预览物体，不响应投放。

### 物理规则

参考项目使用 Cocos 物理系统：

- 重力向下。
- 物体是圆形碰撞体。
- 左右墙和底部限制运动。
- 物体落到底部或碰到其他物体后进入堆叠。
- 可通过阻力和弹性调节手感。

我们继续使用自写轻量物理，但体验目标要对齐：

- 圆形物体不能穿墙、穿底、明显重叠。
- 底部反弹要弱，最终能稳定堆叠。
- 横向速度要有阻尼，避免物体长期滑动。
- 同等级合成前需要先处理碰撞接触，避免一帧多次合成。
- 新合成物体在合成点生成，带轻微弹出或缩放反馈。

### 合成规则

参考项目核心逻辑：

- 只有相同等级碰撞才合成。
- 等级 0 到 8：两个相同等级合成一个 `level + 1`。
- 等级 9：两个半瓜合成等级 10，并触发最大水果庆祝和额外奖励。
- 等级 10 不继续普通合成。
- 合成时两个旧物体先禁用碰撞，再生成新物体，避免重复触发。

我们需要对齐：

- 合成结果以碰撞双方中心点或接触点为生成位置。
- 每个物体增加 `isMerging` 或 `mergeLockedUntil`，防止重复合成。
- 最大等级合成时保留等级 10 物体，并触发一次最大水果反馈。
- 等级 10 与等级 10 碰撞不再升级，最多做普通碰撞分离。

### 计分规则

参考项目的主要计分方式：

- 普通合成得分约为 `源等级 + 1`。
- 合成最终大西瓜时，除等级 9 合成分外，还会额外加 100 分。
- 结束流程中会按剩余物体等级逐个追加一点分数和碎片反馈。

当前实现是按合成后的等级分值加固定奖励，分数增长明显更快。

建议对齐为：

- 普通合成：`score += sourceLevel + 1`。
- 等级 9 合成等级 10：先加 `10`，再加最大水果奖励 `100`。
- 结算时是否追加剩余物体分数先做成配置项，默认关闭或弱化，避免玩家不理解“结束后还涨分”。

如果后续想要更爽快的分数，可以在鸡窝主题版本另设一套分值表，但需要在文档里明确“不是参考项目规则”。

### 失败线规则

参考项目有顶部高度线：

- 物体堆叠高度接近线时，线会闪烁提醒。
- 物体超过线后不会立刻结束。
- 已下落进入容器的物体持续超过线约 3 秒后，触发结束。
- 新生成但还没释放的当前物体不参与失败判定。

我们需要对齐：

- 失败线固定在容器顶部附近，不跟随物体移动。
- 检测对象只包括已释放、已进入容器的正式物体。
- 单个物体超过失败线后开始计时，回落则清零。
- 持续超线时间建议设为 `3000ms`。
- 临近失败线时提供明显但不吓人的提示，例如线条变亮、轻微闪烁。

### 反馈规则

参考项目反馈较完整：

- 新物体出现有缩放弹出。
- 合成时旧物体靠近、碎片飞散、音效、分数跳动。
- 最大西瓜有专门庆祝。
- 失败时逐个清场并展示结算。

本阶段可以先做轻量占位：

- 新物体预览：圆形从小到大弹出。
- 普通合成：新圆形短暂放大，出现 `+N` 浮动文字。
- 最大合成：屏幕中心短庆祝，文案可暂定“巨型鸡窝西瓜！”。
- 失败：保留当前结算弹窗，后续再做主题化卡片。

## 对齐后的模块设计

按 `docs/鸿蒙原生开发规范.md`，后续改造建议拆成：

```text
entry/src/main/ets/games/suika/
  SuikaPage.ets       页面入口，组装 Canvas、HUD、弹窗
  SuikaTypes.ets      FruitDef、FruitBody、GameStatus、配置类型
  SuikaRules.ets      生成、合成、计分、失败线等纯规则
  SuikaPhysics.ets    重力、边界、圆碰撞、堆叠稳定
  SuikaRenderer.ets   Canvas 绘制容器、物体、预览、反馈
  SuikaTheme.ets      颜色、尺寸、临时占位风格
  SuikaHud.ets        顶部按钮、分数、最高分
  SuikaOverlay.ets    暂停和结算弹窗
```

第一阶段最小拆分可以先做到：

```text
SuikaPage.ets
SuikaTypes.ets
SuikaRules.ets
SuikaPhysics.ets
```

原因是玩法对齐要先可测试，UI 和渲染可以随后再拆。

## 玩法对齐清单

优先级从高到低：

1. 固定开局序列：第一颗 0，后续 `0, 0, 1, 2, 2, 3`，再随机小等级。
2. 投放状态：当前物体未释放前不参与物理，松手后才下落。
3. 下一颗生成：释放后延迟约 500ms 出现下一颗。
4. 合成边界：0 到 8 普通升级，9 合成 10，10 不继续升级。
5. 计分对齐：普通合成按源等级 + 1，最大合成额外 +100。
6. 失败线：只检测已释放物体，持续超线 3000ms 才结束。
7. 物理手感：弱反弹、强阻尼、稳定堆叠，不穿墙不穿底。
8. 合成锁：避免同一对物体在多帧重复合成。
9. 反馈占位：合成放大、分数浮动、失败线警示。
10. 模块拆分：把规则和物理从 `SuikaPage.ets` 中拆出。

## 临时美术方案

素材未补齐前使用程序绘制：

- 物体：圆形或轻微椭圆，保留柔和高光。
- 文案：大等级显示名称，小等级只显示颜色。
- 背景：鸡窝主题色占位，例如浅木板、暖黄地面。
- 容器：先用线框和半透明底色，后续换成切图边框。
- 特效：用 Canvas 绘制小圆点、短线、浮动分数，不依赖图片。

内部等级名称先保留水果，等主题定稿后再统一替换为鸡蛋或鸡窝对象。

## 自动化验证建议

玩法规则应优先做纯函数测试，避免只靠真机观察。

建议新增脚本：

```text
scripts/test-suika-rules.mjs
```

覆盖：

- 固定开局序列是否正确。
- 随机池是否只生成小等级。
- 同等级合成结果是否正确。
- 等级 10 是否不会继续升级。
- 计分是否符合规则。
- 新生成预览物体是否不参与失败线。
- 已释放物体持续超线 3000ms 后才失败。

构建验证仍使用：

```bash
./scripts/build-app.sh --stacktrace
```

## 下一步执行建议
