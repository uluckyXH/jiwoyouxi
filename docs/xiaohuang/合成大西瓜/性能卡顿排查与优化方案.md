# 合成大西瓜性能卡顿排查与优化方案

## 背景

当前体验现象：

- 水果数量少时基本顺畅。
- 水果堆叠较多后，尤其是大量小水果堆在容器底部时，会出现明显掉帧和卡顿。
- 最近新增了水果贴图、阴影、合成粒子、音效和轻微视觉滚动，视觉反馈更完整，但也放大了原有性能压力。

结论：卡顿不是单一素材或单一音效导致的，更主要是当前主循环在水果数量增大后同时触发了 **全量物理碰撞计算** 和 **全量 Canvas 重绘**。

## 当前实现位置

核心文件：

```text
entry/src/main/ets/games/suika/SuikaPage.ets
entry/src/main/ets/games/suika/SuikaPhysics.ets
entry/src/main/ets/games/suika/SuikaTypes.ets
```

主循环在 `SuikaPage.ets`：

```text
step()
  activatePendingFruit(now)
  applyPhysics()
  resolveCollisions(now)
  checkOverflow(now)
  draw()
```

物理层在 `SuikaPhysics.ets`：

```text
integrateFruits(...)
resolveSuikaCollisions(...)
```

渲染层在 `SuikaPage.ets`：

```text
draw()
  clearRect(...)
  drawDropZone(...)
  drawContainer(...)
  for each fruit:
    drawFruitShadow(...)
    drawFruit(...)
  drawMergeEffects(...)
  drawMergeFeedback(...)
```

## 本地压测结果

本次用 Node 加载当前 `SuikaTypes / SuikaRules / SuikaPhysics` 做了一个物理层压测。该测试只测物理，不包含 ArkUI、Canvas、图片绘制、真机 GPU/CPU 成本。

| 水果数量 | 物理层平均耗时 |
| --- | --- |
| 20 | 0.116 ms/frame |
| 40 | 0.205 ms/frame |
| 80 | 0.628 ms/frame |
| 120 | 1.321 ms/frame |
| 160 | 2.191 ms/frame |
| 220 | 3.979 ms/frame |

这个数值在桌面 Node 环境看起来不夸张，但趋势很明确：水果数量上升后不是线性增长，而是接近平方增长。真机 ArkTS 运行时再加上 Canvas 绘制、图片缩放和 UI 线程压力，实际卡顿会更明显。

## 主要原因

### 1. 碰撞检测是全量两两检查

当前 `resolveSuikaCollisions(...)` 对所有水果做双重循环：

```text
for i in fruits:
  for j in i + 1 .. fruits:
    checkCollision(i, j)
```

复杂度是 `O(n²)`。

大致检查次数：

| 水果数量 | 每帧水果对数 |
| --- | --- |
| 40 | 780 |
| 80 | 3160 |
| 120 | 7140 |
| 160 | 12720 |
| 220 | 24090 |

而且当前每一对都会先算：

```text
sqrt(dx * dx + dy * dy)
```

实际大多数水果对并不会碰撞，但仍然付出了开方成本。

### 2. 每帧创建大量临时对象和数组

当前 `integrateFruits(...)` 每帧都会：

- 新建 `next: FruitBody[]`。
- 为每个水果 `next.push({ ... })` 创建新对象。
- `resolveSuikaCollisions(...)` 里 `fruits.slice()`。
- 发生合成时再创建 `kept: FruitBody[]`。

水果越多，每秒创建的对象越多。60fps 下，100 个水果就是每秒至少 6000 个 `FruitBody` 临时对象，还不算碰撞、粒子、数组切片。

这会增加 GC 压力。表现上可能不是持续慢，而是堆叠一多后突然卡一下。

### 3. Canvas 每帧全量清屏重绘

当前每帧都会：

- 清空整张 Canvas。
- 重新绘制背景和容器。
- 给每个水果绘制两层阴影。
- 给每个水果绘制贴图。
- 对水果贴图做 `save / translate / rotate / drawImage / restore`。
- 绘制粒子和浮动文字。

水果数量多时，渲染成本是线性增长。它和物理层的平方增长叠加后，堆叠场景就容易掉帧。

### 4. 水果贴图都是 512x512

当前水果素材：

```text
entry/src/main/resources/rawfile/games/suika/fruits/*.png
```

全部是 `512 x 512`。

运行时 11 张贴图的解码内存不是主要问题，但每帧把 512 图缩放绘制到不同尺寸，会有额外成本。水果少时没问题，水果多时会和阴影、旋转一起放大开销。

### 5. 主循环使用固定 16ms 定时器

当前循环：

```text
setInterval(() => this.step(), 16)
```

这相当于强行按 60fps 推进。轻负载时没问题，但如果某一帧计算和绘制已经超过 16ms，下一次 tick 仍然会继续尝试执行，容易进一步挤压 UI 线程。

更合理的是帧同步或自调度循环：上一帧完成后再安排下一帧，并根据耗时决定是否降级绘制。

## 优化方向

### 优先级 P0：先降低碰撞检测成本

这是最应该先做的，因为它是平方增长。

#### 方案 A：先用平方距离过滤

当前是每一对都开方。

建议改成：

```text
distSq = dx * dx + dy * dy
target = ar + br
if distSq >= target * target:
  continue
dist = sqrt(distSq)
```

这样只有真正接近或重叠的水果才开方。

优点：

- 改动小。
- 风险低。
- 不改变玩法结果。
- 很适合作为第一步优化。

#### 方案 B：加入空间分区

当前所有水果互相检查。可以改成只检查附近水果。

可选实现：

```text
SuikaSpatialHash.ets
  buildSpatialHash(fruits)
  queryNearbyPairs(fruit)
```

推荐做法：

- 用统一网格或空间哈希。
- 每个水果按包围盒覆盖的格子加入索引。
- 只检查同格子和相邻格子的水果。
- 用 pair key 去重，避免同一对重复检查。

目标：

- 从 `O(n²)` 降到接近 `O(n * k)`。
- `k` 是每个水果附近的候选数量，堆叠合理时远小于总水果数。

注意：

- 不要只按中心点入一个格子，否则大水果可能漏检。
- 应按水果包围盒覆盖多个格子。
- 空间分区只做 broad phase，最终碰撞仍用圆形距离判断。

### 优先级 P1：减少每帧对象分配

当前帧循环会不断创建新数组和新对象。

建议：

- `integrateFruits(...)` 改为原地更新，或者引入对象池。
- `resolveSuikaCollisions(...)` 尽量复用数组。
- 合成删除两个水果时，可以用 swap-remove，避免每次重建 `kept`。
- 粒子数组用原地 compact，不要每帧创建 `activeParticles` 和 `activeRings` 新数组。

示意：

```text
let writeIndex = 0
for each particle:
  if alive:
    particles[writeIndex] = particle
    writeIndex += 1
particles.length = writeIndex
```

这类优化不改变玩法，但能明显降低 GC 抖动。

### 优先级 P1：渲染降级策略

视觉效果要保留，但堆叠很多时需要自动降级。

建议：

- 水果数量超过 80 时，阴影从两层椭圆降为一层。
- 水果数量超过 120 时，静止水果不绘制阴影，或只绘制底部接触阴影。
- 粒子数量继续保留上限，目前 `mergeParticles` 上限 160 可接受，但可以根据水果数量降低到 80。
- 对完全静止的水果，可以减少旋转 transform 或使用更简单绘制路径。

判断静止：

```text
abs(vx) < 0.03 && abs(vy) < 0.03 && abs(visualSpin) < 0.001
```

### 优先级 P2：拆出渲染器和性能计数器

现在 `SuikaPage.ets` 同时承担页面、输入、渲染、特效和生命周期，排查性能时不够清楚。

建议新增：

```text
entry/src/main/ets/games/suika/
  SuikaRenderer.ets
  SuikaPerformance.ets
```

`SuikaPerformance.ets` 可以记录：

```text
fruitCount
collisionPairCount
collisionHitCount
physicsMs
drawMs
particleCount
frameMs
```

开发期可以只 `console.info`，后续也可以做隐藏开发者面板。

### 优先级 P2：图片尺寸和缓存

当前正式水果贴图是 512x512。

建议：

- 保留原图归档，但 app 内实际运行素材可以出一套 256x256。
- 小水果等级可以用 192 或 256，大水果用 384。
- 如果后续发现 Canvas 缩放成本明显，可以预生成常用尺寸缓存。

这不是第一优先级，因为目前最明显的复杂度问题仍在碰撞。

### 优先级 P2：调整主循环

当前是固定 `setInterval(..., 16)`。

建议改为：

- 使用 HarmonyOS 可用的帧同步能力；或
- 改成自调度 `setTimeout`，上一帧完成后再安排下一帧；或
- 检测帧耗时，超过预算时跳过部分视觉效果。

目标不是降低玩法刷新频率，而是避免在卡顿时继续积压帧任务。

## 推荐实施顺序

### 第一步：低风险优化

1. 碰撞检测先改平方距离过滤，减少开方。
2. 给 `resolveSuikaCollisions(...)` 增加 `pairChecks / collisionHits` 统计，方便验证。
3. 粒子数组改原地 compact，减少临时数组。
4. 保持玩法和视觉不变。

验收：

- 自动化规则测试通过。
- 构建通过。
- 同样堆叠场景下卡顿有下降。

### 第二步：结构性优化

1. 新增 `SuikaSpatialHash.ets`。
2. 碰撞 broad phase 改为空间分区。
3. 保留圆形精确碰撞作为 narrow phase。
4. 新增压测脚本，比较 80、120、160、220 个水果的物理耗时。

验收：

- 120 个水果以上物理耗时不再接近平方增长。
- 合成结果不变。
- 失败线和计分规则不变。

### 第三步：渲染降级

1. 新增 `SuikaRenderer.ets`，把 Canvas 绘制从页面拆出去。
2. 根据水果数量降低阴影和粒子复杂度。
3. 评估是否需要 256x256 运行素材。

验收：

- 水果少时视觉保持现在的质感。
- 水果多时宁可阴影少一点，也不能明显卡顿。

## 当前判断

这次卡顿最可能的直接原因是：

```text
O(n²) 全量碰撞检测
+ 每帧所有水果全量绘制
+ 每帧大量临时对象导致 GC 抖动
```

音效文件不是主要原因。512x512 水果贴图会增加渲染压力，但不是第一优先级。刚加的轻微视觉滚动也会增加一点 Canvas transform 成本，但它只是放大因素，不是根因。

先做平方距离过滤和空间分区，收益会比继续微调阴影、粒子更明确。
