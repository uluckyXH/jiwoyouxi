# 弹射鸡蛋横屏进入引导

## 目标

弹射鸡蛋需要横屏空间才能玩得舒服。第一版采用“首次提醒 + 动态请求窗口横屏 + 横屏世界兜底”的方案：

- 第一次进入弹射鸡蛋时，如果是竖屏，提醒用户横屏体验更好。
- 如果进入时已经是横屏，不显示提醒，直接开始。
- 进入弹射鸡蛋时，请求主窗口切到横屏。
- 退出弹射鸡蛋时，恢复主窗口默认方向。
- 游戏世界始终按横屏布局设计。
- 已经开始游玩后，如果用户又转回竖屏，不盖遮罩、不暂停、不重置关卡。
- 竖屏期间游戏继续运行，用户自己决定是否转回横屏。

## 为什么不全局强制横屏

当前项目是小游戏合集，所有游戏都挂在同一个 `EntryAbility` 和 `pages/Index` 下。

如果直接把 Ability 或 `module.json5` 配成横屏，会影响：

- 大厅。
- 扫雷。
- 其他竖屏更合适的小游戏。

所以第一版采用游戏页生命周期策略：只在 `EggSlingPage.ets` 进入时动态请求窗口横屏，离开时恢复默认。这样状态栏和窗口会真正横过来，但不会把整个小游戏合集写死为横屏。

## 方向检测

第一版实现里不能只依赖游戏内容层的尺寸。因为弹射鸡蛋会把游戏世界按横屏旋转展示，内容层尺寸可能已经是横屏，不能完全代表设备首次进入时的真实方向。

当前实现采用两层判断：

1. 优先使用 HarmonyOS SDK 的 `display.getDefaultDisplaySync().orientation`。
2. 如果 SDK 方向读取失败，再退回页面根容器 `.onAreaChange(...)` 拿到的宽高。

`EggSlingPage.ets` 会通过 `.onAreaChange(...)` 更新页面尺寸：

```text
pageWidth
pageHeight
```

兜底方向判断：

```text
pageWidth > pageHeight  => 横屏
pageWidth <= pageHeight => 竖屏
```

实现里保留辅助方法：

```text
private isLandscape(): boolean
```

后续游戏世界布局和交互围绕横屏世界尺寸走；首次提示优先看 SDK display 方向。

## 窗口方向请求

进入弹射鸡蛋时：

```text
window.Orientation.LANDSCAPE
```

退出弹射鸡蛋时：

```text
window.Orientation.UNSPECIFIED
```

`LANDSCAPE` 是固定横屏，优点是用户竖屏进入时也能稳定切横屏，不依赖传感器或自动旋转开关。

后续如果希望左右横拿都跟随传感器，可以评估替换为：

```text
window.Orientation.AUTO_ROTATION_LANDSCAPE
```

当前先不使用它，避免不同模拟器或真机的传感器行为影响第一版验收。

## 状态设计

建议新增一个提醒状态：

```text
hasShownLandscapeHint: boolean
```

含义：

- `false`：本次进入弹射鸡蛋还没展示过横屏提醒。
- `true`：本次已经提示过，不再反复打扰用户。

状态流转：

```text
进入页面
  |
  |-- SDK/页面判断当前是竖屏 --> 显示一次横屏提醒，hasShownLandscapeHint = true
  |
  |-- SDK/页面判断当前是横屏 --> 不显示提醒，直接进入游戏
  |
  |-- 请求主窗口 LANDSCAPE

游戏中
  |
  |-- 转回竖屏 --> 不遮罩、不暂停，继续横屏世界
  |
  |-- 再转横屏 --> 继续正常显示
```

## 首次进入体验

竖屏首次进入时，显示轻量提醒，但不把提醒设计成强制阻断。

推荐内容：

```text
请横屏开始游戏
```

提醒层建议保持克制：

- 可以用 Toast、短暂浮层或轻量弹窗。
- 显示后自动消失，或用户点一下关闭。
- 不显示复杂教学文字。

提醒不应该反复出现。用户如果坚持竖屏玩，就让游戏继续。

## 横屏进入

如果用户一开始就是横屏进入：

- 不显示横屏提醒。
- 仍请求窗口横屏，确保系统窗口状态一致。
- 直接初始化或恢复关卡。
- 允许拖拽鸡蛋。
- 开启物理更新。

如果用户一开始竖屏进入：

- 显示一次横屏提醒。
- 请求窗口切到横屏。
- 游戏仍然可以初始化。
- 不等用户横屏才开始。

## 游戏中转回竖屏

用户已经开始玩之后，如果转回竖屏，第一版不重置关卡。

建议处理：

- 不显示黑屏遮罩。
- 不暂停拖拽输入。
- 不暂停物理 tick。
- 保留当前鸡蛋、目标、分数、剩余鸡蛋状态。

这样最不打扰用户。后续如果测试发现竖屏继续运行体验差，再评估是否改成暂停或遮罩。

## UI 分层

页面渲染建议分三层：

1. 游戏场景层：Canvas 或后续素材场景。
2. HUD 层：关卡、分数、剩余鸡蛋、暂停、返回。
3. 首次提醒层：只在竖屏首次进入时短暂显示。

逻辑：

```text
if (!isLandscape && !hasShownLandscapeHint) {
  显示一次横屏提醒
  hasShownLandscapeHint = true
}

显示横屏世界游戏
```

游戏场景层始终按横屏坐标设计，不为竖屏另做一套布局。

## 输入规则

只有在下面条件同时满足时，才允许拖拽发射：

```text
!paused
!gameOver
eggsLeft > 0
egg.resting
```

如果不满足，触摸事件直接忽略。

## 物理更新规则

物理 tick 不因为竖屏暂停：

```text
if (paused || gameOver) {
  return
}
```

这样用户转回竖屏时，鸡蛋仍然继续飞行、碰撞和结算。

## 是否记录用户已看过引导

第一版不需要持久化“已看过引导”。

原因：

- 横屏提示和本次进入方向强相关。
- 用户下次竖屏进入仍然可以再看到提醒。
- 不增加存储字段，避免污染全局记录结构。

## 验收标准

- 竖屏第一次进入弹射鸡蛋，显示一次横屏提醒。
- 横屏第一次进入弹射鸡蛋，不显示横屏提醒。
- 进入弹射鸡蛋后，窗口和状态栏切到横屏。
- 竖屏第一次进入时，游戏仍可继续初始化。
- 游戏中转回竖屏，当前局不重置、不遮罩、不暂停。
- 游戏中竖屏时，鸡蛋物理继续运行，触摸输入仍按横屏世界处理。
- 转回横屏后继续当前局。
- 返回大厅后不影响大厅和其他游戏方向布局。
