# 底部菜单图标对齐分析

更新时间：2026-06-10

## 背景

当前首页底部菜单的四个按钮肉眼看起来没有对齐在一条线上。这里先只分析原因和图标方案，不直接改代码。

## 当前实现

位置：`entry/src/main/ets/pages/HubPage.ets`

当前底部菜单在 `BottomTabBar()` 里写死四个文本符号：

```ts
this.BottomTabItem(0, '⌂', '首页')
this.BottomTabItem(1, '♡', '收藏')
this.BottomTabItem(2, '♕', '成就')
this.BottomTabItem(3, '⚙', '设置')
```

`BottomTabItem()` 内部用 `Text(icon)` 渲染图标，再用另一个 `Text(label)` 渲染文案。也就是说，现在底部菜单图标不是图片、不是 SVG、不是鸿蒙 SymbolGlyph，也不是图标库，而是 Unicode 字符。

当前项目资源目录 `entry/src/main/resources/base/media/` 只有：

- `app_icon.png`
- `start_icon.png`
- `thanos.png`

当前 `oh-package.json5` 和 `entry/oh-package.json5` 没有任何图标库依赖。

## 没对齐的主要原因

### 1. Unicode 字符的视觉框不一致

`⌂`、`♡`、`♕`、`⚙` 属于不同 Unicode 字符。即使它们都设置 `fontSize(20)`，每个字符的实际字形高度、宽度、上留白、下留白、视觉重心都不一样。

结果是：布局层面可能已经居中，但视觉上仍然会偏上、偏下、偏左或偏右。

### 2. 图标 Text 没有固定绘制盒子

当前 `Text(icon)` 只有：

```ts
.fontSize(20)
.fontColor(...)
```

没有固定：

- `width`
- `height`
- `lineHeight`
- `textAlign`

因此每个图标的文本行框由字体和字符自身决定。不同字符会产生不同的视觉占位。

### 3. Column 居中的是整体，不是图标基线

当前结构是：

```ts
Column({ space: 3 }) {
  Text(icon)
  Text(label)
}
.height('100%')
.justifyContent(FlexAlign.Center)
```

`Column` 会把“图标 Text + 文字 Text + space”这个整体居中。它不能修正每个图标字形自身的视觉重心。

### 4. 底部手势区不是横向不齐的根因

当前底部 Tab 已经有底部安全区承接：

- Tab 总高度：`96vp`
- 底部预留：`24vp`
- `expandSafeArea([SafeAreaType.SYSTEM], [SafeAreaEdge.BOTTOM])`

这些会影响整个 Tab 的垂直位置，但对四个按钮是一致影响。四个按钮之间不在一条线，主要还是图标来源和图标盒子不统一。

## 鸿蒙项目可选图标方案

### 方案 A：ArkUI SymbolGlyph

优先级：高

这是鸿蒙原生方案。ArkUI 提供 `SymbolGlyph` 组件，并支持系统符号资源，例如官方示例里出现过：

```ts
SymbolGlyph($r('sys.symbol.chevron_left'))
new SymbolGlyphModifier($r('sys.symbol.bell')).fontColor([Color.Red])
```

`SymbolGlyph` 支持：

- `fontSize`
- `fontColor`
- `fontWeight`
- `renderingStrategy`
- `effectStrategy`
- `symbolEffect`

适合底部导航这类统一图标场景。它比 Unicode 文本符号更适合做图标，因为系统符号资源的设计、尺寸和渲染语义更统一。

风险：

- 需要确认当前 SDK 中是否存在合适的首页、收藏、成就、设置图标。
- 系统符号命名需要在 DevEco / SDK 文档里逐个验证，不能直接猜。

建议：

- 下一步优先试用 `SymbolGlyph`。
- 如果 `sys.symbol` 中有完整可用图标集，就不引入三方库。

### 方案 B：项目自有 SVG / PNG 图标资源

优先级：高

把四个 Tab 图标作为项目资源放到：

```text
entry/src/main/resources/base/media/
```

然后通过：

```ts
Image($r('app.media.xxx'))
```

统一以固定 `24vp x 24vp` 或 `26vp x 26vp` 的图标盒子渲染。

优点：

- 视觉最可控。
- 适合做“鸡窝游戏”自己的品牌风格。
- 不依赖三方库。
- 图标可以由设计稿直接导出。

风险：

- 需要维护浅色/深色/选中/未选中状态。
- 如果使用 PNG，多状态和不同分辨率管理成本更高。

建议：

- 如果我们希望首页更有游戏品牌感，用自有 SVG/PNG 更合适。
- 如果只是做标准功能图标，优先用 `SymbolGlyph`。

### 方案 C：ArkUI 高阶组件的图标能力

优先级：中

ArkUI 的一些高级组件支持 `SymbolGlyphModifier` 或图标配置，例如 Toolbar、ChipGroup、TabTitleBar 等。它们适合标准工具栏、标题栏或标签栏。

当前首页底部导航是自定义 `Row + Column`，没有直接使用这些高级组件。如果后续要完全走鸿蒙标准导航组件，可以再评估 `Tabs` / `TabContent` / `TabTitleBar`，但这会比只替换底部图标更大。

建议：

- 当前阶段不建议为了图标对齐重构整个首页导航组件。
- 先保留自定义底部 Tab，只把图标来源和固定盒子做好。

### 方案 D：ohpm 三方图标库

优先级：低

理论上可以从 ohpm 引入图标相关包，但当前不建议作为第一选择。

原因：

- 当前项目未安装任何三方图标依赖。
- 本机环境没有 `ohpm` 命令，无法直接查询和验证包。
- ohpm 官网页面无法在当前环境稳定抓取可验证包信息。
- 三方包需要额外确认维护状态、授权协议、ArkTS 兼容性、包体积和深浅色适配方式。

建议：

- 如果 `SymbolGlyph` 和自有 SVG 都不能满足，再单独做三方库选型。
- 三方库进入项目前，需要先落一份依赖评估：包名、版本、许可证、维护状态、体积、是否支持 ArkTS/HarmonyOS NEXT。

## 推荐改造方向

下一步建议做一个低风险版本：

1. 新增 `HomeTabIcon` 或 `BottomTabIcon` 小组件。
2. 每个图标固定一个视觉盒子，例如 `26vp x 26vp`。
3. 图标和文字都固定高度：
   - 图标盒子：`26vp`
   - 标签文字：`14vp` 到 `16vp`
   - 图标和文字间距：`2vp` 到 `4vp`
4. 优先尝试 `SymbolGlyph($r('sys.symbol.xxx'))`。
5. 如果某个系统符号不好看，再用自有 SVG/PNG 补齐。

推荐结构示意：

```ts
Column({ space: 3 }) {
  Stack({ alignContent: Alignment.Center }) {
    // SymbolGlyph 或 Image
  }
  .width(26)
  .height(26)

  Text(label)
    .height(16)
    .fontSize(11)
    .textAlign(TextAlign.Center)
}
```

这样对齐依据从“字符字形”变成“固定图标盒子”，肉眼对齐会稳定很多。

## 初步结论

当前没对齐不是 Tab 容器对齐失败，而是 Unicode 文本符号不适合当正式图标使用。

最合理的下一步不是继续调 `fontSize`，而是统一图标来源和图标盒子：

- 标准功能图标：优先 `SymbolGlyph`。
- 品牌化游戏图标：使用项目自有 SVG/PNG。
- 三方 ohpm 图标库：暂缓，等原生和自有资源都不满足再评估。

## 试用记录

2026-06-10 已先落一个鸿蒙原生系统符号低风险版本：

- 首页：`sys.symbol.house`
- 收藏：`sys.symbol.heart`
- 成就：`sys.symbol.star`
- 设置：`sys.symbol.more`

这版同时给图标增加固定 `26vp x 26vp` 视觉盒子，避免继续由 Unicode 字符的字形框决定对齐。

构建结果：

- `./scripts/build-app.sh --stacktrace`：通过。
- `git diff --check`：通过。

待真机确认：

- 四个底部图标是否已经在同一视觉基线上。
- `sys.symbol.more` 作为“设置”是否语义足够明确。
- 浅色/深色模式下图标颜色和文字对比度是否稳定。
