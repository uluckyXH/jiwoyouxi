# HarmonyOS 原生开发规范

本文是《鸡窝游戏》当前阶段的开发约束。目标不是追求形式完整，而是保证后续新增游戏、改造游戏和修复问题时，代码可维护、可测试、可真机验收。

适用范围：

- 只适用于本仓库 HarmonyOS NEXT 原生 ArkTS / ArkUI 工程。
- 不引入 ArkWeb、WebView、HTML 游戏页或跨端运行壳，除非项目负责人明确批准。
- 新代码必须遵守本文；历史代码在被改动时逐步收敛到本文。

## 1. 总体原则

- 原生优先：界面、动画、输入、渲染、存储都优先使用 ArkTS / ArkUI / Canvas / preferences 等原生能力。
- 模块清楚：页面只负责组织 UI 和绑定状态，规则、类型、渲染、存储、音频、动效 token 分别放到对应模块。
- 小步可测：新增或改造功能要能单独构建、单独真机验证；纯规则优先写单元测试。
- 留档可追溯：每次有意义的测试都要在 `测试报告/` 下保留 Markdown 测试报告。
- 不把复杂度藏进大文件：不要把一个游戏的所有逻辑、UI、主题、弹窗、规则全部写在一个 `Page.ets` 里。

## 2. 目录职责

当前项目结构总体可用，后续应沿用并收紧边界。

```text
entry/src/main/ets/
  app/           应用级存储、应用级服务，不放 UI。
  components/    可复用 ArkUI 组件，至少被两个页面或游戏复用时再放入。
  design/        视觉、颜色、尺寸、动效 token，例如 CoopTheme、CoopMotion。
  entryability/  Ability 生命周期入口。
  games/         各小游戏独立目录。
  model/         共享领域模型和纯规则，当前主要服务 RPS/通用模拟。
  pages/         应用页面和薄路由，例如 Index、HubPage。
  render/        Canvas 或渲染器实现。
  services/      跨模块服务，例如 AudioService。
  shell/         游戏注册表、模块定义、统一转场和游戏壳能力。
```

目录边界要求：

- `pages/Index.ets` 只做应用级路由、状态编排和入口分发，不写具体游戏规则。
- `shell/` 放合集壳层能力，例如 `GameRegistry`、`GameSceneTransition`；游戏私有逻辑不要放到这里。
- `design/` 只放稳定 token 和设计系统能力，不放业务判断。
- `app/CoopStorage.ets` 统一管理本地持久化；游戏页不要直接读写 preferences。
- `games/<gameId>/` 是游戏私有代码边界。一个游戏内部优先自给自足，不把单游戏私有组件丢到全局 `components/`。
- `components/` 只接收真正可复用的通用组件。组件第一次出现时先留在游戏目录，第二个游戏复用时再提升。

## 3. 游戏模块拆分

新增或重做游戏时，推荐按下面结构拆分：

```text
entry/src/main/ets/games/<gameId>/
  <GameName>Page.ets          页面入口，接收 bestScore、exitToHub、recordScore。
  <GameName>Types.ets         类型、枚举、常量接口。
  <GameName>Rules.ets         纯规则函数，适合单元测试。
  <GameName>Theme.ets         游戏私有颜色、尺寸、动效参数。
  <GameName>Board.ets         核心棋盘、画布或操作区。
  <GameName>Hud.ets           分数、时间、状态栏、工具栏。
  <GameName>Overlay.ets       暂停、结算、教程等覆盖层。
```

可以根据规模增减，但必须遵守以下拆分规则：

- `Page.ets` 只负责组装子组件、管理页面生命周期、连接 `recordScore` 和 `exitToHub`。
- 纯玩法规则必须放在 `Rules.ets` 或独立系统文件中，不能混在 ArkUI `build()` 里。
- 类型、枚举、对象结构放在 `Types.ets`，避免多个文件各自声明一份近似类型。
- 主题参数放在 `Theme.ets` 或 `design/` token，避免在组件里散落魔法颜色和魔法数。
- 棋盘、HUD、弹窗分别拆组件。只要 UI 区块能独立命名，就优先拆出文件。
- 页面超过 800 行、单个 `build()` 超过 160 行、或同一文件同时包含规则和 3 个以上 UI 区块时，必须拆分后再合并。
- 文件超过 1000 行只能作为临时过渡，改动该文件时必须同步制定拆分计划。

小游戏最小可接受结构：

```text
<GameName>Page.ets
<GameName>Types.ets
<GameName>Rules.ets
```

如果确实是很小的展示型功能，可以暂时只有 `Page.ets`，但一旦出现计分、胜负、随机生成、碰撞、计时、棋盘状态等规则，就要拆 `Types` 和 `Rules`。

## 4. ArkTS 编码规范

项目采用 ArkTS，按华为 ArkTS 编程规范收敛。

命名：

- 类名、枚举名、命名空间名使用 `UpperCamelCase`。
- 变量名、方法名、参数名使用 `lowerCamelCase`。
- 常量和枚举值使用全大写加下划线，例如 `MAX_GRID_SIZE`。
- 布尔变量使用 `is`、`has`、`can`、`should` 等正向含义前缀，避免 `isNotReady` 这类否定命名。
- 不使用拼音命名，不使用含义不清的 `data`、`info`、`tmp` 作为关键业务对象名。

格式：

- 使用 2 个空格缩进，不使用 Tab。
- 单行尽量不超过 120 个字符。
- `if`、`for`、`while`、`switch` 等语句必须使用大括号。
- 每行只声明一个变量，不把多个赋值语句写在同一行。
- 注释解释“为什么这样做”，不要复述代码“做了什么”。

类型：

- 能明确类型时不要使用 `any`、`object` 或宽泛联合类型。
- 跨文件传递的数据结构要声明 `interface`、`class` 或明确的类型别名。
- ArkUI 状态对象不要随手塞匿名大对象。需要跨组件传递时，先放到 `Types.ets`。
- `@State`、`@Prop`、`@Link` 等状态字段要尽量小而稳定，不把完整引擎对象暴露给多个组件同时改。
- 数组元素类型必须稳定。不要在同一个数组混合数字、字符串、对象等不同结构。

变量和函数：

- 不会重新赋值的变量使用 `const`。
- 函数要表达单一意图。纯规则函数优先设计成输入明确、输出明确、无 UI 副作用。
- 性能敏感路径避免可选参数和频繁闭包；必要时使用默认参数或显式参数传递。
- 异常用于异常场景，不在循环或游戏帧更新里频繁抛异常。

## 5. ArkUI 页面和组件规范

- 游戏首屏必须是可玩界面，不做营销落地页。
- 页面根布局要明确 `.width('100%')` 和 `.height('100%')`，核心区域尺寸要稳定。
- 按钮、棋盘、工具栏、计分器等固定格式 UI 要有稳定宽高、比例或约束，避免状态变化导致布局跳动。
- 点击热区不得小于 `40vp x 40vp`。
- 小屏上不得遮挡系统手势条；全屏游戏必须提供明确的“大厅”或关闭入口。
- 系统返回键在游戏内返回大厅，在大厅交给系统。
- 自定义手势不能和系统返回、系统底部手势冲突。
- 适配不同屏幕尺寸时，优先保证竖屏手机体验；平板、折叠屏和宽屏不能出现明显遮挡、重叠或不可操作。
- 避免卡片套卡片。游戏核心区域要清楚、稳定，工具和信息层级要克制。

## 6. 原生动画规范

项目采用 HarmonyOS 原生 ArkUI 动画，不做 Web 动画方案。

统一原则：

- 全局转场、时长、曲线、位移距离优先集中在 `entry/src/main/ets/design/CoopMotion.ets`。
- 进入游戏、退出游戏等壳层动效优先由 `entry/src/main/ets/shell/GameSceneTransition.ets` 统一处理。
- 游戏内部微交互可以在游戏目录内实现，但要复用 `CoopMotion` 的时长和曲线。
- 动画只表现状态变化，不承载业务正确性。分数记录、状态清理、路由切换不能依赖某一帧动画必然执行。

推荐动效：

- 大厅进入游戏：使用层级转场，主体内容左右位移加轻微淡入淡出。
- 游戏返回大厅：和进入方向相反，保持空间关系一致。
- 暂停、结算、设置等覆盖层：使用淡入、轻微缩放或垂直位移。
- 分数变化、连击、命中反馈：使用短时缩放、透明度或颜色过渡，不改变核心布局尺寸。
- 拖拽、滑动类游戏：手指跟随优先，松手后用 ArkUI 动画或 Canvas 帧更新完成回弹、惯性和吸附。

禁止事项：

- 不在多个页面各自硬编码一套进入/退出动画。
- 不为了动画引入 WebView、CSS 动画或 HTML 容器。
- 不使用导致文字重叠、按钮漂移、棋盘尺寸跳变的动画。
- 不把长耗时计算放进动画回调或 ArkUI `build()`。

## 7. 状态、生命周期和存储

- `aboutToDisappear` 中必须清理定时器、动画循环、输入监听、音效播放或其他持有资源。
- 游戏暂停、重开、结算、返回大厅要有清晰状态机，不用多个布尔值互相猜状态。
- 最高分统一通过 `recordScore(score)` 上报，页面不要直接写存储。
- 需要持久化的数据统一收敛到 `CoopStorage` 或明确的应用级存储模块。
- `recordScore` 的分数必须是非负数，结算时只上报最终有效分数。
- 游戏状态如果要做恢复，先定义快照类型，再考虑写入存储。

## 8. 性能规范

ArkTS 和 ArkUI 性能敏感点在游戏场景里很容易放大，以下要求必须遵守：

- 不在 `build()` 里做大量随机生成、路径搜索、碰撞计算、排序或资源加载。
- 高频帧更新优先放到规则函数、控制器或 Canvas 渲染器，避免每帧重建大量 ArkUI 组件。
- 大量移动实体、粒子、棋盘绘制优先考虑 Canvas；按钮、HUD、弹窗继续使用 ArkUI。
- 循环内不反复读取不会变化的属性，先提取到局部常量。
- 数值数组保持元素类型稳定；纯数值密集计算可考虑 `TypedArray`。
- 避免稀疏数组，例如不要创建空数组后直接写入很大的下标。
- 游戏主循环里不要频繁创建异常、复杂对象或临时闭包。
- 音频、图片等资源要复用，不能在每次点击或每帧中重复加载。

## 9. 测试规范

测试分为三层：构建检查、自动化测试、真机验收。新增功能至少完成构建检查和真机验收；涉及纯规则的代码必须补单元测试。

构建检查：

```bash
./scripts/build-app.sh --stacktrace
```

单元测试：

- 纯规则、计分、胜负、棋盘生成、碰撞、状态机等逻辑必须设计成可单元测试。
- 单元测试优先使用 HarmonyOS 官方 Hypium / ArkXTest 体系。
- 推荐测试目录为 `entry/src/ohosTest/ets/test/`，按功能命名测试文件。
- 单元测试覆盖正常路径、边界值、异常输入和回归问题。
- 不能测试的原因必须写进测试报告，不允许只口头说明。

UI 自动化测试：

- 大厅进入游戏、返回大厅、结算弹窗、最高分更新等稳定流程，适合逐步沉淀 UITest。
- 自动化 UI 测试应优先通过可稳定定位的文本、组件类型或可访问属性查找控件。
- 涉及拖拽、长按、连续点击的游戏，至少保留手工真机测试步骤；稳定后再补自动化。

真机验收：

- 每个新增游戏至少验证进入、返回、重开、暂停/继续、结算、最高分、低分辨率布局。
- 动画相关改动必须真机观察进入和退出是否顺滑，是否出现闪屏、黑屏、跳帧、重叠。
- 布局相关改动必须检查小屏和常规手机屏；有条件时补平板或折叠屏。
- 音频、触摸、长按、滑动、系统返回等体验问题不能只靠预览器判断。

## 10. 测试报告留档

测试报告统一放在根目录 `测试报告/`，使用 Markdown 文件提交。

命名格式：

```text
测试报告/YYYY-MM-DD-<测试范围>.md
```

示例：

```text
测试报告/2026-06-10-游戏切换动画.md
测试报告/2026-06-10-扫雷规则.md
```

报告至少包含：

- 测试日期和测试人。
- 分支、提交号或变更范围。
- 设备型号、HarmonyOS 版本、DevEco Studio 版本。
- 构建命令和测试命令。
- 测试用例列表、结果、失败原因。
- 真机截图或录屏文件位置。大型截图、录屏和日志默认留本机 `phone-shots/`，报告中记录文件名或路径。
- 遗留问题和下一步处理。

如果本次只是文档改动，可以在提交说明中写明“未运行构建，文档-only”，不强制新增测试报告。

## 11. PR 和提交检查

合并前按下面清单自查：

- 是否仍然是 HarmonyOS 原生 ArkTS / ArkUI 方案。
- 新增或改造的游戏是否按模块拆分，没有把规则、UI、弹窗、主题全部塞进一个文件。
- 共享能力是否放到了正确目录，没有把单游戏私有代码提前放到全局组件。
- 纯规则是否有单元测试，或在测试报告中说明暂未覆盖的合理原因。
- 是否运行 `./scripts/build-app.sh --stacktrace`，失败原因是否已记录。
- 是否完成受影响游戏的真机验收。
- 是否在 `测试报告/` 留下测试报告。
- 是否更新相关文档，例如 `docs/新增小游戏接入指南.md`、`docs/真机验收清单.md` 或设计文档。
